# 文章・図解スタイルガイド

Node-RED ガイドサイトの解説文と図解を、落ち着いた日本語技術文書のトーンにそろえるための基準。
グローバルの `japanese-tech-writing` スキルを、本サイトの HTML ガイド向けに翻案したもの。
新規ガイドの執筆時と既存ガイドの推敲時に、このチェックリストで点検する。

## 前提

- 本サイトは初心者向けの静的 HTML ガイドである。
- 文章の論理と記法は技術文書の規範にそろえる。一方で、図解とカード UI は学習体験の一部として活かす。
- 完全な装飾排除はしない。装飾は「意味を持つもの」を残し、「飾りだけのもの」を落とす。

## カテゴリ別チェックリスト

### A 記法

- 中黒（・）を一般語の並列に使わない。読点か「や」に置き換える。
  - 例：「映像・音声のストリーミング」→「映像や音声のストリーミング」
  - 例外：プロトコル名やポート名など固有名詞の列挙（`HTTP・MQTT・Modbus`）は、短く読みやすいので中黒のままでよい。
  - 例外：単一の固有名詞の内部（`TCP/UDP` のスラッシュなど）はそのまま。
- 2 倍ダッシュ（——）と em ダッシュ（—）を地の文と見出しに使わない。
  - 同格や補足の挿入は括弧（）に入れる。
  - 言い換えや敷衍は句点で二文に分けるか、読点でつなぐ。
  - 例外：書誌情報（`RFC 5321 — SMTP`）、表中の「該当なし」記号、英語の複合語（`Curry–Howard`）は対象外。
- 見出しに区切り記号で 2 要素を詰め込まない。
  - 例：「HTTP — リクエスト/レスポンス型」→「HTTP（リクエスト/レスポンス型）」
  - 例：「プロトコル早見表 — どれをいつ使う？」→「プロトコル早見表：どれをいつ使う？」
- 用語と定義を並べる箇条書きは「**用語**：説明」の形にする（区切り記号で代用しない）。

### B 装飾の抑制

- 見出し（`h1`〜`h4`）の先頭から装飾的な絵文字を外す。
  - 見出しは内容を表す句だけにする。
- callout（`.tip` `.important` `.warning`）の先頭絵文字は外し、ラベルは短い語で残す。
  - 種別は枠の色と左罫線が示すので、絵文字がなくても役割は伝わる。
- 図解の中のアイコンは残してよい。
  - 内容に対応するアイコン（手紙＝📮、電話＝📞 など）は図の一部であり、装飾ではない。
- 過剰な色つきハイライト（`.highlight-word` の乱用など）は、本当に強調が要る箇所だけにする。

### C 太字の抑制

- 本文中の `<strong>` は、論理の要所に限る。1 段落あたり 1〜2 箇所まで。
  - 残すべき太字：誤読を防ぐ否定、節の結論、初出で定義する用語。
  - 落とすべき太字：文の半分を覆うような強調、雰囲気づくりの強調。
- 用語を初めて定義・導入するときは太字にしてよい。以後その語を話題として指すときは「」を使う。
- 表のキー列やラベルの太字は、視認性のために残してよい。

### D 論証と正確さ

- 1 段落に 1 トピック。場面が変わるところで段落を分ける。
- 段落の先頭で、前の段落との関係を接続表現で示す。
- 「必ず」「すべて」など断定しすぎる表現を点検し、条件つきで正確に述べる。
- ページ内のセクションを「第 N 章」と呼ばない。サイトの「第 0 章」と紛れる。
  - 「冒頭のたとえ」「セクション 1」のように、ページ内の位置として指す。
- 前方参照（「次のページで扱う」）は段落末か節末に置く。論の途中に挟まない。

### E 視点と語り

- 読者を「あなた」と呼ばない。「読者」「開発者」などの役割名か、無人称で書く。
- 架空の人物設定を無意味に持ち込まない。
- 修辞疑問や溜め（「ここには〜が潜んでいる」）は、緊張が議論に効く要所だけにする。
- 語りかけ（「〜ですね」「〜だったのです」）は、初心者向けの範囲で控えめに使う。連発しない。

### F LLM っぽい表現

- 「重要なのは〜である」「本ページでは〜を扱う」などの予告と総括を避け、主張をそのまま書く。
- 「不可欠」「核心的」「根本的」などの空虚な形容で強調しない。
- 指す内容が一意に決まらない比喩を避ける。平易な動詞で言う。

### G 図解

- フロー図やカードのラベルから、中黒並列と冗長な語を取り除く。
- 図と本文の対応を明確にする。図で示したことを本文で繰り返さない。
- ASCII の罫線図は、列の見出しと位置をそろえる。

### H 設定項目の表記と説明

設定項目の表（「設定項目一覧」）は、読者が Node-RED エディターのどの欄を指しているかを迷わず判断でき、かつその欄を何のために設定するかを理解できることを目標にする。

表記の基準。

- エディターで日本語化されている項目は、その日本語表示をそのまま見出しに使う。英語の併記はしない。
  - 例：XML ノードの「プロパティ」「文字列を参照するための接頭辞」。
- エディターで英語表示のままの項目は、英語ラベルに日本語訳を添える。形式は「**English Label**（日本語訳）」。
- 助詞や前置詞で始まる機械翻訳調の語（「に基づく」など）を項目名にしない。意味が伝わる名詞句にする。
- 日本語化されているか英語のままかは、ノードのソース（`<node>.html` の `data-i18n` キー）と locale ファイル（`locales/ja/messages.json`・`locales/en-US/messages.json`）で確認してから決める。

説明の基準。

- 項目名の翻訳だけでは説明にならない。その項目が何を制御し、どんなときに何のために設定するのかを書く。
- 前提概念が要る項目は、概念を補ってから説明する。
  - 例：XML ノードの「XMLタグ属性を次のプロパティ名として表現」は、XML をオブジェクトへ変換するとき、要素の属性（`<book id="1">` の `id`）をまとめて格納するキー名（プレースホルダ表示は `$`）を指す。この前提を補わずに名前だけ載せても役割は伝わらない。
- 表に挙げる項目は、エディターのアクティブなプロパティを抜けや重複なく網羅する（CLAUDE.md のプロパティ分類に従う）。

構成の基準。

- 項目が多いノードは、すべてを 1 つの表に平らに並べない。エディターのタブやセクションの区切りに合わせてグループに分けて解説する。
- よく使う主要な項目と、既定のままでよい詳細な項目を分けて示す。
- グループの単位はエディターの実構造に合わせる。ガイド独自の分類で実エディターと食い違わせない。

作り方。

- 設定項目の節はエージェントへ一括で任せない。1 ノードずつ、ソースと実際の挙動を確認しながら書く。

## 検証手順

1. サンプルフロー（`<textarea>` 内の JSON）は、編集後に `JSON.parse` 相当でパースが通ることを確認する。
2. 内部リンク（章間参照、`#glossary` などのアンカー）が生きていることを確認する。
3. 装飾のトーンがページ全体で一貫しているか目視で確認する。
4. 1 ファイル＝ 1 コミット（`refactor:` 型）で、差分を追いやすくする。
