## 2. 全体像：3 テーマは 1 本の動線で繋がる

A・B-1・C は「**バイナリデータをどう読み書きするか**」という 1 つの軸でつながる。
FTP（B-2）はファイル転送の話題で、この軸からはやや独立する。

```
A バイナリ/テキストの基礎（Buffer とは何か）
        ↓ 「では実際にどう変換する？」
B-1 buffer-parser（バイト列 ⇔ 数値・文字列の変換ノード）
        ↓ 「現場のバイト列の代表例 = PLC レジスタ」
C PLC レジスタ（エンディアン・ワードスワップ・符号・32bit 跨ぎ）
        ↘ 既存 modbus ガイドへ接続

B-2 FTP（サーバー/クライアント）… 独立テーマ（ファイル転送）
```

学習者が「Buffer って何？」→「変換する道具」→「実データ（PLC）での落とし穴」と
段階的に進める導線にするのが狙い。各ガイド末尾で次のガイドへ橋渡しする。

特筆点：buffer-parser（B-1）はビッグ/リトルエンディアン変換を備える。A の概念と
C の実応用を、この 1 ノードが橋渡しする構造になっている。

## 3. 既存ガイドのカバー範囲（重複マップ）

新規ガイドが既存と重複しないよう、現状のカバー範囲を確認した。

| 既存ガイド | バイナリ/Buffer 関連の現状記述 |
| --- | --- |
| `nodered-variable-types-guide.html` | データ型の一つとして Buffer に言及 |
| `nodered-variable-types-part2-guide.html` | Buffer/バイナリに複数言及 |
| `nodered-tcp-node-guide.html` | TCP 受信データが Buffer になる文脈で言及 |
| `nodered-batch-node-guide.html` | Buffer に言及 |
| `nodered-modbus-node-guide.html` | 「3. MODBUS の基礎知識」「11. 用語集」。0 始まりアドレス・4xxxx 形式は既出 |

**含意**：

- A は `variable-types` と切り分けが必要。既存は「型一覧の中の Buffer」という浅い扱いなので、
  A は「Buffer の中身（バイト・エンコーディング・16 進）」を深掘りする独立テーマにできる。
- C は `modbus` ガイドと最も重複する。アドレス体系（0 始まり・4xxxx）は既出なので、
  C は「**レジスタ値のバイト解釈**（エンディアン・ワードスワップ・符号・float/32bit）」に
  焦点を絞れば重複を避けられる。

## 4. 各項目の詳細

### A. バイナリデータとテキストデータ（Node-RED の Buffer）

- **種別**：概念解説ガイド（特定ノードに依存しない）
- **想定読者**：文字列は扱えるが「バイト」「16 進」「文字コード」で混乱する初心者
- **カバー内容（案）**：
  - テキスト（文字列）とバイナリ（バイト列）の違い
  - 文字コード（UTF-8 / ASCII）と、1 文字が複数バイトになる話
  - Node-RED の Buffer 型とは（`msg.payload` が Buffer になる場面：TCP/Serial/File/HTTP）
  - Buffer の見え方（`[0x48, 0x65, ...]` / 16 進ダンプ）
  - 文字列 ⇔ Buffer の相互変換（標準ノード・Function での `Buffer.from` / `.toString`）
  - よくある事故（文字化け、改行コード、null 終端、結合での分割）
- **方針確定（2026-06-29）**：**新規ファイルは作らず、データ型ガイド
  （`variable-types`、part2 を軸に）へ節として追記する**。既存が「型一覧の中の Buffer」と
  浅い扱いのため、そこを上記カバー内容まで掘り下げて拡充する。
- **連携**：追記した節から B-1（buffer-parser）と C（PLC レジスタ）へ「詳しくはこちら」で送る。

### B-1. buffer-parser ノード（サードパーティ）

- **パッケージ**：`node-red-contrib-buffer-parser` v3.2.2（公開 2022-01-30）
- **作者 / 出典**：Steve-Mcl — https://github.com/Steve-Mcl/node-red-contrib-buffer-parser
- **説明**：値を buffer/array と相互変換。ビッグ/リトルエンディアン対応。
- **種別**：サードパーティガイド（インストール方法・GitHub リンク・バージョン・免責を明記）
- **位置づけ**：A の「変換の概念」を実際に行うノード。バイト列 ⇔ int8/16/32・float・
  文字列・bool 配列などをスペック表で一括変換できる。Modbus/Serial の後段で多用される。
- **カバー内容（案）**：
  - 何を解決するノードか（Function で書きがちな変換を GUI 化）
  - specification（item ごとの type / offset / length / mask）
  - スワップ（byte/word swap）設定 ← C と直結する重要点
  - 入力 Buffer → 出力オブジェクト、その逆（オブジェクト → Buffer）
  - Modbus 配列（16bit ワード配列）からの実数復元という代表例
- **注意点**：エンディアン・スワップの説明は C と共通。重複させず相互リンクで分担する。

### B-2. FTP / SFTP（サーバー・クライアント）（サードパーティ）

方針確定（2026-06-29）：**FTP と SFTP の双方**を、**サーバー・クライアント両対応**で扱う。
npm を網羅調査した結果、1 パッケージで client+server を兼ねるものは無く用途別に分かれる。
各カテゴリで「DL 数・star・更新」を比較し定番を選定した（下表は実調査値）。

**FTP（平文）**

| 役割 | パッケージ | DL/月 | star | 最終更新 | 提供ノード |
| --- | --- | --- | --- | --- | --- |
| クライアント | `node-red-contrib-ftp` | 約 6,810 | 13 | 2022-06 | `ftp` / `ftp-in` / `ftp-out` |
| サーバー | `node-red-contrib-ftp-server` | 約 6,151 | 18 | 2022-03 | `ftp-server` |

**SFTP（SSH 上の安全な転送）**

| 役割 | パッケージ | DL/月 | star | 最終更新 | 提供ノード | 備考 |
| --- | --- | --- | --- | --- | --- | --- |
| クライアント（更新重視・推奨） | `node-red-contrib-ssh-sftp` | 約 218 | 少 | **2024-01** | `sftp` | ssh2-sftp-client ベース・唯一の近年更新 |
| クライアント（実績重視） | `node-red-contrib-sftp` | 約 339 | 21 | 2018-04 | `sftp-get` / `sftp-put` / `sftp-delete` | DL・star 最多だが古い |
| サーバー | `node-red-contrib-sftp-server` | 約 143 | 少 | 2019-04 | `sftp-server` | 選択肢が乏しい |

- 出典: ftp=joeartsea / ftp-server=biddster / sftp=mbrooks / ssh-sftp=elmagopy（各 GitHub）。
- **SFTP クライアントの推奨**：DL・star 最多は `node-red-contrib-sftp`（2018）だが、唯一
  2024 に更新され定番の ssh2-sftp-client ベースなのは `node-red-contrib-ssh-sftp`。
  セキュリティ用途では更新性を重視し **ssh-sftp を第一候補**、実績重視なら sftp、と両論を
  提示して着手時に最終確定する。
- **SFTP サーバー**：Node-RED ノードは選択肢が乏しく更新も古い。SFTP サーバーは通常 OS の
  SSH デーモンで提供するのが一般的である旨も補足する。

**メンテ状況の総括**：FTP 系は 2022、SFTP サーバーも 2019 が最終。近年更新は SFTP
クライアント `ssh-sftp`（2024）のみ。着手時に各 DL・更新を再確認する。

- **種別**：サードパーティガイド（FTP・SFTP を 1 ガイドで併説）
- **セキュリティ注記（必須）**：平文 FTP は認証情報が平文で流れる。LAN 内・検証用に限定し、
  インターネット越しでは SFTP/FTPS を勧める。パッシブ/アクティブモードとファイアウォールの
  注意も添える。
- **カバー内容（案）**：
  - FTP / SFTP / FTPS の違い（平文か暗号化か、ポート、認証方式）
  - FTP クライアント：接続設定・取得/送信・ファイル一覧
  - FTP サーバー：エンドポイント定義・受信トリガー・保存
  - SFTP クライアント：SSH 鍵/パスワード認証・取得/送信
  - SFTP サーバー：選択肢と OS 側 SSH デーモンでの代替
  - セキュリティと運用注意

### C. PLC のレジスタ（エンディアン・Modbus 注意点）

- **種別**：概念解説ガイド（Modbus と密結合だが、ノード非依存の「データ解釈」に焦点）
- **想定読者**：Modbus で値は取れたが「数値が化ける/桁が違う/符号が変」で詰まる人
- **既存との重複**：`modbus` ガイドが最も近い。アドレス体系（0 始まり・4xxxx 形式）は既出。
  → C は**重複を避け、バイト解釈に特化**する。
- **カバー内容（案）**：
  - レジスタとは（16bit ワード単位、ビット/ワード/ダブルワード）
  - データ型の対応（符号なし/符号付き 16bit、32bit 整数、32bit float）
  - **エンディアン**（ビッグ/リトル）と **ワードスワップ**（ABCD / CDAB / BADC / DCBA）
  - 32bit 値が 2 レジスタに跨る話と結合順
  - スケーリング（生値 × 係数）とオフセット
  - buffer-parser（B-1）での実装例にリンク
  - メーカー差（同じ float でもワード順が違う）の注意
- **方針確定（2026-06-29）**：**新規独立ガイド** `nodered-plc-register-guide.html` とする。
  テーマが複雑なため独立させ、`modbus` ガイドなど関連ノードからの**参照先**（「詳しくはこちら」）
  として位置づける。modbus 側には深入りせず本ガイドへ送る。

## 6. 優先順位案（推奨）

依存関係（A が B-1・C の前提知識になる）と、既存資産の活用度から提案する。

1. **A バイナリ/Buffer 基礎** … データ型ガイドへの追記。他テーマの土台。最初に行うと後段が書きやすい。
2. **B-1 buffer-parser** … A の直接の実践。エンディアン対応で C への布石になる。
3. **C PLC レジスタ** … 新規独立。既存 modbus から参照される。B-1 の応用例として完結させやすい。
4. **B-2 FTP / SFTP** … 独立テーマ。FTP・SFTP の client・server をまとめて扱う。最後に着手。

## 7. 各ガイド共通の品質基準（再掲）

着手時は以下を必ず満たす（詳細は CLAUDE.md と style-guide.md）。

- 設定項目名はエディターの実表示に完全一致。サードパーティで ja locale が無い場合は
  「**English Label**（日本語訳）」形式。
- サンプルフローの JSON は `python3 -m json.tool` でパース成功。
- サードパーティは「インストール方法・GitHub リンク・バージョン・免責」を明記。
- 設定項目の節はエージェントへ丸投げせず、1 ノードずつソースと挙動を確認して書く。
- フローティングホームボタン・前後リンク・コピーボタンを備える。
- 既存ガイドへの橋渡しリンクは実在ファイル名を指す。
