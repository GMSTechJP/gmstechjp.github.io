📖 変数とは？

変数とは、データを入れておく「箱」のようなものです。プログラムで使うデータ（数値、文字、情報など）を一時的に保存しておく場所と考えてください。

🏠 日常生活での例え

📦 引き出し

引き出しにラベル（変数名）を貼って、中に物（データ）を入れる

温度 = 25度 湿度 = 60% 場所 = "リビング"

🏷️ 付箋メモ

メモ帳に「温度: 25度」と書いて保存しておくイメージ

メモ1: カウント = 5回 メモ2: 名前 = "センサーA" メモ3: 状態 = オン

📊 Excel のセル

セルにデータを入れて、他のセルから参照する（参照される側が変数）

A
B

💡 実際のExcel：
• B1セルには「=A1*2」と入力するが、表示は「50」
• B2セルには「=A2+A1」と入力するが、表示は「55」
A1, A2セル = 変数（データを保存）
B1, B2セル = 関数の結果（計算結果を表示）

💻 Node-REDでの変数

Node-REDでは、変数を使ってこんなことができます：

💾 データの保存

センサーから受け取った温度データを保存して、後で使う

🔄 データの受け渡し

あるノードから次のノードへデータを渡す

🔢 計算や処理

保存したデータを使って計算したり、比較したりする

📊 状態の記録

「何回ボタンが押されたか」などの状態を覚えておく

🎨 変数の3つの要素

要素 説明 例
1️⃣ 名前 変数の識別子 temperature, counter, isActive
2️⃣ 型（タイプ） データの種類 数値、文字列、真偽値
3️⃣ 値 実際のデータ 25, "センサーA", true
具体例：
temperature (数値) = 25

「temperature」という名前の、「数値」型の変数に、「25」という値が入っている

💡 初心者の方へ： 最初は難しく感じるかもしれませんが、「変数 = ラベル付きの箱」と覚えておけば大丈夫です。Node-REDを使いながら、少しずつ慣れていきましょう！

📋 1. データ型（値のタイプ）

Node-REDで扱えるデータ型について、具体例とともに説明します。これらのデータ型は、msgオブジェクトのプロパティにも、Part 2で扱うcontext系変数にも、共通して使用できます。

a_z文字列（String）

テキストデータを保存します

"こんにちは" "温度: 25度" "sensor001"
使用例：
センサー名、メッセージ、住所など

0_9数値（Number）

数字のデータを保存します

25 3.14 -10 100.5
使用例：
温度、湿度、カウント数、計算結果など

◉真偽値（Boolean）

true（真）またはfalse（偽）の2つの値のみ

true（オン、正しい） false（オフ、間違い）
使用例：
スイッチのON/OFF、条件判定の結果など

[]配列（Array）

複数のデータを順番に並べて保存

[1, 2, 3, 4, 5] ["apple", "banana", "orange"] [{"name": "A"}, {"name": "B"}]
使用例：
センサー値のリスト、複数デバイスのデータ、履歴データなど

{}JSON（オブジェクト）

複数のデータをキーと値のペアで保存

{ "温度": 25, "湿度": 60, "場所": "リビング" }
使用例：
センサーの複数データ、設定情報など

01バッファ（Buffer）

バイナリデータ（画像、音声など）

<Buffer 48 65 6c 6c 6f>
使用例：
画像データ、ファイルの中身など

🕐日時（Timestamp）

現在の日時を数値で保存

1696582800000 （ミリ秒単位のUnix時間）
使用例：
データ取得時刻、イベント発生時刻など

J:JSONata式

データを加工・計算する式

$number(payload) * 2 $substring(topic, 0, 5)
使用例：
データの変換、計算、文字列操作など

$環境変数

設定ファイルから値を読み込む

$env("API_KEY") $env("SERVER_URL")
使用例：
APIキー、サーバーアドレスなどの設定値
📌 配列とJSONの違い：
  • 配列：順序のあるデータの集まり。インデックス番号（0, 1, 2...）でアクセス
  • JSON：名前（キー）と値のペアの集まり。キー名でアクセス
// 配列：インデックスでアクセス fruits[0] → "apple" fruits[1] → "banana" // JSON：キー名でアクセス sensor.温度 → 25 sensor.湿度 → 60

📨 2. msgオブジェクトとは

Node-REDのフローを流れるデータは、すべて msgオブジェクト という入れ物に入っています。msgは「ノードからノードへ届ける小包」のようなもので、中に payload（主役のデータ）や topic（タグ）などの情報を入れて、ワイヤーに沿って次のノードへ運ばれます。

📦 msgオブジェクトの正体

msgは、JavaScriptの オブジェクト（キーと値のペアを持つデータ構造）です。自由にプロパティを追加できますが、Node-REDの慣例として特定の名前のプロパティが広く使われます。

// msgオブジェクトの中身（一例） { payload: 25, // 主役のデータ（温度値など） topic: "temperature", // データの種類を表すタグ timestamp: 1696582800000, // 受信時刻（Injectが付けることがある） _msgid: "abc123..." // Node-REDが自動付与するメッセージID }

🏷️ よく使うmsgのプロパティ

プロパティ 役割 代表的な値の型
msg.payload メインで扱うデータ本体。ほぼすべてのノードで読み書きする 任意（数値・文字列・JSON・配列・Buffer など）
msg.topic データの種類・分類を示すタグ。振り分けや表示に使う 文字列（例: "temperature"）
msg.timestamp メッセージ発生時刻（Injectノードが自動付与する場合あり） 数値（Unix時間 ミリ秒）
msg.任意の名前 独自のプロパティを自由に追加できる（例: msg.unit, msg.sensor） 任意
📌 msgオブジェクトの3つの特徴：
  • 一時的：ノードの処理が終わると、次のノードに渡されて役目を終える
  • 1本のフロー限定：ワイヤーで繋がれたノードの間でしか共有されない
  • 自由に拡張可能：msg.xxx の形で独自プロパティを追加できる

🔀 msgはどう流れるのか

Injectノードで作られたmsgは、ワイヤーに沿って次のノードに渡されます。各ノードはmsgを受け取って加工し、次のノードへ渡す役割を担います。Debugノードで途中経過を確認するのが開発の基本です。

Inject → msg.payload = 25 → Change → msg.payload = "25℃" → Debug
💡 msgは「コピー」されるの？「参照」されるの？

同じmsgが複数の出力先に分岐するとき、各ノードは同じオブジェクトを参照します（コピーではない）。そのため、あるブランチでmsgを直接書き換えると、別のブランチにも影響が出ます。安全に扱うには、Changeノードの「値の代入」を使うか、Functionノード内で RED.util.cloneMessage(msg) を使います（詳しくは各ノードガイドを参照）。

⚙️ 3. Injectノードでmsgを作る

Injectノードは、フローの「入口」として 新しいmsgオブジェクトを生成するノードです。ボタンで手動起動したり、定期実行したりして、msg.payloadやmsg.topicに指定した値を入れてワイヤーに流します。

Inject →
生成されるmsgオブジェクト
msg.payload = 25 （数値）
msg.topic = "temperature" （文字列）

Injectで指定できる型

Injectノードの設定画面で、msg.payloadの 値の型 を選択できます。目的のデータに合わせて適切な型を選びます。

型 用途 例
文字列 テキストメッセージ、識別子 "Hello"、"sensor-A"
数値 温度、カウント、計算値 25、3.14
真偽値 ON/OFF、判定結果 true、false
JSON 複数の情報をまとめて送信 {"temp": 25, "humid": 60}
配列 複数データのリスト [1, 2, 3]
日時 現在時刻の送信 1696582800000（ミリ秒）
💡 Injectの応用：
  • 複数プロパティを一度に送信：設定画面の「＋追加」でpayload以外のプロパティ（topic、任意の名前）を同時に設定できる
  • 定期実行：「繰り返し」欄で「指定した時間間隔」を選ぶと、一定間隔で自動起動
  • Node-RED起動時に実行：「デプロイ後に実行」にチェックを入れると、フロー起動直後に1回流す

🔧 4. Changeノードでmsgを変更する

Changeノードは、流れてきたmsgオブジェクトの プロパティを追加・変更・移動・削除 するためのノードです。JavaScriptを書かずに、GUI操作だけでmsgを加工できます。

Changeノードの4つのアクション

アクション 説明 例
値の代入（Set） プロパティに値を設定する（上書きまたは新規追加） msg.payload を 数値 25 に設定
値の変更（Change） 文字列内の特定部分を置換する msg.payload の "old" を "new" に変換
移動（Move） プロパティを別の名前に移す（元は削除） msg.payload を msg.temperature に移動
削除（Delete） 指定したプロパティを消す msg.tempData を削除

値のタイプ（「値の代入」で使える型）

「値の代入」で右辺に指定できる値の型は、Injectノードと基本的に同じです。文字列・数値・真偽値・JSON・配列・日時・JSONata式・環境変数などから選べます。

💡 このページではmsg内の操作だけ扱います：

Changeノードの「値の代入」では、対象として msg 以外に flow や global（context系変数）も選べます。これらは Part 2 で詳しく扱うので、このPart 1では msg内のプロパティ操作 に絞って説明します。

📝 5. msg操作の実践例

ここでは、msgオブジェクトの操作だけで完結する基本的な実践例を3つ紹介します。いずれもInjectノード・Changeノード・Debugノードの3つだけで試せます。

例1: msg.payloadに値を設定してDebugで確認する（最小構成）

Inject → Change → Debug

Injectノードの設定：

msg.payload: 日時（タイムスタンプ）でデフォルトのまま

Changeノードの設定：

ルール: msg.payload を 文字列 "Hello Node-RED" に設定

動作： Injectのボタンを押すと、Changeがmsg.payloadを文字列に上書きし、Debugサイドバーに「Hello Node-RED」が表示される。

例2: 複数のプロパティを持つJSONをmsg.payloadに設定する

Inject → Change → Debug

Changeノードの設定：

ルール: msg.payload を JSON に設定 値: { "sensor": "A", "temperature": 25, "humidity": 60, "location": "リビング" }

動作： Debugサイドバーで payload.temperature や payload.location のようにドット表記でアクセスできる構造化データが流れる。

💡 なぜJSON型を使うのか： 複数の関連する値（センサー名・温度・湿度・場所）を1つのmsgにまとめると、後続のノードで1回の処理として扱える。別々のmsgに分けるより効率的。

例3: msg.topicでデータを分類し、msg.payloadにメインデータを入れる

Injectノード1つでmsg.payload（値）とmsg.topic（タグ）の両方を同時に送信できます。msg.topicは後続のSwitchノードで振り分けたり、Debugの表示を分かりやすくしたりするのに使えます。

Inject
温度=25
topic="temp" → 単位付加 → Debug

Injectノードの設定（2プロパティ）：

msg.payload: 数値 → 25 msg.topic: 文字列 → "temperature"

Changeノードの設定（2ルール）：

ルール1: msg.unit を 文字列 "℃" に設定（新規プロパティ追加） ルール2: msg.payload を JSONata式 payload & unit に設定（"25℃" になる）

動作： Debugサイドバーに topic: temperature と payload: 25℃ が並んで表示される。msg.topicが付いていると、Debugサイドバーの見出しにトピック名が表示されて識別しやすくなる。

📌 3つの例から学べること：
  • Changeノードは msgのプロパティを1つずつ操作する
  • 複数のルールを上から順に適用できる（後のルールは前のルールの結果を参照できる）
  • msg.payloadは何でも入れられる（文字列・数値・JSONなど）
  • msg.topicやmsg.unitなど、自由にプロパティを追加できる

アイコン 型名 使用場面 例

🎓 まとめ

📌 Part 1 の要点：
  • msgオブジェクト = ノード間を流れる「小包」のようなデータ
  • 代表的なプロパティ: msg.payload（主役）、msg.topic（タグ）。独自のプロパティも自由に追加できる
  • msgは1本のフロー限定・処理後は消える（一時的）
  • Injectノードは msgを作る、Changeノードは msgを加工する
  • 型セレクタ（▼）で文字列／数値／JSON／配列／日時／JSONataなどを選ぶ
  • Debugノードで中身を確認しながら開発する

ここまでで、msgオブジェクトの基礎と、Inject・Changeを使ったmsg操作の基本が身についたはずです。次のステップとして、「値を覚えておきたい」「複数のフローで値を共有したい」 という発展的なニーズに応える context系変数 を Part 2 で学びましょう。
