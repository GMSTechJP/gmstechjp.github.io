Node-RED MODBUS ノードガイド

⚠️ クラウド可（自分専用ポート） 環境とは？

これはサードパーティノードです

MODBUS は Node-RED の標準ノードには含まれていません。node-red-contrib-modbus というパッケージを追加インストールして使います（本ガイド執筆時点の最新は v5.45.2）。インストール手順は第2章で解説します。

通信の全体像がまだあいまいな方は、先に 通信の基礎（ネットワーク入門） を読むと、MODBUS が「ポーリング型（見回り点検）」の通信であることがすっきり理解できます。

1. 概要：MODBUS とは

MODBUS（モドバス）は、1979 年に Modicon 社（現 Schneider Electric）が発表した、産業機器どうしをつなぐための通信プロトコルです。40 年以上の歴史があり、いまも工場・プラント・ビル設備で最も広く使われている「産業界の共通言語」です。

🔦 見回り点検にたとえると：MODBUS は「担当者（親機）が各機械（子機）を定期的に巡回し、『いま何度？』『カウンタはいくつ？』と聞いて回る」方式です。子機は聞かれたことに答えるだけ。親機が会話の主導権を握る、シンプルで予測しやすい通信です。

どんな機器で使われる？

  • PLC（シーケンサ）— 工場の自動制御装置
  • インバータ— モーターの回転数制御
  • 電力計・電力モニタ— 電圧・電流・電力量の計測
  • 温調計・センサー— 温度・湿度・圧力の測定
  • I/O ユニット— 接点入出力の集約

MODBUS の特徴

特徴内容
マスター / スレーブ方式
（新呼称: クライアント / サーバー）		親機（マスター ＝ クライアント）が要求し、子機（スレーブ ＝ サーバー）が応答する。Node-RED は通常マスター側。Modbus 公式は近年「Client / Server」へ用語を切り替えたが、実機マニュアルやノードのプロパティ名は「Master / Slave / Unit-Id」のまま残っているケースが多い（→ 用語集）。
ポーリングマスターが定期的に問い合わせて値を集める。スレーブから勝手に送ってくることはない。
レジスタ番地で指定「何番地の値が欲しい」と番地（アドレス）を指定してやり取りする。
オープン仕様仕様が公開されており、メーカーをまたいで相互接続できる。

2. インストール方法

サードパーティノードは、Node-RED エディターの「パレットの管理」からインストールします。MODBUS に限らず、すべての追加ノードで共通の手順です。

  1. エディター右上の ≡ メニュー →「パレットの管理」を開く
  2. 「ノードを追加」タブを選ぶ
  3. 検索欄に node-red-contrib-modbus と入力
  4. 該当パッケージの「ノードを追加」ボタンを押す
  5. インストール完了後、左のパレットに「Modbus」カテゴリのノード群が現れる
💡 コマンドラインでも入れられる
Node-RED のユーザーディレクトリ（通常 ~/.node-red）で次を実行しても同じです。実行後は Node-RED の再起動が必要です。
cd ~/.node-red npm install node-red-contrib-modbus
⚠️ シリアル接続を使う場合
RS-485／USB シリアル変換器で実機につなぐ場合、OS 側にシリアルポートのドライバと権限が必要です。Linux では dialout グループへの所属（sudo usermod -a -G dialout $USER）が要ることがあります。まずは次章以降のシミュレータ（TCP）で動作を覚えるのがおすすめです。

3. MODBUS の基礎知識

3-1. RTU / ASCII / TCP（3 つの伝送方式）

MODBUS には主に 3 つの「運び方」があります。どれも“会話の中身（レジスタの読み書き）”は同じで、物理的な経路と誤り検出方法が違うだけです。

🔌 Modbus RTU

経路: シリアル（RS-485 / RS-232）

形式: バイナリ（コンパクト）

誤り検出: CRC-16（→ 用語集）

特徴: 1 本の線に複数のスレーブをぶら下げる。長距離・ノイズに強い。古くからの定番。

指定: ポート名・ボーレート・パリティ等

📝 Modbus ASCII

経路: シリアル（RS-485 / RS-232）

形式: ASCII 文字（人間も読める）

誤り検出: LRC（→ 用語集）

特徴: RTU よりデータ量が約 2 倍で低速。連続送信が苦手な機器・通信路向け（文字間に最大 1 秒の遅延が許される）。現代では RTU/TCP に置き換えが進んでおり採用例は少ない。

🌐 Modbus TCP

経路: イーサネット（LAN）

形式: バイナリ + TCP/IP ヘッダ

誤り検出: TCP 層が担当（Modbus 側にチェックサムなし）

特徴: IP アドレスとポート（既定 502）で指定。配線が楽で高速。新しい機器で主流。

指定: ホスト（IP）・ポート 502

💡 まずは TCP から
学習・検証は Modbus TCP が断然ラクです。実機がなくてもパソコン上のシミュレータに 127.0.0.1:10502 でつなげば、配線なしで一通り体験できます（第7章参照）。

3-2. データモデル（4 種類の入れ物）

MODBUS のスレーブは、データを 4 種類の「入れ物」に分けて持っています。「ビット（ON/OFF）か、ワード（16bit 数値）か」「読み取り専用か、書き込みもできるか」で 4 つに分かれます。

データ種別単位読み書きイメージ用途例
Coil（コイル）1 ビット読み書き可ON/OFF の出力スイッチリレー、ランプ、起動/停止
Discrete Input（離散入力）1 ビット読み取り専用ON/OFF の入力接点押しボタン、リミットスイッチ
Input Register（入力レジスタ）16 ビット読み取り専用測定された数値温度、電流などのセンサー値
Holding Register（保持レジスタ）16 ビット読み書き可設定できる数値目標温度、回転数の設定値

3-3. ファンクションコード（FC）の役割

ファンクションコードは「どの入れ物に・読みか書きか」を表す番号です。node-red-contrib-modbus では、ノードの FC ドロップダウンで選びます（読取系と書込系で選択肢が異なります）。

FC操作対象データノード設定値（dataType）
FC 1読み取りCoilCoil（FC 1: Read Coil Status）
FC 2読み取りDiscrete InputInput（FC 2: Read Input Status）
FC 3読み取りHolding RegisterHoldingRegister（FC 3: Read Holding Registers）
FC 4読み取りInput RegisterInputRegister（FC 4: Read Input Registers）
FC 5書き込み（1 点）CoilCoil（FC 5: Force Single Coil）
FC 6書き込み（1 点）Holding RegisterHoldingRegister（FC 6: Preset Single Register）
FC 15書き込み（複数）CoilMCoils（FC 15: Force Multiple Coils）
FC 16書き込み（複数）Holding RegisterMHoldingRegisters（FC 16: Preset Multiple Registers）

3-4. アドレスの「0 始まり」問題（つまずきポイント）

⚠️ 機器の仕様書の番地と、ノードに入れる番地が 1 ずれることがある
機器のマニュアルには「保持レジスタ 40001」のように書かれることが多いですが、これは人間向けの参照番号です。MODBUS プロトコル上の実際のアドレスは 0 始まりで、40001 はプロトコルアドレス 0 を指します。
node-red-contrib-modbus の Address 欄には、この0 始まりのプロトコルアドレスを入れます。値が読めるのに 1 つズレている、というときはまずここを疑ってください。
マニュアル表記（参照番号）データ種別ノードの Address 欄
40001Holding Register0
40002Holding Register1
30001Input Register0
10001Discrete Input0
00001Coil0

4. ノード構成

node-red-contrib-modbus は 14 個のノードを提供します。最初は次の主要 5 ノードだけ押さえれば十分です。

Modbus Read Modbus Getter Modbus Write Modbus Flex Getter Modbus Server
ノード入力役割たとえ
Modbus-Readなし（自動）設定した間隔で定期的に読み取る（ポーリング）定時の見回り
Modbus-Getterありメッセージが来た時だけ読み取る（オンデマンド）呼ばれたら確認
Modbus-Writeあり値を書き込む設定を変更
Modbus-Flex-GetterありFC・アドレス・数量を msg で動的に指定して読み取るその都度指示して確認
Modbus-Client（設定ノード）—接続先（TCP/シリアル）を定義。各ノードから共有参照する連絡先の住所録
📌 接続は「設定ノード」に集約される
Read / Getter / Write などの各ノードは、自分では接続先を持ちません。共通の Modbus-Client（設定ノード）を参照します。1 台の機器に対して設定ノードは 1 つ作り、複数の読み書きノードでそれを共有するのが基本です。
なお、1 つの設定ノードにぶら下げる読み書きノードが10 個を超えるときは、公式ヘルプでも Modbus-Flex-Getter の利用や接続の分割が推奨されています。

5. 設定項目一覧（プロパティ表）

node-red-contrib-modbus（v5.45.2）のエディターは英語表示のみで、日本語ロケールは提供されていません。そのため各項目は「英語ラベル（日本語訳）」の形で示します。設定画面はタブ（Settings と Optionals、設定ノードだけは Queues も）に分かれているので、本ガイドもそのタブの区切りに合わせて掲載します。まずは Settings タブの主要項目を押さえれば動かせます。Optionals タブはログ表示や IO ファイルなどの詳細設定で、多くは既定のままで構いません。

6. 実用的な使用パターン

パターン1: センサー値を定期収集してダッシュボード表示

Modbus Read → scale → chart

Modbus-Read で温度センサー（Input Register）を 5 秒ごとに読み、Function で実値にスケーリング（例: 生値 ÷ 10）して、Dashboard のチャートへ。最も典型的な構成です。

パターン2: ボタン操作で機器を起動/停止（書込）

button → Modbus Write → debug

Dashboard のボタンやスイッチで msg.payload = true を送り、Modbus-Write（FC5）で Coil を ON/OFF。装置の起動・停止やリレー制御に。

パターン3: 状態を要求時に確認（Getter）

「画面を開いたときだけ現在値を取りたい」「他の処理の途中で 1 回だけ読みたい」場合は、Modbus-Read（常時ポーリング）ではなく Modbus-Getter をトリガ起動で使います。通信量を抑えられます。

💡 読み取った値はどこに入る？
読取系ノードの 出力 1 の msg.payload に、読み取った値の配列（PDU）が入ります。あわせて msg.responseBuffer（生のバイト列）や元の入力 msg も付きます。出力 2 は同じ情報を順序違いで返すので、まずは出力 1 だけを使えば十分です。

7. サンプルフローとシミュレータの準備

📌 このサンプルは「シミュレータ前提」です
MODBUS は通信相手（スレーブ）がないと動きません。実機がなくても、下の3 つのいずれかで 127.0.0.1:10502 に擬似スレーブを立てれば、フローを動かして確認できます。

7-1. どの環境で動かす？（クラウド、ローカル、ラズパイ）

MODBUS は環境によって勝手が変わります。詳しくは 学習環境ガイド もどうぞ。

環境やり方ポート
☁️ 共有クラウドModbus-Server ノードで擬似スレーブを立てる。非特権ポート＋自分専用番号10502, 10503…（502 は不可）
🖥️ ローカルPCModRSsim2 等の GUI シミュレータ、または Modbus-Server502 でも自由
🍓 ラズパイRS-485 で実機接続（RTU）、または同上実機は通常 502
⚠️ なぜ本サンプルは 10502 ？
Modbus TCP の標準ポートは 502 ですが、502 は管理者専用の特権ポートで、共有クラウドの一般ユーザーでは開けません。そこで本ガイドのサンプルは非特権の 10502 を使います。自分のPC（管理者）やラズパイなら 502 のままでも構いません。共有サーバーでは他の学習者と衝突しないよう、自分専用の番号（10502, 10503…）にしてください。

8. 演習問題

演習1: ポーリング間隔を変える 初級

サンプルの Modbus-Read の Poll Rate を「5 秒」から「2 秒」に変更し、Debug の出力頻度が上がることを確認してください。

ヒント

Modbus-Read をダブルクリック → Poll Rate の数値を 2、単位を second(s) に。デプロイ後、Debug サイドバーの出力間隔を見ます。

演習2: 別のアドレスを読む 中級

Modbus-Getter の Address を 0 から 10 に、Quantity を 2 から 5 に変更し、Input Register の 10〜14 番地を読み取ってください。シミュレータ側にその範囲の値を用意しておきましょう。

ヒント

Address は 0 始まりです。マニュアルで「30011」と書かれていれば、Input Register のプロトコルアドレスは 10 になります（3-4 参照）。

演習3: 単一コイルを ON/OFF する 中級

新しい Modbus-Write を追加し、FC 5: Force Single Coil でアドレス 0 のコイルを ON/OFF してください。inject で msg.payload = true と false を送り分けます。

ヒント

FC を「FC 5」にすると単一コイル書込になります。inject の payload 型を boolean にして true / false を 2 つ用意すると切り替えやすいです。

演習4: Flex-Getter で動的に読む 上級

Modbus-Flex-Getter を使い、Function ノードで fc・address・quantity を組み立てて、読み取り対象を実行時に切り替えるフローを作ってください。

ヒント

5-5 の payload 形式を Function で生成して Flex-Getter に渡します。複数の番地を順番に読みたいときは、Inject や Split と組み合わせます。

9. まとめ

  • MODBUS は産業機器の共通言語。Node-RED では node-red-contrib-modbus を追加して使う（サードパーティ）。
  • 通信はマスター（Node-RED）がスレーブ（機器）に問い合わせるポーリング型。
  • 伝送方式は RTU（シリアル）と TCP（イーサネット）。学習は TCP が手軽。
  • データは Coil / Discrete Input / Input Register / Holding Register の 4 種類。読み書き可否で使い分ける。
  • 操作は ファンクションコード（FC）で指定。読取 FC1–4、書込 FC5/6/15/16。
  • Address は 0 始まり。マニュアルの参照番号（40001 等）と 1 ずれることに注意。
  • 接続先は Modbus-Client 設定ノードに集約し、Read/Getter/Write で共有する。
  • 実機がなくても Modbus-Server ノード / diagslave / ModRSsim2 で検証できる。

11. 用語集（追加リファレンス）

セクション 3「MODBUS の基礎知識」で扱いきれなかった、または別の場所に散らばっていた用語の補完リファレンスです。レジスタ・コイル・ファンクションコード等の主要概念は 第 3 章、通信全般の用語（クライアント/サーバー・プロトコル・ポートなど）は 通信の基礎ガイドの用語集 を参照してください。

Master / Slave（クライアント / サーバー）

MODBUS の役割を表す用語ですが、2 系統の呼び方が並行して使われています。

古い呼び名（伝統的）新しい呼び名（推奨）役割
マスター（Master）クライアント（Client）会話の主導権を握り、要求を出す側。Node-RED は通常こちら（node-red-contrib-modbus の Read / Write / Getter ノードはマスター動作）
スレーブ（Slave）サーバー（Server）要求に応える側。PLC・センサ・電力計などの被計測機器がこの役。Node-RED の Modbus-Server ノードを使えば Node-RED 自身をサーバー役にもできる
💡 Modbus.org（公式）は「Client / Server」を推奨 していますが、実機マニュアル・既存ドキュメント・node-red-contrib-modbus のプロパティ名（Unit-Id 等）は今も Master/Slave のまま のものが大半です。本ガイドでは混乱を避けるため両方を併記します。

Unit ID（スレーブ ID）

同じ通信路に複数のスレーブがぶら下がっているとき、どの機器宛ての要求かを示す番号。シリアル接続が前提の概念です。

  • 範囲：有効値は 1〜247（248〜255 は予約）。0 はブロードキャスト（全スレーブ宛て）として扱われる
  • マスター側にはアドレスが不要（送信元として暗黙）
  • Modbus TCP では本来不要（IP アドレスで宛先を識別できるため）ですが、Modbus シリアルを TCP にブリッジするゲートウェイ越しに「シリアル先のどの機器か」を伝える目的で残されています
  • 1 対 1 接続のときは 1 でほぼ問題ありません

エンディアン（Endian / ワード順序）

MODBUS のレジスタは 16 bit（1 ワード）単位ですが、温度（32 bit float）や 32 bit カウンタなど 1 レジスタに収まらない値を扱うときに問題になります。

32 bit 値を 2 つの 16 bit レジスタに分けて格納するとき、どちらのワードを上位にするかは機器メーカーごとに異なります。

並べ方32 bit 値 0x12345678 をレジスタ N と N+1 に
ビッグエンディアン（上位ワード先）レジスタ N: 0x1234、N+1: 0x5678。Modbus の標準仕様（バイト順）
リトルエンディアン（下位ワード先）レジスタ N: 0x5678、N+1: 0x1234。多くの PLC やパソコン CPU が内部で採用
⚠️ 値が読めるのに桁が大きすぎる／符号が逆／変な数になるときは、まずワード順を疑ってください。node-red-contrib-modbus 側でスワップする処理を Function ノードで挟むか、機器マニュアルの「Word Order」「Byte Swap」項目を確認します。

CRC-16 / LRC（誤り検出）

シリアル系の Modbus でメッセージが壊れていないかを検査する仕組み。受信側で計算して送信側と一致しなければ捨てます。

  • CRC-16（Cyclic Redundancy Check 16 bit）：Modbus RTU が使用。バイナリ全体から 16 bit の検査値を計算
  • LRC（Longitudinal Redundancy Check）：Modbus ASCII が使用。ASCII バイトの合計から 8 bit の検査値を計算
  • Modbus TCP：TCP 層自体がチェックサムを提供するため、Modbus 側には CRC/LRC を入れない

node-red-contrib-modbus やシミュレータは自動的に正しい CRC/LRC を付けてくれるため、利用者が手計算することは通常ありません。

PLC / シーケンサ / インバータ（産業機器）

用語意味
PLC（Programmable Logic Controller）
日本では「シーケンサ」と呼ぶことが多い		 工場の自動制御を担う産業用コンピュータ。センサー入力・スイッチ入力を受け、内蔵プログラム（ラダー言語等）に従ってモーター・バルブ・ランプなどを制御する。Modbus はこの PLC の値を Node-RED から読み書きするための窓口になる
インバータ（Inverter） 交流モーターの回転数を可変させる装置。本来は「直流を交流に変換する装置」だが、産業界では「VVVF インバータ」（Variable Voltage Variable Frequency）を指すことが多い。Modbus で目標周波数・運転指令を送ったり、現在電流値を読み取ったりする
RS-485 / RS-232 Modbus RTU/ASCII の物理層に使われるシリアル通信規格。RS-485 は 1 本のバスに最大 32 台ぶら下げられ、長距離（最大 1.2 km 程度）に強い。RS-232 は 1 対 1 接続向け
