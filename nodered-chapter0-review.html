① そもそもプログラムとは？
Q1. プログラムの基本的な動きを 3 段階で表すと、どれが正しいですか？
  • A入力 → 処理 → 出力
  • B開始 → 実行 → 停止
  • C設計 → 作成 → 販売
  • D命令 → 確認 → 保存
✅ 答えを見る
正解: A（入力 → 処理 → 出力）

プログラムは「データを受け取り（入力）」「加工して（処理）」「結果を返す（出力）」という 3 段階が基本。Excel の数式も、Node-RED のフローも、この構造は変わりません。

Q2. プログラムの「3 つの基本構造」として、次のうち正しい組み合わせはどれですか？
  • A入力・処理・出力
  • B順次・分岐・繰り返し

プログラムの流れ方は、この 3 つの組み合わせで表現できるとされています。Node-RED の配線も、ワイヤーで順次つなぐ／Switch で分岐する／Delay や Loop で繰り返す、として対応します。

① そもそもプログラムとは？
Q3. Node-RED でフローを流れる「メッセージ」は一般的に何と呼ばれますか？
  • Adata
  • Bevent
  • Cmsg
  • Dpayload
✅ 答えを見る
正解: C（msg）

Node-RED ではノード間を流れるオブジェクト全体を msg（メッセージ）と呼びます。その中身として msg.payload（本文）や msg.topic（タイトル）などのプロパティが入っています。

② イベント駆動とは？
Q4. Node-RED は次のうちどちらの方式で動いていますか？
  • A手続き型（上から順に実行して終了）
  • Bイベント駆動型（何かが起きたら動く）
  • C関数型（数式だけで構成）
  • Dオブジェクト指向型（クラスを書く）

正解: B（イベント駆動型）

Node-RED はデプロイ後、Inject ボタンが押された、タイマーが鳴った、MQTT にメッセージが届いた等の「イベント」が発生したときだけフローが動きます。何も起きなければ、ずっと待機状態です。

③ 最初の10分ハンズオン
Q7. Inject ノードの payload 種別で「数値」を選んで 25 を入れた場合と「文字列」を選んで 25 を入れた場合の違いは？
  • A違いはない。同じ 25 になる。
  • B「数値」は計算できる 25、「文字列」は文字としての "25"。
  • C「数値」は整数、「文字列」は小数も扱える。
  • D「文字列」の方が処理が速い。
✅ 答えを見る
正解: B（数値は計算できる 25、文字列は文字の "25"）

型の違いはプログラミングで最も重要な概念の 1 つ。Debug サイドバーでは、文字列は "25" のようにクオート付き、数値はクオート無しで表示されて区別できます。

③ 最初の10分ハンズオン
Q8. ダッシュボード（ui-text 等）を動かすのに必要な「設定ノード」はどれですか？
  • Aui-base のみ
  • Bui-page のみ
  • Cui-base, ui-theme, ui-page, ui-group をそれぞれ作る必要がある
  • Dダッシュボードは設定ノード不要
✅ 答えを見る
正解: C（ui-base, ui-theme, ui-page, ui-group が必要）

Node-RED Dashboard 2.0 はウィジェットをツリー構造で管理します。ui-base（ダッシュボード本体）→ ui-theme（見た目）／ui-page（ページ）→ ui-group（画面内の箱）→ 各ウィジェット、という階層が必要です。初回だけ設定すれば、後は使い回せます。

Q11. 「エッジで処理する」利点として当てはまらないものはどれ？
  • A応答が速い
  • B通信量が少なくて済む
  • Cネットが切れても動き続けられる

⑤ IoT全体像
Q12. MQTT の仕組みで「メッセージを仲介する存在」を何と呼びますか？
  • Aサーバー
  • Bブローカー
  • Cクライアント

3

複数の Inject から異なる値を送り、1 つの Debug で見分ける応用

🎯 ゴール: 「温度」用と「湿度」用の 2 つの Inject ボタンを作り、それぞれ違う値を送っても 1 つの Debug ノードで「これは温度」「これは湿度」と見分けられるようにする。
📋 要件:
  • 使用ノード: Inject × 2, Change × 0〜2（任意）, Debug × 1
  • 2 つの Inject はそれぞれ別の topic（例: temperature と humidity）を持つ
  • Debug ノードは「完全なメッセージ」を表示する設定にして、topic と payload の両方が見えるようにする
  • 1 つの Inject は数値（例: 25）、もう 1 つの Inject は文字列（例: 60%）を送る
  • どちらのボタンを押したかが Debug サイドバーの内容から一目で判別できる
💡 ヒント: Inject の設定画面で、payload の行の下に topic 入力欄があります（無ければ行追加で topic を足す）。Debug ノードは「対象」を「msg.payload」から「完全なメッセージ」に変えると、msg オブジェクト全体が表示されます。Change は使わなくても成立します（topic を Inject 側で設定すればよい）。
