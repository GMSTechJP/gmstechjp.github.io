📖 1. 概要

WebSocket ノードは、双方向リアルタイム通信を実現するためのノードです。HTTPと異なり、一度接続を確立すると、サーバーからクライアントへプッシュ送信が可能になります。

HTTP vs WebSocket

🔄 HTTP（リクエスト-レスポンス）

クライアントが毎回リクエストを送信

  • リクエスト → レスポンス → 接続終了
  • サーバーからプッシュ不可
  • オーバーヘッドが大きい

⚡ WebSocket（双方向通信）

一度接続したら維持

  • 接続確立 → 双方向通信 → 切断
  • サーバーからプッシュ可能
  • 低オーバーヘッド

📞 電話に例えると：HTTPは「毎回電話をかけ直す」、WebSocketは「電話をつなぎっぱなしにしておく」イメージです。つなぎっぱなしなので、相手からも自由に話しかけることができます。

📥 WebSocket In

WebSocket In

役割: WebSocketメッセージを受信

  • サーバーモード: クライアントからの接続受付
  • クライアントモード: サーバーへ接続

📤 WebSocket Out

WebSocket Out

役割: WebSocketメッセージを送信

  • 接続中のクライアント全員に送信
  • 特定クライアントに送信

⚙️ 2. WebSocket設定ノード

WebSocket In/Out ノードは、設定ノード（WebSocket-listener または WebSocket-client）を共有します。In/Out ノード側では「種類」で「待ち受け」（サーバー）か「接続」（クライアント）を選び、対応する設定ノードを指定します。

待ち受け（サーバー）

Node-REDがWebSocketサーバーとして動作し、クライアントからの接続を受け付けます。設定ノード WebSocket-listener の項目は次のとおりです。

設定項目説明既定値
パス 待ち受けるWebSocketエンドポイントのパス。Node-RED のベースURL（例 ws://localhost:1880）からの相対パスになります。 （必須・既定なし）
送信/受信 やり取りするデータの単位を選びます。「ペイロードを送信/受信」は msg.payload だけを送受信します。「メッセージ全体を送信/受信」は msg 全体をJSON文字列にして送受信します（→ 用語集）。 ペイロードを送信/受信
// クライアントからの接続URL ws://localhost:1880/ws/chat

接続（クライアント）

Node-REDがWebSocketクライアントとして、外部サーバーに接続します。設定ノード WebSocket-client の項目は次のとおりです。

設定項目説明既定値
URL 接続先WebSocketサーバーのURL。ws:// または wss:// で始めます。 （必須・既定なし）
TLS設定 wss:// で接続するときのTLS設定ノード。自己署名証明書やクライアント証明書を使う場合に指定します。 （なし）
サブプロトコル 接続時に提示するWebSocketサブプロトコル名。サーバーが特定のサブプロトコルを要求する場合に指定します。 （空）
送信/受信 やり取りするデータの単位。「ペイロードを送信/受信」か「メッセージ全体を送信/受信」を選びます（→ 用語集）。 ペイロードを送信/受信
ハートビートを送信 チェックを入れると、指定した秒間隔で Ping/Pong フレームを送り、接続を維持します。 オフ
ヘッダ 接続要求（HTTP Upgrade）時に付与するHTTPヘッダ。認証トークンなどを渡すときに使います。 （なし）

送信/受信の選択肢

選択肢説明用途
メッセージ全体を送信/受信msg全体をJSON文字列にして送受信Node-RED間通信
ペイロードを送信/受信msg.payload のみ送受信一般的なWebSocket通信

⚙️ 3. WebSocket In / Out ノード

WebSocket In ノード

受信時に設定される msg プロパティ：

プロパティ説明
msg.payload受信データ
msg._sessionセッションID（クライアント識別用）

WebSocket Out ノード

送信時に使用できる msg プロパティ：

プロパティ説明
msg.payload送信データ
msg._session特定クライアントに送信（省略時は全員）

💡 ポイント:

  • msg._session を指定すると特定のクライアントのみに送信
  • msg._session を削除すると全クライアントにブロードキャスト
  • WebSocket In で受け取った msg をそのまま WebSocket Out に渡すと、送信元に返信

📝 5. 演習問題

演習1: シンプルなエコーサーバー 初級

📋 課題: 受信したメッセージをそのまま送信元に返すエコーサーバーを作成してください。

🎯 要求仕様:

  • エンドポイント: /ws/myecho
  • 受信したデータをそのまま返信

✅ 成功の条件:

  • ブラウザのコンソールやWebSocketクライアントで ws://localhost:1880/ws/myecho に接続できる
  • 送信したメッセージがそのまま同じ内容で返ってくる（エコー）
  • デバッグパネルに受信メッセージが表示され、ステータスが「受信」になる
💡 ヒント

WebSocket In ノード:

  • 種類: 待ち受け
  • パス: /ws/myecho

WebSocket Out ノード: 同じ設定ノードを選択

WebSocket In の出力を直接 WebSocket Out に接続すれば、msg._session が保持されて送信元に返信されます。

演習3: ブロードキャスト配信 中級

📋 課題: Injectノードからメッセージを送信すると、接続中の全クライアントに配信されるフローを作成してください。

🎯 要求仕様:

  • Injectノードでトリガー（手動）
  • 送信メッセージ: {"type": "broadcast", "content": "全員へのお知らせ"}
  • 全クライアントに配信

✅ 成功の条件:

  • 複数のブラウザタブで ws://localhost:1880/ws/broadcast に接続できる
  • Inject ボタンを押すとすべての接続済みクライアントに同じメッセージが届く
  • 後から接続したクライアントも次の Inject 実行時にメッセージを受信できる
💡 ヒント

ブロードキャストのポイント:

  • WebSocket Out に渡す msg に _session がないと全員に送信されます
  • Changeノードの「削除」ルールで msg._session を削除すると、ブロードキャスト送信になります
  • Inject ノードから生成した msg には _session がないので、そのまま送信可能です

📋 課題: 複数のクライアントが参加できるチャットルームを作成してください。

  • クライアントからのメッセージ形式: {"name": "ユーザー名", "message": "内容"}
  • 全員に配信する形式: {"type": "chat", "name": "...", "message": "...", "timestamp": "..."}

✅ 成功の条件:

  • 複数のクライアントが ws://localhost:1880/ws/chatroom に同時接続できる
  • あるクライアントが送信したメッセージが全ての接続中クライアントに届く
  • デバッグパネルに type, name, message, timestamp を含むJSONが表示される
  • JSON 以外の文字列を送信しても「匿名」として正常に処理される
💡 ヒント

  1. WebSocket In で受信
  2. Function で JSON パース＆整形
  3. delete msg._session でブロードキャスト準備
  4. WebSocket Out で全員に送信

✅ 6. まとめ

🎯 重要ポイント:

  • 双方向通信: サーバーからクライアントへプッシュ送信が可能
  • サーバーモード: ws://localhost:1880/path で接続受付
  • クライアントモード: 外部WebSocketサーバーに接続
  • msg._session: あり→特定クライアント、なし→全員にブロードキャスト
  • データ形式: JSON文字列で送受信が一般的

⚠️ 注意事項:

  • WebSocket接続はステートフル（状態を持つ）なので、サーバー再起動時に切断される
  • クライアント側で再接続処理を実装することを推奨
  • 大量のクライアント接続時はメモリ使用量に注意
  • 外部公開時はWSS（WebSocket Secure）を使用

📘 9. 用語集（WebSocket 固有の用語）

WebSocket の仕様（RFC 6455）特有の用語をまとめました。クライアント／サーバー、プロトコル、ポートなど通信全般の用語は 通信の基礎ガイドの用語集 を参照してください。

🤝 オープニングハンドシェイク（HTTP Upgrade）

WebSocket 接続は 最初の 1 回だけ HTTP リクエスト として始まり、サーバーが受け入れるとその同じ TCP 接続を WebSocket プロトコルに切り替える仕組みです。

  • クライアント: GET /path HTTP/1.1 に Upgrade: websocket ヘッダを付けて送信
  • サーバー: 受け入れる場合 HTTP/1.1 101 Switching Protocols を返す
  • 以後は同じソケット上で 双方向のフレーム送受信 が始まる
  • HTTP ポート（80 / 443）で動くため、HTTP プロキシや CDN を経由できるのが大きな利点
💡 通信基礎ガイドの ハンドシェイク は TCP の 3 ウェイハンドシェイクが主ですが、WebSocket のオープニングハンドシェイクは その上に乗る HTTP レベルの追加の握手 です。

🔄 ステートフル(stateful)接続

WebSocket は 「いったん繋いだら接続を保ち続ける」 通信です。これに対し HTTP は 「1 リクエスト 1 レスポンスで毎回切断」 するステートレス（stateless）通信です。

HTTP（ステートレス）WebSocket（ステートフル）
接続リクエストごとに張り直す1 回張ったら保持し続ける
サーバーからの送信不可（クライアントが聞きに来るのを待つ）いつでも push できる
サーバー再起動の影響次のリクエストで自動復帰全クライアントが切断される
状態の保持セッション情報の管理が別途必要接続が状態そのもの（msg._session でクライアント識別可能）

📦 フレーム（Text / Binary）

WebSocket は 「メッセージ単位」で送受信 します。データはバイト列に区切られた フレーム として運ばれ、用途に応じて 2 種類があります。

  • テキストフレーム: UTF-8 エンコードの文字列。JSON 文字列のやり取りで最も使われる
  • バイナリフレーム: 任意のバイト列。画像・音声・カスタムプロトコルなど

大きなメッセージは複数のフレームに 分割（fragmentation） されて送られることもありますが、受信側ではアプリには 1 つのメッセージとして見えます。

💓 Ping / Pong フレーム（キープアライブ）

長時間データのやり取りがない場合、ネットワーク機器（プロキシ・NAT・ルーター）が 「もう使われていない」と判断して接続を切る ことがあります。これを防ぐのが Ping / Pong フレーム。

  • Ping（制御フレーム）: 一方が「生きてる？」と問いかける
  • Pong（制御フレーム）: 受け取った側はすぐに同じペイロードを Pong として返す（仕様で必須）
  • 用途: キープアライブ／死活確認／遅延（レイテンシ）測定
💡 Node-RED の WebSocket 設定にある 「ハートビートを送信」（クライアント設定の項目。秒数を指定）が、この Ping を一定間隔で自動送信する機能です。クラウド経由や Cloudflare 経由などで切断が起きやすい環境では、間隔を短くすることで安定性が増します。

👋 クローズフレームとクローズコード

接続を終了するときは、どちらかが クローズフレーム を送り、理由を クローズコード（4 桁）で示します。

コード意味
1000正常終了（Normal Closure）
1001サーバー停止・ブラウザ閉じた等（Going Away）
1006異常切断（クローズフレームなしの切断）
1008ポリシー違反
1011サーバー内部エラー

Node-RED の WebSocket では、切断時の close コードは msg._session 内などで参照できる場合があります（実装による）。

🔁 再接続戦略（reconnection）

WebSocket 仕様（RFC 6455）には 再接続のルールは含まれていません。「切れたら繋ぎ直す」のは アプリ側で実装する責務 です。

  • Node-RED の WebSocket Out（client モード）は切断時に自動で再接続を試みる実装になっている
  • ブラウザ側 JavaScript で書く場合は、onclose ハンドラ内で setTimeout で再接続するのが定石
  • 無限ループ・即時再接続は避ける ─ 指数バックオフ（1秒→2秒→4秒→…最大30秒）にして、サーバー過負荷時の雪崩を防ぐ
  • 「サーバー再起動時に全クライアントが切断され、一斉再接続でサーバーが落ちる」のは典型的なハマりどころ

📝 ペイロードを送信/受信 と メッセージ全体を送信/受信（Node-RED 固有の選択肢）

Node-RED の WebSocket 設定ノードでは、「送信/受信」の欄で 送受信するデータの形を 2 つから選びます。

選択肢送受信される内容用途
ペイロードを送信/受信msg.payload のみ通常の WebSocket 通信（ブラウザの new WebSocket() と素直に互換）
メッセージ全体を送信/受信msg オブジェクト全体を JSON 化Node-RED 同士の通信（topic 等の付加情報も丸ごと運びたい）
⚠️ Node-RED 独自の挙動: 「メッセージ全体を送信/受信」は WebSocket 標準ではなく Node-RED の便利機能です。ブラウザ等の外部クライアントから JSON 文字列として届く ため、受け取り側で JSON.parse() が必要になります。
