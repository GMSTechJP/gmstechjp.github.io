📖 1. 概要

WebSocket ノードは、双方向リアルタイム通信を実現するためのノードです。HTTPと異なり、一度接続を確立すると、サーバーからクライアントへプッシュ送信が可能になります。

HTTP vs WebSocket

🔄 HTTP（リクエスト-レスポンス）

クライアントが毎回リクエストを送信

  • リクエスト → レスポンス → 接続終了
  • サーバーからプッシュ不可
  • オーバーヘッドが大きい

⚡ WebSocket（双方向通信）

一度接続したら維持

  • 接続確立 → 双方向通信 → 切断
  • サーバーからプッシュ可能
  • 低オーバーヘッド

📞 電話に例えると：HTTPは「毎回電話をかけ直す」、WebSocketは「電話をつなぎっぱなしにしておく」イメージです。つなぎっぱなしなので、相手からも自由に話しかけることができます。

📥 WebSocket In

WebSocket In

役割: WebSocketメッセージを受信

  • サーバーモード: クライアントからの接続受付
  • クライアントモード: サーバーへ接続

📤 WebSocket Out

WebSocket Out

役割: WebSocketメッセージを送信

  • 接続中のクライアント全員に送信
  • 特定クライアントに送信

⚙️ 2. WebSocket設定ノード

WebSocket In/Out ノードは、設定ノード（WebSocket-listener または WebSocket-client）を共有します。

サーバーモード（Listen）

Node-REDがWebSocketサーバーとして動作し、クライアントからの接続を受け付けます。

プロパティ説明例
種類Listen at（サーバー）-
パスWebSocketエンドポイント/ws/chat
送受信データ形式entire message / payload
// クライアントからの接続URL ws://localhost:1880/ws/chat

クライアントモード（Connect）

Node-REDがWebSocketクライアントとして、外部サーバーに接続します。

プロパティ説明例
種類Connect to（クライアント）-
URL接続先WebSocketサーバーws://example.com/socket
ハートビート送信キープアライブ送信25秒ごと

Send/Receive オプション

オプション説明用途
entire messagemsg全体をJSON化して送受信Node-RED間通信
payload onlymsg.payloadのみ送受信一般的なWebSocket通信

⚙️ 3. WebSocket In / Out ノード

WebSocket In ノード

受信時に設定される msg プロパティ：

プロパティ説明
msg.payload受信データ
msg._sessionセッションID（クライアント識別用）

WebSocket Out ノード

送信時に使用できる msg プロパティ：

プロパティ説明
msg.payload送信データ
msg._session特定クライアントに送信（省略時は全員）

💡 ポイント:

  • msg._session を指定すると特定のクライアントのみに送信
  • msg._session を削除すると全クライアントにブロードキャスト
  • WebSocket In で受け取った msg をそのまま WebSocket Out に渡すと、送信元に返信

使用パターン

パターン1: エコーサーバー（受信→返信）

サンプルフローの「エコーサーバー（受信→返信）」を参照してください。

WS In → 処理 → WS Out

ポイント:

  • msg._session が保持されていると、送信元クライアントにのみ返信
  • 受信したメッセージを加工して返信可能

パターン2: ブロードキャスト（全員に配信）

サンプルフローの「ブロードキャスト（全員に配信）」を参照してください。

データ生成 → _session削除 → WS Out

ポイント:

  • msg._session を削除すると、接続中の全クライアントに送信
  • センサーデータなどのプッシュ配信に最適

パターン3: サーバーからのプッシュ配信

サンプルフローの「サーバーからのプッシュ配信」を参照してください。

定期実行 → データ生成 → 全員に配信

ポイント:

  • Injectノードのrepeat設定で定期的にデータを配信
  • リアルタイムモニタリングやダッシュボード更新に活用

パターン4: ブラウザテスト用HTMLページ

サンプルフローの「ブラウザテスト用HTMLページ」を参照してください。

HTTP In → テストHTML → HTTP Response

ポイント:

  • WebSocket接続をテストするためのHTMLページを提供
  • ブラウザから /ws-test にアクセスしてテスト可能
  • JavaScript の WebSocket API でサーバーに接続

📝 5. 演習問題

演習1: シンプルなエコーサーバー 初級

📋 課題: 受信したメッセージをそのまま送信元に返すエコーサーバーを作成してください。

🎯 要求仕様:

  • エンドポイント: /ws/myecho
  • 受信したデータをそのまま返信

✅ 成功の条件:

  • ブラウザのコンソールやWebSocketクライアントで ws://localhost:1880/ws/myecho に接続できる
  • 送信したメッセージがそのまま同じ内容で返ってくる（エコー）
  • デバッグパネルに受信メッセージが表示され、ステータスが「受信」になる
💡 ヒント

WebSocket In ノード:

  • Type: Listen at
  • Path: /ws/myecho

WebSocket Out ノード: 同じ設定ノードを選択

WebSocket In の出力を直接 WebSocket Out に接続すれば、msg._session が保持されて送信元に返信されます。

📋 課題: 受信したメッセージに現在時刻を追加して返信してください。

  • 返信: {"message": "こんにちは", "receivedAt": "2024-01-01T12:00:00.000Z"}

  • 返信に message（送信した内容）と receivedAt（ISO 8601形式の日時）が含まれている
  • 複数のメッセージを送るたびに receivedAt の値が更新される

演習3: ブロードキャスト配信 中級

📋 課題: Injectノードからメッセージを送信すると、接続中の全クライアントに配信されるフローを作成してください。

🎯 要求仕様:

  • Injectノードでトリガー（手動）
  • 送信メッセージ: {"type": "broadcast", "content": "全員へのお知らせ"}
  • 全クライアントに配信

✅ 成功の条件:

  • 複数のブラウザタブで ws://localhost:1880/ws/broadcast に接続できる
  • Inject ボタンを押すとすべての接続済みクライアントに同じメッセージが届く
  • 後から接続したクライアントも次の Inject 実行時にメッセージを受信できる
💡 ヒント

ブロードキャストのポイント:

  • WebSocket Out に渡す msg に _session がないと全員に送信されます
  • Changeノードの「削除」ルールで msg._session を削除すると、ブロードキャスト送信になります
  • Inject ノードから生成した msg には _session がないので、そのまま送信可能です

演習4: 簡易チャットシステム 上級

📋 課題: 複数のクライアントが参加できるチャットルームを作成してください。

🎯 要求仕様:

  • クライアントからのメッセージ形式: {"name": "ユーザー名", "message": "内容"}
  • 全員に配信する形式: {"type": "chat", "name": "...", "message": "...", "timestamp": "..."}
  • 接続/切断時も通知（任意）

✅ 成功の条件:

  • 複数のクライアントが ws://localhost:1880/ws/chatroom に同時接続できる
  • あるクライアントが送信したメッセージが全ての接続中クライアントに届く
  • デバッグパネルに type, name, message, timestamp を含むJSONが表示される
  • JSON 以外の文字列を送信しても「匿名」として正常に処理される
💡 ヒント

処理の流れ:

  1. WebSocket In で受信
  2. Function で JSON パース＆整形
  3. delete msg._session でブロードキャスト準備
  4. WebSocket Out で全員に送信

✅ 6. まとめ

🎯 重要ポイント:

  • 双方向通信: サーバーからクライアントへプッシュ送信が可能
  • サーバーモード: ws://localhost:1880/path で接続受付
  • クライアントモード: 外部WebSocketサーバーに接続
  • msg._session: あり→特定クライアント、なし→全員にブロードキャスト
  • データ形式: JSON文字列で送受信が一般的

⚠️ 注意事項:

  • WebSocket接続はステートフル（状態を持つ）なので、サーバー再起動時に切断される
  • クライアント側で再接続処理を実装することを推奨
  • 大量のクライアント接続時はメモリ使用量に注意
  • 外部公開時はWSS（WebSocket Secure）を使用

🔧 7. トラブルシューティング

症状原因解決方法
接続できない URLが間違っている ws://（httpsならwss://）を確認
すぐ切断される サーバー側の問題 Node-REDのログを確認
メッセージが届かない 設定ノードが異なる In/Outで同じ設定ノードを使用
全員に送信されない _sessionが残っている delete msg._session
JSONパースエラー データ形式の問題 try-catchでエラーハンドリング
