1. 概要

Node-RED には HTTP通信に関連する3つのノードが用意されています。これらを組み合わせることで、WebサーバーやAPIクライアントの機能を実装できます。

💡 そもそも「API」とは？

API（Application Programming Interface）とは、プログラム同士がデータをやり取りするための「窓口」です。レストランに例えると分かりやすくなります。

  • あなた（クライアント）が、お店（サーバー）の店員（API）に注文（リクエスト）を伝える
  • 店員が注文を受け取り、厨房で処理して料理を用意する
  • できあがった料理（レスポンス）があなたのもとに届く

このガイドで扱うHTTPは、この「注文のやり取り」に使われる共通のルール（プロトコル）です。「APIを呼び出す」とは、決まった形式でリクエストを送り、レスポンスを受け取ることを指します。

HTTP In

HTTP In

役割: HTTPリクエストを受信（サーバー機能）

URLエンドポイントを定義し、外部からのリクエストを受け付けます

HTTP Response

HTTP Response

役割: HTTPレスポンスを送信（サーバー機能）

リクエストに対してステータスコードとデータを返します

HTTP Request

HTTP Request

役割: 外部APIにリクエストを送信（クライアント機能）

外部のWebサービスやAPIを呼び出します

重要: HTTP In と HTTP Response はペアで使用します。HTTP Response がないと、リクエスト元はタイムアウトまで待ち続けます。

基本フローパターン

サーバー機能（API提供）:

HTTP In → 処理 → HTTP Response

クライアント機能（API呼び出し）:

Inject → HTTP Request → Debug

2. HTTP In ノード

主要プロパティ

プロパティ説明例
メソッド受け付けるHTTPメソッドGET, POST, PUT, DELETE
URLエンドポイントのパス/api/data, /users/:id
名前ノードの表示名GET /api/sensors

💡 URLパラメータ（補足）: 上の表の「URL」に :パラメータ名 を含めると、動的なパスを定義できます。

// URL設定: /users/:userId/posts/:postId // リクエスト: GET /users/123/posts/456 // msg.req.params の内容: { "userId": "123", "postId": "456" }

出力される msg のプロパティ（HTTP In が自動設定）

プロパティ説明例

msg.req.headersリクエストヘッダー{"content-type": "..."}
msg.req.bodyリクエストボディ（生）POSTデータ

エンドポイントのアクセス方法

http://localhost:1880/[設定したURL]

例: URLを /api/hello に設定した場合

http://localhost:1880/api/hello

コード意味用途

401Unauthorized認証必要
404Not Foundリソースが存在しない

主要プロパティ

プロパティ説明デフォルト
メソッドHTTPメソッド（GET/POST/PUT/DELETE等）GET
URLリクエスト先のURL（空）

Method（HTTPメソッド）オプション

オプション用途ボディ
GETデータ取得、検索なし
POSTデータ送信、新規作成あり
PUTデータ更新（全体置換）あり
DELETEデータ削除任意

Return（レスポンス形式）オプション

オプション説明用途
UTF-8文字列文字列として受信テキスト、HTML、JSON
バイナリバッファBufferオブジェクト画像、ファイル
JSONオブジェクト自動でJSONパースREST API

動的なURL設定

URLを空にすると、msg.url でURLを動的に指定できます：

// Function ノードで動的にURL設定 msg.url = "https://api.example.com/users/" + msg.payload.userId; return msg;

動的なヘッダー設定

// Function ノードでヘッダーを設定 msg.headers = { "Content-Type": "application/json", "Authorization": "Bearer " + flow.get("apiToken") }; return msg;

5. 実用的な使用パターン

📥 サンプルフローを試す手順:

フローJSONは、Node-REDエディタの「読み込み」画面に貼り付けるためのものです。ブラウザの開発者ツール（Console）に貼り付けても何も起きません。

  1. Node-REDを起動し、ブラウザでエディタを開く（http://localhost:1880）
  2. 下の「公式サンプルフローJSON」を「コピー」ボタンでコピー
  3. エディタ右上の メニュー → 読み込み を選択し、JSONを貼り付けて「読み込み」をクリック
  4. エディタ右上の「デプロイ」ボタンをクリック（クリックするまでフローは動きません）
  5. 後述の「動作確認方法」の手順で、動作を確認する

動作確認方法

⚠️ 動作確認の前提: フローを読み込んで「デプロイ」ボタンをクリックした後に行ってください。デプロイ前はエンドポイントがまだ存在しないため、すべて失敗します。

例1〜5は「エンドポイント（http in → http response）」と「それを呼び出すテスト用フロー（inject → http request → debug）」のセットになっており、エディタ内とブラウザの両方で確認できます。例6・7はブラウザから直接確認します。

  1. 例1〜5: 各例の injectノードの左端のボタンをクリックすると、debugサイドバー（右パネルの虫アイコン）に、エンドポイントが返したレスポンス（HTMLやテキスト）が表示されます
  2. ブラウザでも確認できます: http://localhost:1880/hello、http://localhost:1880/hello-query?name=Taro、http://localhost:1880/hello-param/Taro、http://localhost:1880/hello-headers を開くと、フローが生成したページが表示されます
  3. 例6（ファイルアップロード）: ブラウザで http://localhost:1880/hello-upload を開き、ファイルを選択して送信すると、受信結果が表示されます
  4. 例7（Cookie）: ブラウザで http://localhost:1880/hello-cookie を開くと現在のCookie一覧が表示され、ページ内の「Add a cookie」「Clear cookies」リンクで追加・削除を試せます

🖥️ 本ガイドはローカル環境前提です: サンプルフローのHTTP Requestノードは http://localhost:1880/... へ接続する（自分自身のAPIを呼び出す）設定です。httpsで公開された共有クラウド環境では、この接続先が存在しないため公式サンプルは動作しません（同じループバック構成のwebsocketガイドで実測確認済み）。自分のPCにインストールしたNode-REDで学習してください（Windowsインストールガイド／Raspberry Piインストールガイド）。なお、Raspberry Pi上のNode-REDを別のPCのブラウザから操作している場合、ブラウザで開くURLの localhost はPiのアドレス（例: http://raspberrypi.local:1880）になります。フロー内部の http://localhost:1880/... はPi自身への接続なので、そのままで動きます。

応用フローの動作確認

作成したAPIの動作確認方法:

応用フローをデプロイしたら、以下のいずれかの方法でHTTPリクエストを送って動作を確認できます。初心者の方は、まず「HTTP Request ノード」か「ブラウザ」から試すのがおすすめです。

  • HTTP Request ノード（追加ツール不要）: 上の応用フローの「セルフテスト」部分のように、Node-RED内にInject→HTTP Requestノードを置いて自分のAPIを呼び出します。新しいツールを覚える必要がなく、GET以外のメソッドも試せます。
  • ブラウザ: GETリクエストのみですが、アドレスバーにURL（例: http://localhost:1880/api/hello）を入力するだけで確認できます。POST/PUT/DELETEはブラウザから直接送信できません。
  • curl（任意・コマンドライン操作に慣れている場合）: ターミナル/コマンドプロンプトからHTTPリクエストを送れるコマンドラインツールです。多くのOSに標準搭載されています。例: curl http://localhost:1880/api/hello（オプションを付けない場合は常にGETリクエストになります。POST/DELETE等を送るには下記のオプション早見表を参照）。
  • Postman（任意・GUIで操作したい場合）: 無料でダウンロードできる、画面操作でHTTPリクエストを組み立てて送信できる専用アプリです。POSTのボディ入力なども視覚的に行えます。

7. まとめ

重要ポイント:

  • HTTP In + HTTP Response はペアで使用（サーバー機能）
  • HTTP Request は外部APIを呼び出すクライアント機能
  • URLパラメータ: :id 形式で定義 → msg.req.params.id で取得
  • クエリパラメータ: ?key=value → msg.req.query.key で取得
  • POSTボディ: msg.payload に自動でパース
  • ステータスコード: msg.statusCode で設定と確認
  • ヘッダー: msg.headers でContent-Type等を設定
  • 動的URL: HTTP Request で msg.url を設定すると実行時にURL指定可能

注意事項:

  • HTTP Response を忘れるとタイムアウトになる
  • 外部公開時は認証と認可を実装すること
  • 本番環境ではHTTPSを使用（リバースプロキシ等）
  • 外部APIにはレート制限がある場合が多い（連続呼び出しに注意）
  • エラーハンドリングは必須（Catchノードとの組み合わせ推奨）
  • APIキーなど機密情報は環境変数で管理

CORSヘッダーの追加方法

💡 CORSとは: Cross-Origin Resource Sharing（クロスオリジン・リソース・シェアリング）の略です。ブラウザには「今見ているWebページと違うドメイン（オリジン）のAPIには、無断でJavaScriptからアクセスさせない」というセキュリティの仕組みがあります。これをクロスオリジン制限と呼び、制限に引っかかって通信がブロックされることをCORSエラーと呼びます。サーバー側（Node-RED）が「このオリジンからのアクセスは許可します」という許可証にあたるレスポンスヘッダーを付けて返すことで、ブラウザは通信を許可するようになります。

// Functionノードで設定 msg.headers = { "Content-Type": "application/json", "Access-Control-Allow-Origin": "*", "Access-Control-Allow-Methods": "GET, POST, PUT, DELETE", "Access-Control-Allow-Headers": "Content-Type" }; return msg;
