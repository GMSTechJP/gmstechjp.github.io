📖 1. 概要

Link ノードは、フロー内やフロー間で「仮想的なワイヤー」を作成するノードです。物理的なワイヤーを引かずにノード同士を接続でき、複雑なフローを整理するのに役立ちます。

📞 内線電話に例えると：Link ノードは会社の内線電話システムのようなものです。離れた部署（別のフロータブ）にいる人でも、内線番号（Link名）を使えば直接つながることができます。

💡 Link ノードを使うメリット:

  • ワイヤーの交差を減らしてフローを見やすくする
  • 異なるタブ（フロー）間でメッセージを送受信
  • 共通処理を再利用可能にする
  • サブルーチンのような呼び出しが可能（Link Call）

🔧 2. 3種類のLinkノード

📥 Link In

Link In

役割: メッセージの受信口

特徴:

  • Link Out または Link Call からメッセージを受け取る
  • 複数の Link Out から受信可能

📤 Link Out

Link Out

役割: メッセージの送信口

特徴:

  • Link In にメッセージを送信
  • 複数の Link In に同時送信可能

📞 Link Call

Link Call

役割: サブフローを呼び出して結果を受け取る

特徴:

  • Link In を呼び出し、Link Out（return）で結果を受け取る
  • 関数呼び出しのような動作

Link Out と Link Call の違い

特徴Link Out（通常モード）Link Call
動作送りっぱなし（Fire and Forget）結果を待って受け取る
戻り値なしあり（Link Out return経由）
用途通知、ログ記録、分岐共通処理の呼び出し
接続先Link InLink In（return用Link Out必須）

使用パターン

パターン1: 基本的な Link Out → Link In

サンプルフローの「基本: Link Out → Link In」を参照してください。

Inject → Link Out - - - Link In → Debug

ポイント: ワイヤーの交差を避けたり、離れた場所にメッセージを送る

パターン2: Link Call（サブルーチン呼び出し）

サンプルフローの「Link Call（サブルーチン呼び出し）」を参照してください。

Inject → Link Call → 結果
【呼び出し先】 共通処理（3倍） → 処理 → Return

ポイント:

  • Link Call は呼び出し先から結果が返るまで待機
  • Link Out の Mode を「Return」に設定すると結果を返せる

パターン3: 複数の送信先

サンプルフローの「複数の送信先」を参照してください。

Link Out - - - 受信A
- - - 受信B

ポイント: 1つの Link Out から複数の Link In に送信可能

パターン4: 共通検証処理の再利用

サンプルフローの「検証処理（共通）」「テスト呼び出し」を参照してください。

テストA → 検証呼出 → 結果
テストB → 検証呼出 → 結果

ポイント: 同じ検証ロジックを複数箇所から呼び出して再利用

📝 5. 演習問題

演習1: 基本的なLink接続 初級

📋 課題: Link Out と Link In を使って、ワイヤーを引かずにメッセージを送信するフローを作成してください。

✅ 成功の条件:

  • Inject と Link Out の間、Link In と Debug の間のみワイヤーで接続されており、Link Out と Link In の間にワイヤーが引かれていない
  • Injectボタンをクリックすると、デバッグサイドバーにメッセージが表示される（Linkを経由してメッセージが届いている）
  • Link Out の設定を開くと、接続先の Link In ノードが選択されている
  • Debugノードのステータス欄にも受信したメッセージが表示される
💡 ヒント

1. Link Out ノードの設定で、接続先の Link In を選択します

2. 名前を分かりやすくつけると管理しやすくなります

演習2: 複数の受信先 初級

📋 課題: 1つの Link Out から2つの Link In にメッセージを送信するフローを作成してください。

✅ 成功の条件:

  • Injectボタンを1回クリックすると、デバッグサイドバーに2件の出力が追加される（1つの送信が2つの受信先に届いている）
  • 「処理A実行」と「処理B実行」の両方のDebugノードにそれぞれメッセージが表示される
  • Link Out の設定を開くと、2つの Link In ノードが接続先として選択されている
  • フロー上の物理的なワイヤーは最小限（Link Out と Link In の間にワイヤーなし）になっている

演習3: Link Call の基本 中級

📋 課題: Link Call を使って「入力値を3倍にする」共通処理を作成し、呼び出すフローを作ってください。

✅ 成功の条件:

  • 「7を入力」ボタンをクリックすると、デバッグサイドバーに 21（7×3）が表示される
  • Debugノードのステータス欄に 21 が表示される
  • Link Callノードへの出力（7）とDebugへの出力（21）が別のノードを経由しており、共通処理（3倍）が分離されている構成になっている
  • 共通処理の Link Out のモードが「return」になっており、Link Call に結果が戻ってくる
💡 ヒント

1. 共通処理: Link In → Function（×3） → Link Out（returnモード）

2. 呼び出し側: Inject → Link Call → Debug

3. Link Out のモードを「return」に設定することが重要です

演習4: 再利用可能な検証処理 上級

📋 課題: 「入力値が正の数かどうかを検証する」共通処理を Link Call で作成し、複数箇所から呼び出せるようにしてください。

🎯 要求仕様:

  • 正の数: msg.valid = true, msg.payload = 元の値
  • 負の数または0: msg.valid = false, msg.error = "正の数を入力してください"

✅ 成功の条件:

  • 「正の数: 10」ボタンをクリックすると、デバッグサイドバーに msg.valid = true、msg.payload = 10 を含む結果が表示される
  • 「負の数: -5」ボタンをクリックすると、msg.valid = false、msg.error = "正の数を入力してください" を含む結果が表示される
  • 両方のInjectボタンが同じLink Call（同じ検証共通処理）を呼び出しており、検証ロジックが1箇所に集約されている
  • Debugノードのステータス欄に msg.valid の値（true / false）が表示される
💡 ヒント

Function ノードで条件分岐を行い、結果を msg に設定して返します。

✅ 6. まとめ

🎯 重要ポイント:

  • Link Out → Link In: ワイヤーなしでメッセージを送信（一方向）
  • Link Call: サブルーチン呼び出しのように結果を受け取れる
  • 異なるタブ間でもメッセージを送受信可能
  • 複雑なフローを整理してメンテナンス性を向上

⚠️ 注意事項:

  • Link Call を使う場合、呼び出し先に Link Out（returnモード）が必要
  • 循環参照（A→B→A）を作らないよう注意
  • タイムアウト設定を適切に行う（デフォルト30秒）
  • 名前を分かりやすくつけて管理しやすくする
