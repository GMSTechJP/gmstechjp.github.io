1. IoT とは「モノがインターネットにつながる」仕組み

IoT（Internet of Things, アイ・オー・ティー）とは、センサーや家電、産業機器などの「モノ」がインターネットに接続され、データをやり取りできるようになる仕組みの総称です。和訳では「モノのインターネット」と言われます。

身近な例だけでも、エアコンの遠隔操作、スマートウォッチの心拍測定、見守りカメラ、工場の稼働監視、駐車場の空き検知――すべて IoT の応用例です。

💡 「スマホと何が違うの？」
スマホやパソコンは「人が操作してネットと会話する端末」。IoT は「人の操作なしに、機器やセンサーが勝手にデータを送受信する」点が違います。これは①②で学んだ「イベント駆動」の考え方そのもの。温度が変化したら送る、ボタンが押されたら送る、といった自動化が本質です。

2. IoT システムの 5 つの階層

どんな IoT システムも、ざっくり分けると次の 5 つの階層で成り立っています。各階層が担う役割を押さえると、自分が今どの階層をいじっているかが分かるようになります。

① センサー層
温度・湿度・振動・光・人感・電力など、現実世界の情報を数値化するハードウェア。
⬇ ケーブル／近距離無線
② エッジ層
センサーに近い場所（現場）で動く小型コンピュータ。例: Raspberry Pi、Jetson。データを集めて整え、必要なら即座に判断して動作する。
⬇ インターネット（MQTT, HTTP）
③ 通信層
エッジとクラウドを結ぶパイプ。MQTT（軽量メッセージ）、HTTP/HTTPS、WebSocket など。
⬇ メッセージ受信
④ クラウド層
大量のデータを蓄え、分析し、通知や保存を担う。AWS、Azure、Google Cloud など。
⬇ 表示要求
⑤ UI 層
人間が見る画面。ダッシュボード、スマホアプリ、メール／LINE 通知など。
📌 すべてを 1 台でやる必要はない
規模が小さければ、エッジだけで全階層をこなすこともあります（センサーを繋いで、Node-RED で処理して、Node-RED のダッシュボードで表示する）。規模が大きくなるほど、クラウドを絡めた分散構成が必要になります。

3. Node-RED はどこで活躍するのか？

Node-RED の強みは、複数の階層をまたいで「つなぐ役目」をこなせることです。具体的には次の 3 つのポジションが代表的です。

🔷 Node-RED が活躍する 3 つのポジション

  1. エッジ層（Raspberry Pi / Jetson 上）: センサーから値を読み、ちょっとした処理を入れて、クラウドに送る。現場で動く IoT ゲートウェイの中心。
  2. クラウド層（AWS 等に Node-RED を立てる）: 複数のエッジから届いたデータを受け取り、条件に応じて保存・通知・集計する。
  3. UI 層（ダッシュボード）: 集まったデータをグラフやカードで可視化する。Node-RED Dashboard 2.0 を使うと、コード無しで画面が作れる。

つまり Node-RED は、どの階層にも顔を出せる万能選手です。これが「GUI でデータを配線できる」利点を最大限に活かしたアーキテクチャです。

4. 登場人物紹介：IoT で出てくる主要ワード

センサー 入力

現実世界の物理量を電気信号に変換する部品。温度、湿度、圧力、照度、加速度、電流など。

アクチュエータ 出力

電気信号で現実世界に影響を与える部品。モーター、リレー、LED、スピーカーなど。

エッジデバイス 処理

現場で動く小型コンピュータ。Raspberry Pi、Jetson、Arduino、ESP32 など。

ゲートウェイ 中継

現場のセンサーをまとめてクラウドへ送る中継機。多くの場合、エッジデバイスが兼任。

MQTT 通信

IoT 標準の軽量メッセージプロトコル。「ブローカー」を経由して、送信者と受信者が疎結合でやり取りできる。

ブローカー 仲介

MQTT メッセージを仲介するサーバー。送信者は「どのトピックに出す」、受信者は「どのトピックを聞く」と宣言するだけで繋がる。

クラウド 蓄積

AWS・Azure・Google Cloud 等のオンライン計算資源。大量データの保管・分析・通知・可視化に使う。

ダッシュボード 可視化

集まったデータをグラフやカードで表示する画面。Node-RED Dashboard 2.0、Grafana、AWS QuickSight など。

データベース 記録

時系列で値を溜める倉庫。InfluxDB、DynamoDB、Timestream などが IoT でよく使われる。

5. MQTT って結局なに？

IoT で最もよく出てくる「MQTT」を、一段だけ噛み砕いて説明します。

MQTT は、① で学んだ msg の考え方とほぼ同じ発想です。「どこ宛て（topic）に、何のデータ（payload）を送るか」をブローカー（仲介サーバー）に渡すと、そのトピックを聞いている全員に配送される、という仕組みです。

MQTT のイメージ

センサーA「temperature」のトピックに "25" を送信 ↓ MQTT ブローカー（仲介サーバー）が受け取る ↓ 「temperature」を購読しているノード全員に配送 ↓ 受信: Node-RED ダッシュボード、Node-RED の別フロー、スマホアプリ、etc.
💡 Node-RED の msg との対応
Node-RED の msg.topic と msg.payload は、もともと MQTT を強く意識して設計されています。MQTT ノードを使えば、フロー内の msg をそのまま MQTT として外に送り出すことも、外から受け取ってフロー内の msg として扱うこともできます。
つまり、ここまで学んできた Node-RED の基本がそのまま MQTT 通信の基本でもあります。

6. エッジ vs クラウド：どちらで処理する？

「処理はエッジでやる」のと「処理はクラウドでやる」の違いをまとめます。IoT システム設計で最初に考えるのが、この切り分けです。

観点 エッジで処理 クラウドで処理
応答速度 速い（ms 単位） 遅い（ネット遅延あり）
通信量 少ない（加工後だけ送る） 多い（生データ全部送る）
ネット切断時 動作継続できる 止まる
大量データ分析 苦手（CPU・メモリ限界） 得意（スケール可能）
複数拠点集約 難しい 得意
運用コスト 電気代のみ 月額課金
向いている処理 即時判断、異常検知、前処理 集計、長期保存、横断分析、通知

実務では両方を組み合わせるのが一般的です。エッジで前処理して必要な値だけクラウドに送り、クラウドで長期保管・集計する、という分業が定石です。

7. 小さく始める IoT のパターン 3 つ

大掛かりな構成から始めるとハードルが高いので、まずは小さいところから積み上げるのがおすすめです。読者の方が実際に試しやすい規模感を 3 つ紹介します。

🅰 パターン A: エッジだけで完結（初級）

例: 部屋の温度を Raspberry Pi で測り、Node-RED Dashboard 2.0 に表示する。

温度センサー → Raspberry Pi → Node-RED → ダッシュボード（同じ Pi 上）

特徴: クラウド不要、設定が最小、0 円で動く。③ のハンズオンが体験した世界の延長線上。

🅱 パターン B: エッジ → MQTT → クラウド Node-RED（中級）

例: 複数の Raspberry Pi から温度データを集め、AWS 上の Node-RED で一括表示。

センサー → Pi (Node-RED) ─MQTT─→ クラウド Node-RED → ダッシュボード ─MQTT─↗

特徴: 複数拠点の集約ができる。MQTT の理解が必須。AWS の EC2 や軽量 VM に Node-RED を立てる構成。

🅲 パターン C: エッジ → クラウドサービス統合（応用）

例: センサーデータを AWS IoT Core で受け、DynamoDB に保存、異常値があれば SNS で通知。

センサー → Pi (Node-RED) ─MQTT/HTTPS─→ AWS IoT Core → DynamoDB（保存） → Lambda → SNS（通知）

特徴: 本格的なクラウドネイティブ構成。AWS 各サービスの組み合わせが学べる。運用コストと設計の複雑さが上がる代わりに、大規模運用に耐える。

📌 最初はパターン A で OK
学び始めは、パターン A で「センサー → フロー → ダッシュボード」の 1 本を作り切ることを目標にしてください。ここができれば、後は MQTT やクラウドを足すだけで B, C へ拡張できます。

8. Node-RED の役割を一言で言うと

IoT の世界で Node-RED の役割を一言で表すなら、こうです。

センサー／クラウド／ダッシュボードを
ノーコードで接着するハブ

プログラムを一行も書かずに、異なる世界のデータを「つなぐ」「整える」「見せる」ところが Node-RED の本領です。ここまで読んだあなたは、もうこの「ハブ」の中身を配線できる素地を持っています。

ロードマップはあくまで目安です。一番大事なのは「気になったノードをその日のうちに試す」習慣をつけること。読むだけでは手に馴染みません。

10. まとめ：地図が手に入った

  • IoT はセンサー → エッジ → 通信 → クラウド → UI の 5 層でできている。
  • Node-RED はエッジ・クラウド・UI のどこでも使える接着剤。
  • MQTT は Node-RED の msg（topic+payload）を外に出すための標準プロトコル。
  • エッジとクラウドは得意分野が違う。両者の組み合わせで設計する。
  • まずはパターン A（エッジ完結）から始めて、必要に応じて B, C へ拡張。

ここまで来れば、世の中の IoT 事例を見たときに「これはどの階層の話で、Node-RED だとどこを担うだろう？」と考えられるようになっています。第 0 章で学んだ基礎が、現実のシステムにどう対応するかのイメージが持てているはずです。
