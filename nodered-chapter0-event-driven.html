1. このフロー、勝手に動くんだけど？

初心者が Node-RED に触れてまず戸惑うのが、「一度デプロイすると、こちらが何もしなくてもフローが勝手に動き出す」という挙動です。Excel のマクロに慣れていると、「実行ボタンを押して初めて動く」感覚でいるので、この挙動は予想外に映ります。

この秘密は「イベント駆動（event-driven）」という仕組みにあります。このページではその仕組みを腹落ちするまで解説します。

2. 2 つのプログラミングスタイル

世の中のプログラムには、動き方で見ると大きく 2 つのスタイルがあります。

📜 手続き型（上から順）

書かれた手順を1 行目から最終行まで順に実行して、終わったらプログラムも終わる。

例: Excel のマクロ、Python のスクリプト、シェルスクリプト。

特徴: スタートとゴールがはっきりしている。人間が「実行」と言わないと動かない。

⚡ イベント駆動型（反応型）

プログラム全体は常に「待機モード」で待っていて、何かが起きたときだけ動く。

例: スマホのアプリ、Web サーバー、Node-RED。

特徴: 一度起動すると電源を切るまで待機し続ける。複数の「起きごと」に同時対応できる。

💡 Node-RED は純粋にイベント駆動型
デプロイした瞬間、すべてのフローは「待機モード」に入ります。そして、あらかじめ決められた「イベントの源（トリガー）」が発火したときだけ、そのフローにデータが流れ始めます。

3. 身近なイベント駆動

「イベント駆動」という言葉は難しく聞こえますが、日常生活の中にすでにたくさんあります。

日常の例 待っているもの 起きたときの動作
玄関のチャイム 誰かがボタンを押すこと 音が鳴る
スマホの通知 LINE のメッセージが届くこと 画面がつく、音が鳴る
自動ドア 人が近づくこと ドアが開く
目覚まし時計 設定時刻が来ること アラーム音が鳴る
エアコン 室温が設定温度を超えること 冷房が自動で動く

これらは全部「常に待っていて、特定のイベントが起きたときに反応する」仕組みです。Node-RED のフローもこれと全く同じ発想で作られています。

4. Node-RED は純粋なイベント駆動

Node-RED のフローは、デプロイした瞬間に全てが「待機モード」に入ります。画面には何もワイヤーが光らず、静かに次のイベントを待っています。そして、どこかの「イベントの源」となるノードで何かが起きると、そこから繋がっているワイヤーを通じてメッセージが流れ始めます。

⏳ 待機中... ➜ 🔔 イベント発生！ ➜ 📨 msg が流れる ➜ ⏳ また待機に戻る

この「待機 → 発火 → 処理 → 待機」のサイクルが、Node-RED のフローの基本動作です。電源を切るまで、これを延々と繰り返します。

📌 言い換えると
Node-RED のフローは、「郵便の仕分けセンター」のようなものです。職員は普段は待機していて、郵便物（msg）が届いたらそれぞれの処理（ノード）を通して、最終的な届け先（出力先）まで運ぶ。届かない日はずっと待機している。

5. Node-RED における「イベントの源」カタログ

Node-RED ではさまざまな種類のイベントを扱えますが、全て同じ仕組みで扱われている点が強力なところです。どのイベント源から来ても、msg としてワイヤーを流れる形になります。

⏰ Inject 定番

手動ボタン押下、一定時刻の到来、Node-RED 起動時など。最も基本的なイベント源。

📨 MQTT In IoT

MQTT ブローカーから特定トピックのメッセージが届いたら発火。IoT の中心的なイベント源。

🌐 HTTP In API

外部からの HTTP リクエストが届いたら発火。Web API やフォーム送信を受ける。

👁️ Watch

指定したファイルやフォルダの中身が変化したら発火。ログ監視やファイル連携に。

🎮 Dashboard 入力ウィジェット

ユーザーがブラウザでボタン押下、スライダー操作、テキスト入力したら発火。

🔌 GPIO In（Raspberry Pi） エッジ

Raspberry Pi の GPIO ピンの電圧が変化したら発火。スイッチやセンサーからの物理入力。

📧 Email In

指定メールボックスに新着メールが届いたら発火。

🔗 WebSocket In

WebSocket 経由でメッセージが届いたら発火。リアルタイム双方向通信に。

🎯 覚えておきたいこと
イベントの種類はたくさんありますが、Node-RED の見方は統一されています。どれも「左側のポートが無く、右側にだけ出力ポートがある」形で、イベントが起きたら msg を流し始めます。これが見抜けると、新しいノードに出会っても怖くありません。
📚 詳細は個別ガイドへ
各イベント源ノードの詳しい使い方は、標準ノードセクションの Inject、MQTT、HTTP、Watch などで解説しています。

6. 複数のイベントが同時に来たら？

イベント駆動型の大事な特徴は、複数のイベントが並行して動けることです。

たとえば、Inject が 1 秒ごとに発火するフローと、MQTT で受信したら動くフローが同じ Node-RED 上に両方ある場合、片方が動いている間にもう片方が止まるわけではなく、それぞれが独立して動きます。

⚠️ ただし注意点
複数のイベントが同時に来たとき、どのフローが先に処理を終えるか、順序は保証されません。これを「非同期」と呼びます。慣れるまでは「同時に来たものは、どっちが先か分からない」と覚えておけば OK です。詳しくは ④ 動かないときの進め方 の「不定期に動かない時」で解説します。

7. フロー設計のコツ：起点を意識する

Node-RED でフローを作るとき、まず最初に決めるべきは「このフローは何をきっかけに動き始めるか？」です。これを「起点（トリガー）を決める」と呼びます。

起点が曖昧だと、後から「あれ、これ誰が動かしてるんだっけ？」となって保守性が下がります。おすすめは次の 3 点です。

  1. 1 つのフローの起点は 1 つに絞る（複数の起点を混ぜない）。どうしても複数の起点から同じ処理を呼びたい場合は、Link ノードでサブフローとして切り出す方法があります。
  2. 起点のノードの隣に Comment ノードを置く。「このフローは○○があったら動く」と書いておくと、後で見返したときに理解が早い。
  3. 同じタブに用途の違う起点を並べない。たとえば「MQTT 受信用」と「定期処理用」は別のタブに分けると、視認性が上がります。
💡 Node-RED エディターで起点を見つけるコツ
「このフローは何で動く？」と確認するときは、ワイヤーを左に辿って、左側の入力ポートが無いノードを探します。それが起点（イベント源）です。

8. Excel マクロとの対比

Excel にもイベント駆動の仕組みは実はあります。VBA を触ったことがある方なら、次の名前を見たことがあるかもしれません。

📊 Excel のイベントハンドラ

  • Workbook_Open ... ブックを開いたときに発火
  • Worksheet_Change ... セルの値が変わったときに発火
  • Worksheet_SelectionChange ... セルの選択が変わったときに発火
これらは全部「◯◯が起きたら、このコードを動かしてね」というイベント駆動の仕組みです。Node-RED はこれをもっと種類豊富に、しかも視覚的に見える形で提供している、と考えると分かりやすいです。
Excel VBA Node-RED
Workbook_Open（起動時） Inject（「Node-RED 起動時」オプション）
Application.OnTime（時刻指定） Inject（繰り返し実行、時刻指定）
Worksheet_Change（値変更） Watch（ファイル変更）、MQTT In（値受信）
ボタンの Click イベント Dashboard の ui-button

9. まとめと次のページ

このページで押さえたかったことは次の 3 つです。

  1. Node-RED はイベント駆動型。デプロイ後は常に待機していて、何かが起きたら動く。
  2. イベントの源は種類が豊富（Inject、MQTT、HTTP、Watch、Dashboard、GPIO など）だが、全て同じ仕組みで扱われる。
  3. フローを作るときは「起点は何か」を最初に決める。
