📖 1. 概要

MQTT（Message Queuing Telemetry Transport）は、IoT向けの軽量メッセージングプロトコルです。センサーデータの収集や機器制御など、多数のデバイス間通信に最適化されています。

MQTTの特徴

  • 軽量：ヘッダーが小さく、帯域幅を節約
  • Pub/Sub モデル：送信者と受信者が直接接続しない
  • QoS（Quality of Service）：メッセージ配信の信頼性を選択可能
  • Retain：最後のメッセージを保持し、新規購読者に配信

プロパティ説明例

トピック発行先トピック（空欄でmsg.topic使用）devices/lamp/control

保持メッセージを保持true/false

QoS（Quality of Service）レベル

QoS説明用途
0At most once（最大1回）センサーデータなど、多少のロスが許容できる場合
1At least once（最低1回）重要データ（重複の可能性あり）
2Exactly once（正確に1回）課金など、厳密な配信が必要な場合

🏷️ 4. トピックとワイルドカード

トピックの階層構造

MQTTのトピックは / で区切った階層構造で設計します：

📁 home
📁 livingroom
📄 temperature
📄 humidity
📄 light
📁 bedroom
📄 temperature
📄 humidity
// トピック例 home/livingroom/temperature → 25.5 home/livingroom/humidity → 60 home/bedroom/temperature → 22.0

ワイルドカード

ワイルドカード説明例
+（シングルレベル） 1階層のみにマッチ home/+/temperature
→ home/livingroom/temperature ✓
→ home/bedroom/temperature ✓
#（マルチレベル） 以降の全階層にマッチ（末尾のみ） home/#
→ home/livingroom/temperature ✓
→ home/bedroom/humidity ✓

💡 トピック設計のベストプラクティス:

  • 意味のある階層構造にする（場所/デバイス/センサー）
  • 小文字とアンダースコアを使用
  • 先頭に / を付けない（home/temp が推奨、/home/temp は非推奨）
  • ユニークな接頭辞を使用（公開ブローカーでは特に重要）

使用パターン

パターン1: 基本 Publish（発行）

サンプルフローの「基本: Publish（発行）」を参照してください。

メッセージ → MQTT Out

ポイント:

  • msg.payload にメッセージ内容を設定
  • トピックはノード設定または msg.topic で指定
  • QoS でメッセージ配信の信頼性を選択

パターン2: 基本 Subscribe（購読）

サンプルフローの「基本: Subscribe（購読）」を参照してください。

MQTT In → メッセージ表示

ポイント:

  • 購読トピックはワイルドカード（#, +）使用可能
  • msg.topic に受信したトピック名が格納される
  • Output設定で自動型検出またはString/Buffer指定

パターン3: JSONデータの送受信

サンプルフローの「JSON データの送受信」を参照してください。

JSON生成 → 送信 ｜ 受信 → JSON確認

ポイント:

  • 送信時: msg.payload にオブジェクトを設定すると自動でJSON文字列化
  • 受信時: Output を auto-detect にすると自動でパース
  • センサー値など構造化データの送受信に最適

パターン4: デバイス制御（コマンド送受信）

サンプルフローの「デバイス制御（コマンド送受信）」を参照してください。

コマンド受信 → ON/OFF分岐 → デバイス制御

ポイント:

  • デバイスごとにトピックを分ける（例: home/light/living）
  • Switchノードでコマンド内容（ON/OFF等）を分岐
  • 制御結果をstatusトピックに発行するとフィードバック可能

📝 6. 演習問題

演習1: メッセージの発行 初級

📋 課題: ボタンをクリックすると、MQTTブローカーに「Hello MQTT!」を発行するフローを作成してください。

🎯 要求仕様:

  • トピック: あなたの名前/test/hello
  • メッセージ: Hello MQTT!
  • ブローカー: test.mosquitto.org

✅ 成功の条件:

  • MQTT Out ノードのステータスが「接続済み」になる
  • Inject ボタンを押した後、MQTTクライアント（mosquitto_subコマンド等）で該当トピックにメッセージが届くことを確認できる
  • Node-RED のデバッグパネルにエラーが表示されない
💡 ヒント

Inject ノード: msg.payload に「Hello MQTT!」を設定

MQTT Out ノード:

  • Server: test.mosquitto.org（新規作成）
  • Topic: あなたの名前/test/hello
  • QoS: 1

✅ 7. まとめ

🎯 重要ポイント:

  • Pub/Sub モデル: 送信者と受信者が直接接続せず、ブローカーを介して通信
  • トピック: 階層構造で設計（例: home/sensor/temperature）
  • ワイルドカード: +（1階層）、#（全階層）
  • QoS: 0=最大1回、1=最低1回、2=正確に1回
  • Retain: 最後のメッセージを保持（新規購読者に配信）

⚠️ 注意事項:

  • 公開ブローカーではユニークなトピック名を使用
  • 機密データは認証・暗号化を設定したブローカーを使用
  • 大量のメッセージを発行する際はQoS 0を検討（軽量）

症状原因解決方法

接続が切れるKeep Alive タイムアウトKeep Alive 値を調整
重複メッセージQoS 1 の再送アプリケーション側で重複排除
古いメッセージが届くRetain メッセージ空のRetainメッセージを発行してクリア
