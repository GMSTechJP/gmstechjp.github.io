📖 1. 概要

UDP（User Datagram Protocol）は、コネクションレスの軽量プロトコルです。TCPと異なり、接続の確立や再送制御を行わないため、高速で軽量な通信が可能です。

📮 郵便に例えると：UDPは「はがき」のようなものです。送ったら終わり。届いたかどうかの確認はしません。一方TCPは「書留郵便」で、相手に届いたことを確認します。

UDPの特徴

  • コネクションレス：接続確立なしに即座に送信
  • 軽量・高速：オーバーヘッドが少ない
  • 信頼性なし：パケット損失や順序逆転の可能性
  • ブロードキャスト可能：ネットワーク全体に一斉送信
  • マルチキャスト対応：特定グループに送信

UDP In

役割: 指定ポートでUDPパケットを受信

📤 UDP Out

UDP Out

役割: UDPパケットを送信（ユニキャスト/ブロードキャスト）

⚖️ 2. UDP vs TCP の違い

特性 UDP TCP
接続 コネクションレス（接続不要） コネクション型（接続確立必要）
信頼性 なし（パケットロスあり得る） あり（再送制御）
順序保証 なし あり
速度 高速 比較的遅い
オーバーヘッド 小さい（8バイト） 大きい（20バイト〜）
ブロードキャスト 可能 不可
用途例 DNS、動画配信、ゲーム、センサーデータ Web、メール、ファイル転送

💡 使い分けのポイント:

  • UDP を選ぶ場合：リアルタイム性重視、多少のデータロスが許容できる、ブロードキャストしたい
  • TCP を選ぶ場合：確実にデータを届けたい、順序が重要、双方向のやり取りが必要

⚙️ 3. UDP In ノード

設定項目

各項目は Node-RED エディターの表示ラベルに対応します。「待ち受け」で「マルチキャストメッセージ」を選ぶと、「グループ」と「ローカルIP」の欄が現れます。

設定項目説明既定値
待ち受け 受信方法を選びます。「UDPメッセージ」は単一ポートで通常のUDPを受信します。「マルチキャストメッセージ」はマルチキャストグループに参加して受信します。 UDPメッセージ
グループ 参加するマルチキャストグループのアドレス。「待ち受け」が「マルチキャストメッセージ」のときだけ表示されます。 （空）
ローカルIP マルチキャスト受信に使うローカルインターフェイスのアドレス。NICが複数あるときに受信側を指定します。空欄なら既定のインターフェイスを使います。 （空）
ポート 待ち受けるポート番号。必須項目です。 （必須・既定なし）
種類 ポート欄の右にあり、IPのバージョンを ipv4 と ipv6 から選びます。 ipv4
出力 受信データを msg.payload に格納する形式。「バイナリバッファ」（Buffer）、「文字列」（UTF-8文字列）、「Base64文字列」から選びます。 バイナリバッファ
名前 エディター上のノード表示名。省略するとポート番号が表示されます。 （空）

受信時の msg プロパティ

プロパティ説明
msg.payload受信データ（String または Buffer）
msg.ip送信元IPアドレス
msg.port送信元ポート番号

⚙️ 4. UDP Out ノード

送信先の設定方法

設定方法説明
ノードで固定 Address と Port をノード設定で指定
メッセージで動的指定 msg.ip と msg.port で送信先を指定

設定項目

各項目は Node-RED エディターの表示ラベルに対応します。「送信」が「マルチキャストメッセージ」のときだけ「ローカルIP」の欄が現れ、「アドレス」のラベルは「グループ」に変わります。

設定項目説明既定値
送信 送信方法を選びます。「UDPメッセージ」は通常の単一宛先送信、「ブロードキャストメッセージ」は同一サブネットへの一斉送信、「マルチキャストメッセージ」はグループ宛て送信です。右側の欄に宛先ポートを入れます。 UDPメッセージ
アドレス 宛先IPアドレス。ブロードキャスト時はブロードキャストアドレス（例 192.168.1.255）、マルチキャスト時はラベルが「グループ」に変わりグループアドレスを入れます。 （空）
種類 アドレス欄の右にあり、IPのバージョンを ipv4 と ipv6 から選びます。 ipv4
ローカルIP マルチキャスト送信に使うローカルインターフェイス。「送信」が「マルチキャストメッセージ」のときだけ表示されます。 （空）
ローカルポート 送信元のローカルポート。ドロップダウンで「ローカルポートをランダムに使用」か「ローカルポートを使用」（固定）を選び、固定時は右の欄にポート番号を入れます。 ランダム
Base64形式のペイロードを復号 チェックを入れると、Base64文字列の msg.payload をバイナリに復号してから送信します。 オフ
名前 エディター上のノード表示名。 （空）

ブロードキャストアドレス

// 限定ブロードキャスト（同一サブネット内） 192.168.1.255 ← サブネット 192.168.1.0/24 の場合 // ローカルブロードキャスト 255.255.255.255

送信時に設定できる msg プロパティ

プロパティ説明
msg.payload送信データ（String または Buffer）
msg.ip送信先IPアドレス（動的指定時）
msg.port送信先ポート（動的指定時）

⚠️ テスト方法:

UDPのテストには以下のツールを使用できます：

  • netcat (nc): echo "Hello" | nc -u localhost 8000
  • socat: echo "Hello" | socat - UDP:localhost:8000

  • Python: socket モジュールでテスト

📝 6. 演習問題

演習1: UDPメッセージの送受信 初級

📋 課題: ポート9000でUDPパケットを受信し、デバッグサイドバーに表示してください。同時に、送信ノードでテストメッセージを送れるようにしてください。

🎯 要求仕様:

  • UDP In: ポート 9000 で待ち受け
  • UDP Out: localhost:9000 にテストメッセージ送信

✅ 成功の条件:

  • Inject ボタンを押すとデバッグパネルに「Hello UDP!」が表示される
  • デバッグパネルのノードステータスに送信元のIPアドレスが表示される
  • echo "test" | nc -u localhost 9000 を実行してもデバッグパネルに受信データが表示される
💡 ヒント

UDP In ノード:

  • 待ち受け: UDPメッセージ
  • ポート: 9000
  • 出力: 文字列

UDP Out ノード:

  • アドレス: localhost
  • ポート: 9000

演習2: 送信元への返信 初級

📋 課題: 受信したデータに「[REPLY]」を付けて、送信元に返信してください。

🎯 要求仕様:

  • ポート: 9001
  • 受信: Hello
  • 返信: [REPLY] Hello

✅ 成功の条件:

  • echo "Hello" | nc -u localhost 9001 を実行すると「[REPLY] Hello」が返ってくる
  • 異なるメッセージを送っても、常に「[REPLY] 」が先頭に付いて返ってくる
  • 返信が送信元の IP アドレスとポートに対して送られている（送信元に届く）

✅ 7. まとめ

🎯 重要ポイント:

  • コネクションレス：接続確立不要で即座に送信可能
  • 信頼性なし：パケットロスや順序逆転の可能性があることを考慮
  • ブロードキャスト：255.255.255.255 または サブネットアドレス
  • 返信：msg.ip と msg.port を使って送信元に返信可能
  • 用途：センサーデータ、発見プロトコル、リアルタイム通信

⚠️ 注意事項:

  • 信頼性が必要な場合はTCPを使用
  • ブロードキャストは同一サブネット内でのみ有効
  • ファイアウォール設定でUDPポートを開放
  • パケットサイズ：MTU（通常1500バイト）を超えると分割される

🔧 8. トラブルシューティング

症状原因解決方法
データが届かないファイアウォール/ポート未開放FW設定確認、別ポート試行
ブロードキャストが届かない異なるサブネット/ルーター越え同一サブネット内で使用

📘 10. 用語集（UDP 固有の用語）

UDP はシンプルなプロトコルですが、配信形態（ユニキャスト / マルチキャスト / ブロードキャスト）や信頼性の用語で混乱しやすい点があります。データグラム・パケット・ソケット・ポートなどは 通信の基礎ガイドの用語集 を参照してください。

📭 コネクションレス（connectionless）

「接続を張らずに、いきなり荷物を投げる」方式。TCP の「電話をかけてから話す」のと対照的に、UDP は「はがきを投函する」イメージです。

  • 事前の ハンドシェイクなし（送信側はすぐ送信を開始できる）
  • 状態を持たないので、サーバー側も軽量（少ないリソースで大量送信先を扱える）
  • 逆に、相手が受信できる状態かどうか送信側は知らない

🎯 ベストエフォート配信（best-effort）

UDP は 「届くか届かないか保証しない、順番もずれる、重複する可能性もある」方針。これらを保証するのは TCP の役割であり、必要ならアプリ側で実装します。

  • 届かない（再送なし）：1 通失われても気づかれない
  • 順番が入れ替わる：後から送った方が先に着くことがある
  • 重複する：同じデータグラムが 2 回届くことがある

こうした「不確実さ」を承知の上で、速度・軽さ・1 対多配信のために UDP を選びます。

📡 ユニキャスト / マルチキャスト / ブロードキャスト（3 つの配信形態）

形態送信先例典型用途
ユニキャスト（unicast）特定の 1 台に送る192.168.1.10 宛て通常の 1 対 1 通信
ブロードキャスト（broadcast）同じサブネット内の全員に送る192.168.1.255（限定）/ 255.255.255.255（全体）サービス発見、DHCP、機器探索
マルチキャスト（multicast）「このグループに加入している人」だけに送る224.0.0.0 〜 239.255.255.255 の範囲動画/音声のストリーミング配信、産業制御
💡 マルチキャストはブロードキャストとユニキャストの中間です。ブロードキャストのように「興味ない人にも届いて負荷をかける」ことなく、ユニキャストのように「同じデータを N 回コピー送信する」非効率もなく、受信したい人だけが受け取れます。
⚠️ クラウド／インターネット越しでは届かない: ブロードキャストとマルチキャストは ルーターを跨ぐと伝わらないのが原則です。同一サブネット（同じ LAN セグメント）内でのみ有効と考えてください。クラウド共有環境ではほぼ動きません。

📏 MTU（Maximum Transmission Unit）と分割（fragmentation）

1 回で送れるIP パケットの最大サイズ。物理ネットワークごとに決まっており、Ethernet では 通常 1500 バイトです。

  • UDP データグラムが MTU を超えると、IP 層で複数のフラグメント（断片）に分割される
  • 分割されたフラグメントのうち 1 つでも失われると、データグラム全体が無効になる（UDP には再送がない）
  • そのため UDP では 1 データグラムを MTU 以下に収めるのが鉄則。安全圏は 1472 バイト（1500 − IP ヘッダ 20 − UDP ヘッダ 8）
  • 大きなデータを送りたい場合は、アプリ側で自前で分割・再構成する必要がある（または TCP を使う）

✅ チェックサム（checksum）

UDP ヘッダとペイロードの誤り検出用の 16 bit 値。受信側で計算し直して送信時と一致しなければそのデータグラムを破棄します。

  • IPv4 では任意（0 にすれば未使用扱い）
  • IPv6 では必須（IPv6 自身がチェックサムを持たないため）
  • これはあくまで誤り「検出」であって、誤り訂正や再送はしない（再送が欲しければ TCP かアプリ層で実装）

🔁 ポート再利用（SO_REUSEADDR）

UDP では「同じポートに複数のプロセスが Bind したい」状況がよくあります（マルチキャスト受信や、サーバー再起動時の即時再 Bind など）。

  • SO_REUSEADDR オプションで「このポートを他と共有してよい」と OS に伝える
  • Node-RED の UDP In ノードは内部で適切にこのオプションを設定するため、明示的な設定は通常不要
  • 「ポートがすでに使われています」エラーが出るときは、別のプロセス（別の Node-RED フローや netcat 等）が同じポートを占有していないかを確認
