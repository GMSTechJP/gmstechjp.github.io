📚 1. このガイドの目的

FunctionノードはJavaScriptを書く必要があるため、初心者にとって最初の壁になりがちです。一方、ChatGPT等のAIに「こういう処理を書いて」と頼めば、それらしいコードはすぐ手に入ります。問題は——そのコードが本当に動くかどうか別の話、ということです。

💡 このガイドが目指す姿
「AIに頼めばOK」でも「自分で全部書く」でもなく、「AIに正確に依頼 → Node-RED上で必ず単体テスト → 結果をAIに返して修正」というループを回す方法を身につけます。これは初心者が最短で動くフローを作るのに、もっとも実用的なアプローチです。

🎯 2. AIに「何を」「どう」依頼するか

第0章で学んだ 「すべてのプログラムは 入力 → 処理 → 出力」 のおさらいです。Functionノードも例外ではありません。

📥 入力
msg(payload など) ➜ ⚙️ 処理
Functionノード(JS) ➜ 📤 出力
msg(加工後)

つまりFunctionノードを書くとは、「この入力を、この処理で、この出力にする」を決めること。AIに依頼するときも、この3つを伝えれば的確なコードが返ってきます。逆に、これを伝えないAIには的外れなコードしか書けません。

🪪 AIに渡すべき「3点セット」

📥 入力サンプル
実際のmsg構造 ➜ 🤖 AIに依頼
やりたいこと ➜ 📤 期待する出力
欲しいmsg構造

① やりたいこと（What）

「処理の目的を1〜2文」で。「文字列を分割」「合計を計算」「異常値を抜き出す」など。あいまいさを排除するのがコツです。

NG例: 「いい感じに整形して」
OK例: 「カンマ区切り文字列を、ヘッダ行をキーにしたオブジェクトの配列に変換して」

② 入力の形（Input shape）

実際の msg をJSON形式で1件貼り付けるのが最強です。型・ネストの深さ・空文字の有無まで伝わります。

とくに payload が文字列／数値／オブジェクト／配列のどれなのかは、AIの選ぶコードを大きく左右します。

③ 欲しい出力の形（Output shape）

同じくJSONで完成形を1件。「これに変換したい」を見せるだけで、AIは正しいプロパティ名・型に合わせて書きます。

型が定まらない場合は「数値で」「文字列のままで」など型を明記。

📝 なぜサンプルを貼るのが効くのか
自然言語で「2列のCSVを連想配列に」と言うより、"name,age\nAlice,30" という生データを見せる方が、AIに残された解釈の幅がほぼゼロになります。サンプル1件は、説明文100文字より雄弁です。

📋 3. コピペで使えるプロンプトテンプレート

まずは下のテンプレートを丸ごとコピーして、3つの「ここを埋める」部分を差し替えるだけで、安定した品質のコードが得られます。

プロンプトテンプレNode-RED の Function ノード用 JavaScript を書いてください。 【やりたいこと】 {{ここに処理目的を1〜2文で。例：温度の配列から、10〜35℃の範囲外を異常値として抜き出したい}} 【入力 msg のサンプル（実物）】 ```json {{ここに実際の msg を JSON で貼り付け}} ``` 【欲しい出力 msg のサンプル（完成形）】 ```json {{ここに変換後の理想の msg を JSON で貼り付け}} ``` 【制約】 - Node-RED の Function ノード（バージョン4系）で動くコードにしてください。 - 1出力で `return msg;` する形でお願いします（複数出力にしないでください）。 - 外部ライブラリは使わない前提でお願いします（必要な場合は明記してください）。 - 例外処理は最小限で構いません。null/undefined の入力時の挙動だけ簡単にコメントで補足してください。 完成したコードと、簡単な動作説明を返してください。

✅ このテンプレが効く理由

  • 用途を明示 ← 「Node-RED Function ノード用」と書くことで node.send() や msg.payload 前提で書いてくれる
  • サンプル2点 ← 入力と出力の形が固定され、AIが勝手にプロパティ名を作らない
  • 制約セクション ← 出力数・依存ライブラリ・例外処理の指示でブレを抑える

👎 NGプロンプト vs 👍 OKプロンプト

❌ NG例：あいまい・サンプルなし

Node-REDで温度のデータを
処理するコードを書いて。
異常値を見つけたい。

何が起きるか: AIが勝手に「温度の構造」「異常値の閾値」「出力形式」を決め打ちしたコードが返ってくる。あなたのフローには合わない可能性が高い。

✅ OK例：3点セット入り

Node-REDのFunctionノード用JS。
やりたいこと: 配列から10〜35の
範囲外の値だけ抜き出したい。
入力 msg.payload:
[18.5, 22.1, 38.7, -15.2, 25.0]
出力 msg.payload:
{ normal: [...], abnormal: [...] }
1出力でreturn msg;の形で。

何が起きるか: 入力と出力の形が固まっているので、コードの分岐余地がほぼなくなり、初回から動くコードが返ってくる確率が大幅にアップする。

🔄 4. Inject + Function + Debug で「単体テスト」する

AIから受け取ったコードを、いきなり実フローに組み込むのは危険です。必ずテスト用フローで単体動作を確認してから採用しましょう。

🧪 テスト用フローのスケルトン

Inject
入力サンプル → Function
AI生成コード貼り付け → Debug
complete msg

シンプルなこの3ノード構成で、Functionノードが「期待した出力を返すか」を最短で確認できます。Debugノードは 「Output: complete msg object」モード にしておくと、msg全体が見えるので差分確認がラクです。

🛠️ テストの手順（毎回これをやる）

  1. Injectノードに「典型的な入力」を1つ用意 — payloadType を JSON や 文字列 に切り替え、本物に近いデータを入れる
  2. Functionノードに AI のコードをそのまま貼り付け
  3. Debugノードを Complete msg object モードにする — payload以外の msg.topic 等もちゃんと残っているか確認できる
  4. デプロイ → Injectのボタンを押す
  5. Debugサイドバーで「期待した出力」と一致するか確認
💡 境界値も忘れずに
最低でも以下を別々のInjectノードで流すのがおすすめです：
・典型的な入力（普通のケース）
・空の入力（"" や [] や {}）
・想定外の型（数値が来る想定なのに文字列、など）

🤖 5. AIに結果を返して修正してもらうループ

テストで「期待と違う結果」「エラー発生」が起きたら、その事実をそのままAIに伝えるのがコツです。「動かない」と書くだけではAIも当てずっぽうになります。

📨 差し戻し用テンプレート

差し戻しテンプレ先ほどのコードをNode-REDで実行したところ、期待した動作になりませんでした。 事実を共有しますので、原因と修正版コードを提示してください。 【流した入力 msg】 ```json {{Debug出力に出ていた入力 msg をそのまま貼る}} ``` 【期待した出力 msg】 ```json {{こうなって欲しい msg を貼る}} ``` 【実際のDebug出力 msg】 ```json {{Debugサイドバーの出力をコピーしてそのまま貼る}} ``` 【発生したエラー（出ていれば）】 ``` {{デバッグサイドバーの赤いエラー全文をコピペ。出ていなければ「エラーなし」と書く}} ``` 差分の原因と、修正版の完全なコードをお願いします。
⚠️ よくある落とし穴：要約しないこと
「数字が違いました」「なんかエラーが出ました」のような要約はNG。AIが推測で修正するため、別のバグを生むことがあります。Debug出力はコピペでそのまま渡すのが鉄則です。
📸 スクリーンショットを渡すという手
ChatGPT・Claude・Gemini・Microsoft 365 Copilot など画像入力に対応したAIを使っている場合は、エラー文や思い通りでないDebug出力をスクリーンショットで渡すのも有効な手段です。
  • 向いているケース：Debugサイドバーの赤いエラーボックス、ネストの深いオブジェクトの折りたたみ表示、エディタ画面でのノード接続図、ダッシュボード上の表示崩れなど、「テキストにすると情報が落ちる」場面
  • 合わせてやると良いこと：画像を貼るだけで終わらせず、「どの部分が期待と違うか」を文章でも一言添えると精度が上がります（例：「画像の payload.normal が空配列ですが、本来は6件入るはずです」）
つまり「テキスト貼り付け」と「スクリーンショット」は排他ではなく併用。コピペで済むものはコピペで、視覚情報が物を言う場面は画像で、と使い分けましょう。
🔐 大前提：会社業務で使うなら「会社契約のAI」を使うこと
業務データ・センサーログ・顧客情報・APIキー・社内システムのスクリーンショットなどを個人契約のAIや無料サービスに貼り付けるのは、情報漏えいリスクが極めて高い行為です。コピペでもスクリーンショットでも、AIに送った情報は「自分の端末の外」に出ます。
  • OK：会社が契約しているテナント／法人プラン 例: Microsoft 365 Copilot（テナント契約）、ChatGPT Enterprise / Team、Claude for Work（Team / Enterprise）、Gemini for Workspace ── これらは原則として入力データをモデル学習に使わない契約・設定が可能で、監査ログも残せます
  • NG：個人契約・無料サービスを業務利用 個人の ChatGPT Free / Plus、無料の Gemini、Claude.ai 個人プランなどへの業務情報投入は、多くの会社で社内規程上の許可が必要／禁止されている行為です。学習利用の有無は各サービスの利用規約と個別設定（オプトアウトの有無）に依存するため、ユーザー自身では完全に把握しきれず、データはサービス提供者側のサーバーに保存される点も留意が必要です
  • 送信前チェック：スクリーンショットは特に文脈ごと写るので、APIキー・トークン・氏名・メールアドレス・社外秘マーク・URL（社内ドメイン）などの映り込みを必ず確認。テキストよりも漏えいリスクが上がります
  • 迷ったら：自社の情報セキュリティ規程／生成AI利用ガイドラインを確認、または情シス部門に「このサービスを業務で使ってよいか」を確認してから利用すること
個人の学習用フローや、このガイドの演習のようにダミーデータしか含まないものなら、無料サービスでも問題ありません。「業務データかどうか」で線を引くのがポイントです。

🔁 差し戻しの典型パターン

症状 AIに伝えるべき情報 追加で書く一言
出力の型が違う（数値 vs 文字列） 「期待は数値、実際は文字列でした」 「Number() や parseFloat() を使ってください」
プロパティが undefined になる 入力msgの実物 + 出力でundefinedになった箇所 「入力にこのキーが無い場合のフォールバックも入れてください」
配列が1件しか処理されていない 「入力の配列は3件、出力も3件欲しい」 「map や filter で全件処理する形にしてください」
TypeError / ReferenceError エラー全文（赤いボックスの中身） 追加せず、エラーをそのまま貼ればOK
2回目以降で挙動が変わる 「1回目」「2回目」のDebug出力を両方 「msgオブジェクトを直接書き換えていませんか？」

📖 6. 学びを加速させる：AIを「コードの先生」として使う

ここまでは「動くコードを手に入れる」ことに焦点を当ててきました。でも、AIから返ってきたコードを「動いた、よし採用」で終わらせると、あなたのスキルは伸びません。同じAIを、コードの解説者・先生として使い倒すことで、1回のやり取りから学べる量が一気に増えます。

📝 6-1. コード生成と同時に「コメント付き」で頼む

プロンプトの【制約】に1行加えるだけで、勉強用の教材レベルにコードの説明が手厚くなります。

追加する1行- 初心者向けに、各処理の意図が分かるコメントを日本語で入れてください。特に map / filter / reduce / 三項演算子・分割代入など、慣れない構文には1行解説を添えてください。

❌ コメントなしの応答

const lines = text.split("\n").filter(l => l.trim() !== "");
const headers = lines[0].split(",");
const rows = lines.slice(1).map(line => {
    const cols = line.split(",");
    return headers.reduce((obj, key, i) => {
        obj[key] = cols[i];
        return obj;
    }, {});
});

動くけれど、reduce や slice(1) が初見の人には何をしているのか分からない。

✅ コメント依頼ありの応答

// 改行で分割し、空行を除外（先頭/末尾の空白も対象）
const lines = text.split("\n").filter(l => l.trim() !== "");

// 1行目をヘッダ行として "name,age,city" → ["name","age","city"]
const headers = lines[0].split(",");

// 2行目以降を1行ずつオブジェクトに変換
const rows = lines.slice(1).map(line => {
    const cols = line.split(",");
    // headers配列を順に回し、対応する値とペアにしてobjに積む
    // (key, i) は ["name", 0], ["age", 1] ...
    return headers.reduce((obj, key, i) => {
        obj[key] = cols[i];
        return obj;
    }, {});
});

同じコードでも「なぜこう書くか」が読めば分かる状態に。次に似た処理を書くときの参考になる。

💡 さらに踏み込むなら
「コード末尾に、このコードで使っているJavaScriptの構文・メソッドのリストを初心者向け解説付きで列挙してください」と頼むのも有効です。Array.prototype.reduce / アロー関数 / || 演算子のフォールバック用法、などの索引が手元に残ります。

❓ 6-2. 「これ何？」と即座に聞き返す

動くコードが手に入ったあと、1行でも理解できない箇所があったら、その場でAIに聞き返すのが最大の学習チャンスです。検索する手間より早く、その文脈に沿った説明が得られます。

🪪 聞き返しテンプレート

聞き返しテンプレ先ほどのコードについて、以下の部分が理解できませんでした。 初心者向けに、なぜそう書くのか・どんな動きをするのかを解説してください。 【該当箇所】 ```js {{コピペ。前後数行も含めると文脈が伝わりやすい}} ``` 【特に分からないこと】 - {{例：reduce の3番目の引数 i は何？}} - {{例：なぜ `(msg.payload || "")` と書くの？}} - {{例：このアロー関数の `=>` は普通の function と何が違う？}} 可能なら、似たことを別の書き方（for ループ等）で書いた場合と比較もお願いします。

🎯 こう聞くとさらに学びが深まる

  • 「なぜこの書き方？」 — 動作だけでなく設計判断の意図を聞く（例：「なぜ for ではなく map？」）
  • 「他の書き方は？」 — 同じ結果を別解で実装してもらい、複数の引き出しを得る（「for 文版・forEach 版・reduce 版それぞれで書いてください」）
  • 「もっとシンプルにできる？」 — 冗長な部分の改善案を引き出す（と同時に「なぜ短い方が良いか」も学べる）
  • 「逆にもっと初心者向けに、長く書くと？」 — 短すぎて分からない時の逆方向。1行のチェーンを5行のループにほぐしてもらう
  • 「これがバグるとしたらどんな入力？」 — エッジケースを自分で考える前にAIに列挙してもらい、防御的プログラミングの感覚を養う
⚠️ 注意：「分かった気」で終わらせない
AIの解説を読んで「ふーん、なるほど」で済ませると、次に同じ場面で書けるかは怪しいままです。学んだ書き方は、その場でちょっとだけ改造して動かしてみるのが定着の最短ルート。「map を forEach に書き換えても同じ結果になるか？」と試すような小さな実験を積み重ねましょう。

🔁 6-3. 学びを設計に活かす「派生プロンプト」

ひとつのコードから派生させて、視野を広げる質問パターンの代表例：

派生プロンプト 得られる学び
「このコードで使っているJavaScriptの機能を、登場した順に箇条書きで一覧化してください」 自分が今まさに使った構文の索引ができる。後で復習しやすい
「この処理を、Node-REDの他のノード（Change/Switch/Splitなど）の組み合わせで実現するとどうなりますか？」 Functionで書くべきか、標準ノードで足りるかの判断軸が育つ
「想定外の入力（型違い・null・空配列・極端に大きい配列）が来た時、それぞれどう振る舞いますか？」 自分のコードの耐久性を可視化してから採用できる
「このコードに学習目的のミニ演習問題を3つ作ってください。難易度を初級・中級・上級で」 受け身の理解から能動的な練習へ。AIが家庭教師になる

🌱 まとめ：AIは「答えを返す機械」ではなく「対話する先生」

最初の1往復で動くコードを得たら、そこからの2〜3往復で理解度を底上げするのがコツ。「動いた、終わり」ではなく、「動いた、で、これは何？他の書き方は？」と問い続ける姿勢そのものが、AIネイティブ時代のプログラミング学習法です。

🏋️ 7. 演習：1サイクル回してみる

演習1：CSV風文字列をオブジェクト配列に変換 中級

シナリオ: ある外部APIから、改行区切りのCSV文字列が msg.payload に入ってきます。これを「ヘッダ行をキーにしたオブジェクトの配列」に変換したい。

📥 入力サンプル（msg.payload）

"name,age,city\nAlice,30,Tokyo\nBob,25,Osaka\nCarol,40,Nagoya"

📤 期待する出力（msg.payload）

[ { "name": "Alice", "age": "30", "city": "Tokyo" }, { "name": "Bob", "age": "25", "city": "Osaka" }, { "name": "Carol", "age": "40", "city": "Nagoya" } ]

🤖 AIに渡すプロンプト（穴埋め済みの完成形）

プロンプトNode-RED の Function ノード用 JavaScript を書いてください。 【やりたいこと】 改行区切りのCSV風文字列を、ヘッダ行をキーにしたオブジェクトの配列に変換したい。 （数値変換は不要、すべて文字列のままでOK） 【入力 msg のサンプル】 ```json { "payload": "name,age,city\nAlice,30,Tokyo\nBob,25,Osaka\nCarol,40,Nagoya" } ``` 【欲しい出力 msg のサンプル】 ```json { "payload": [ { "name": "Alice", "age": "30", "city": "Tokyo" }, { "name": "Bob", "age": "25", "city": "Osaka" }, { "name": "Carol", "age": "40", "city": "Nagoya" } ] } ``` 【制約】 - 1出力で `return msg;` してください。 - 外部ライブラリ不可。 - 空行があれば無視してください。 コードと簡単な解説をお願いします。

📝 期待されるAI回答（一例）

// CSV風文字列 → オブジェクト配列 const text = (msg.payload || "").trim(); if (!text) { msg.payload = []; return msg; } const lines = text.split("\n").filter(l => l.trim() !== ""); const headers = lines[0].split(","); const rows = lines.slice(1).map(line => { const cols = line.split(","); return headers.reduce((obj, key, i) => { obj[key] = cols[i]; return obj; }, {}); }); msg.payload = rows; return msg;
⚠️ このコードの限界
上記は line.split(",") による単純分割の実装で、RFC 4180 準拠の本格CSVには対応していません。具体的には次のケースで破綻します：
  • フィールド値にカンマを含む（例："Smith, Jr"）
  • フィールド値がダブルクォートで囲まれている
  • フィールド値に改行が埋め込まれている
本格CSVを扱う場合は、Node-RED 標準の CSVノード を使うのが正解です。Function ノードで自前パースが必要なのは「区切り文字や形式が独自仕様で、CSVノードで扱えない場合」のみ、と覚えておくと手戻りが減ります。

シナリオ: 1分おきに集めた温度（℃）の配列が msg.payload に入っています。10〜35℃の範囲を「正常」、それ以外を「異常」として、両方を区別して返したい。

🤖 AIに渡すプロンプト（穴埋め済みの完成形）

プロンプトNode-RED の Function ノード用 JavaScript を書いてください。 【やりたいこと】 温度の配列から、10〜35℃の範囲外を「異常値」として抜き出したい。 正常値・異常値を別々の配列にし、件数のサマリーも付けて返してほしい。 （境界の10と35は正常側に含める） 【入力 msg のサンプル】 ```json { "payload": [18.5, 22.1, 25.3, 38.7, 24.0, 23.5, -15.2, 25.1] } ``` 【欲しい出力 msg のサンプル】 ```json { "payload": { "normal": [18.5, 22.1, 25.3, 24.0, 23.5, 25.1], "abnormal": [38.7, -15.2], "summary": { "total": 8, "normal": 6, "abnormal": 2 } } } ``` 【制約】 - 1出力で `return msg;` してください。 - 入力配列が空 `[]` の場合も落ちないようにしてください。 - 閾値は変数で先頭に切り出してください（後で変えやすいように）。 コードと簡単な解説をお願いします。

📝 期待されるAI回答（一例）

// 温度配列から異常値を分離する const MIN_TEMP = 10; const MAX_TEMP = 35; const values = Array.isArray(msg.payload) ? msg.payload : []; const normal = []; const abnormal = []; for (const v of values) { if (typeof v !== "number" || isNaN(v)) continue; if (v >= MIN_TEMP && v <= MAX_TEMP) { normal.push(v); } else { abnormal.push(v); } } msg.payload = { normal, abnormal, summary: { total: values.length, normal: normal.length, abnormal: abnormal.length } }; return msg;

⚠️ 8. AI生成コードの落とし穴と検証チェックリスト

AIが書いたコードは、見た目は正しそうでも Node-RED 特有の罠を踏むことがあります。採用前に以下をチェックしましょう。

🚧 よくある落とし穴トップ7

# 落とし穴 症状 確認ポイント
1 return msg 漏れ 下流ノードに何も流れない 関数の最後に return msg; があるか
2 node.send() と return の二重送信 同じメッセージが2回流れる node.send() を使う場合は return null; にする
3 非同期処理の return 位置ミス 結果が出る前に空のmsgが流れる await や then 内で node.send(msg) し return null;
4 msg の参照共有による副作用 分岐後の合流点で、片方の変更がもう片方にも反映されてしまう 通常は msg.payload = x の代入は安全。ただし1つの msg を複数ノードに分岐で配り、各々で書き換える場合は RED.util.cloneMessage(msg) でコピーしてから渡す。context に msg 自体を保存するのも参照保持の罠
5 context / flow / global の混同 「保存したのに次のmsgで読めない」 スコープが正しいか確認（変数ガイド Part 2）
6 外部ライブラリの require 「Module not found」エラー FunctionノードのSetupタブでライブラリ宣言が必要
7 console.log での確認 エディタのデバッグサイドバーに何も出ない（Node-REDを起動したターミナル側には出力される） サイドバーで確認したい場合は node.warn() もしくはDebugノードを使う。逆にターミナルで見たい場合は console.log でもOK

✅ 採用前チェックリスト

  • 典型的な入力サンプルで、期待する出力が得られた
  • 空入力（"" / [] / {}）でもエラーにならない
  • 想定外の型（数値想定なのに文字列、など）でもクラッシュしない
  • 2回連続でInjectしても、毎回同じ結果になる（状態漏れがない）
  • Debugサイドバーに赤いエラーが出ていない
  • 関数末尾に return msg;（または非同期なら return null; + node.send()）がある
  • 外部ライブラリを使う場合、Setupタブで宣言済み
  • コード内のコメントが日本語で読める／意図が分かる

🎓 9. まとめ＋プロンプト早見表

🌟 このガイドの3つの要点

  1. AIに渡すのは「やりたいこと・入力サンプル・期待する出力」の3点セット。曖昧さを排除すれば一発で動くコードが返ってくる確率が上がる。
  2. コードは必ず Inject + Function + Debug で単体テスト。実フローに組み込むのは、典型値＋境界値で動作確認した後。
  3. 動かなければDebug出力をそのままAIに返して修正依頼。要約せず、生のmsgとエラー全文を貼るのが最短ルート。

📑 やりたいこと別 プロンプトひな型早見表
