📖 1. 概要

TCP ノードは、低レベルのTCP/IPソケット通信を行うためのノードです。HTTPやMQTTなどの高レベルプロトコルを使わず、直接TCPで通信したい場合に使用します。

📞 電話に例えると：TCPは「糸電話」のようなものです。相手と1対1で線をつなぎ、その線を通じてデータを送り合います。HTTPやMQTTはこの「糸電話」の上に決まった会話ルール（プロトコル）を載せたものです。

TCPの特徴

  • 信頼性：データが必ず順番通りに届く（再送制御あり）
  • コネクション型：通信前に接続を確立
  • 双方向通信：一度接続すれば双方からデータ送信可能
  • ストリーム：データは連続したバイト列として扱う（メッセージ境界はプロトコルで保持されない）

⚙️ 2. TCP In ノード

設定項目

TCP In ノードは、設定した相手から届いたデータを受け取り、後続のノードへ流す。接続の持ち方（自分が待ち受けるか、相手へつなぎに行くか）と、受け取ったバイト列をどの単位と形式で出力するかを決める。

設定項目説明既定値
種類接続の持ち方を選ぶ。「待ち受け」は自分がサーバーになり、指定ポートへの接続を待つ。「接続」はクライアントとして相手のホストとポートへつなぎに行く。待ち受け
ポート待ち受けるポート番号（待ち受け時）、または接続先のポート番号（接続時）。必須。（なし）
ホスト接続先のホスト名または IP アドレス。種類が「接続」のときだけ表示される。（空）
SSL/TLS接続を有効化通信を TLS で暗号化する。オンにすると「TLS設定」欄が現れ、証明書の設定ノードを選ぶ。オフ
出力2つのドロップダウンで出力を決める。前は単位（「ストリーム」＝届いたデータを随時出力／「単一」＝接続が閉じるまでまとめて1メッセージ）、後は形式（「バイナリバッファ」「文字列」「Base64文字列」）。詳しくは用語集。ストリーム／バイナリバッファ
区切り文字出力が「ストリーム」かつ「文字列」のときに現れる。指定した文字でメッセージを区切る。空欄なら区切らない。（空）
区切り文字を再追加区切った各メッセージの末尾に、区切り文字を付け直す。オフ
トピック出力する msg に設定する msg.topic の値。（空）
名前エディター上のノード表示名。（空）

受信時の msg プロパティ

プロパティ説明
msg.payload受信データ（Buffer または String）
msg._session接続セッション識別子
msg.ip接続元IPアドレス
msg.port接続元ポート番号

⚙️ 3. TCP Out ノード

設定項目

TCP Out ノードは、受け取った msg.payload を TCP で送り出す。送る相手の決め方（自分が待ち受けるか、相手へつなぐか、TCP In が受けた接続へ返すか）を「種類」で選ぶ。

設定項目説明既定値
種類送り先の決め方。「待ち受け」は自分がサーバーになり、接続してきたクライアントへ送る。「接続」はクライアントとして相手へつないで送る。「TCP応答」は TCP In が受け取った接続へそのまま返信する（msg._session が要る）。接続
ポート待ち受けるポート、または接続先のポート。種類が「TCP応答」のときは表示されない。（空）
ホスト接続先のホスト名または IP アドレス。種類が「接続」のときだけ表示される。（空）
SSL/TLS接続を有効化通信を TLS で暗号化する。オンにすると「TLS設定」欄が現れ、証明書の設定ノードを選ぶ。種類が「TCP応答」のときは表示されない。オフ
メッセージを送信するたびに接続を切断1メッセージ送るたびに接続を閉じる。オフなら接続を保つ。種類が「TCP応答」のときは表示されない。オフ
Base64メッセージの復号msg.payload が Base64 文字列のとき、復号してから送る。オフ
名前エディター上のノード表示名。（空）

送信時に設定できる msg プロパティ

プロパティ説明
msg.payload送信データ（String または Buffer）
msg._session送信先セッション（種類が「TCP応答」のときに必要）

⚙️ 4. TCP Request ノード

TCP Request ノードは、リクエスト送信→レスポンス待機を1ノードで行います。HTTP Request に似た動作です。

設定項目

設定項目説明既定値
サーバ接続先のホスト名または IP アドレス。（空）
ポート接続先のポート番号。（空）
SSL/TLS接続を有効化通信を TLS で暗号化する。オンにすると「TLS設定」欄が現れ、証明書の設定ノードを選ぶ。オフ
戻り値出力する msg.payload の形式。「バイナリバッファ」か「文字列」。バイナリバッファ
終了返信をどの時点で区切って出力するかの条件。5つの選択肢を下の表に示す。選んだ条件に応じて、右の欄に値（時間・文字・文字数）を入れる。指定時間後
区切り文字戻り値が「文字列」かつ終了が「なし - 接続を維持」のときに現れる。指定した文字でメッセージを区切る。（空）
区切り文字を再追加区切った各メッセージの末尾に、区切り文字を付け直す。オフ
名前エディター上のノード表示名。（空）

「終了」の選択肢

選択肢説明右欄の入力
指定時間後送信してから、指定したミリ秒が経過したら出力する。ミリ秒
指定文字の受信時指定した文字（例: 改行 \n）を受信したら出力する。区切り文字
指定数の文字列指定した文字数を受信したら出力する。文字数
なし - 接続を維持接続を閉じずに保ち、データが届くたびに出力する。（入力なし）
即時 - 応答を待たない送信したらすぐ出力し、返信を待たない。（入力なし）

⚠️ 動作確認方法:

サンプルフローをデプロイしたら、TCPサーバーにデータを送ってみて動作を確認できます。初心者の方は、まず「Node-RED内」の方法から試すのがおすすめです。

  • Node-RED内（追加ツール不要）: サンプルフローの「TCP Request（クライアント）」パターンのように、TCP Requestノードで自分自身のTCPサーバーへ接続してテストします。
  • netcat（nc）: ターミナル（コマンドプロンプト）からTCP通信を手軽に試せるコマンドラインツールです。Linux/Macには標準搭載されています（Windowsは別途インストールが必要）。
  • telnet: netcatと同様にターミナルからTCP接続を試せる、古くからある標準ツールです。
  • Python（任意）: Pythonのsocketモジュールでテスト用クライアントスクリプトを自作する方法です。プログラミング経験がある場合に。

パターン1: TCPサーバー（エコー）

サンプルフローの「TCPサーバー（エコー）」を参照してください。

TCP In (Listen) → 処理 → TCP Out (Reply)

ポイント:

  • TCP In を Listen モードで待ち受け
  • TCP Out を Reply モードで送信元に返信
  • msg._session で接続を識別

パターン2: TCP Request（クライアント）

サンプルフローの「TCP Request（クライアント）」を参照してください。

トリガー → TCP Request → 結果

ポイント:

  • TCP Request は接続→送信→受信→切断を1ノードで実行
  • 外部TCPサーバーにコマンドを送信して結果を取得

パターン3: JSONデータの送受信

サンプルフローの「JSONデータの送受信」を参照してください。

JSONデータ → JSON変換 → TCP Out

ポイント:

  • 構造化データはJSON文字列に変換して送信
  • 受信側もJSONノードでパース
  • デリミタ（改行など）で区切りを明確に

パターン4: 接続状態の監視

サンプルフローの「接続状態の監視」を参照してください。

TCP In → 接続管理 → Welcome返信

ポイント:

  • msg.ip, msg.port でクライアント情報を取得
  • flow変数で接続中クライアントを管理
  • ノードステータスで接続数を表示

演習1: シンプルなTCPエコーサーバー 初級

📋 課題: ポート9100で接続を待ち受け、受信したデータをそのまま返すエコーサーバーを作成してください。

🎯 要求仕様:

  • TCP In ノード: 種類「待ち受け」、ポート 9100
  • 受信したデータをそのまま TCP Out で返信

✅ 成功の条件:

  • echo "Hello" | nc localhost 9100 を実行すると「Hello」がそのまま返ってくる
  • デバッグパネルに受信データとクライアントのIPアドレスが表示される
  • 複数回接続しても毎回正常にエコー応答が返ってくる
💡 ヒント

TCP In ノード:

  • 種類: 待ち受け
  • ポート: 9100
  • 出力: ストリーム / 文字列

TCP Out ノード:

  • 種類: TCP応答

TCP In の出力を直接 TCP Out に接続すれば、msg._session が保持されてクライアントに返信されます。

演習2: 時刻付きレスポンス 初級

📋 課題: 受信データに現在時刻を追加して返信するサーバーを作成してください。

🎯 要求仕様:

  • ポート: 9101
  • 受信: Hello
  • 返信: [2024-01-01T12:00:00] Hello

✅ 成功の条件:

  • echo "Hello" | nc localhost 9101 を実行すると [ISO日時] Hello 形式のレスポンスが返ってくる
  • 異なるメッセージを送っても、常に現在時刻が先頭に付いて返ってくる
  • 返答内の日時がリクエストを送った時刻と一致している

✅ 7. まとめ

🎯 重要ポイント:

  • TCP In: 種類「待ち受け」（サーバー）または「接続」（クライアント）として動作
  • TCP Out: 種類「TCP応答」で受信元に返信、または指定ホストに送信
  • TCP Request: リクエスト→レスポンスを1ノードで処理
  • msg._session: 接続を識別（種類が「TCP応答」のときに必須）
  • 区切り文字: 改行（\n）を使うとデータの境界が明確になる

⚠️ 注意事項:

  • ポート番号: 1024以上を使用（1023以下はroot権限が必要）
  • ファイアウォール: 必要なポートを開放
  • データ区切り: TCPはストリームなので、アプリケーション側でデータ境界を管理
  • 接続管理: 切断時の処理を考慮

📘 10. 用語集（TCP 固有の用語）

TCP は「土台」のプロトコルなので、上位の HTTP / MQTT / WebSocket 等とは違う独特の用語があります。3 ウェイハンドシェイク・クライアント/サーバー・ポートなどの通信全般用語は 通信の基礎ガイドの用語集 を参照してください。

🌊 ストリーム指向（stream-oriented）

TCP は 「区切りのない連続したバイトの流れ」としてデータを扱います。送信側で 3 回に分けて send() しても、受信側では 1 回でまとめて届くこともあれば、もっと細かく分かれて届くこともあります。

  • 送ったメッセージの境界（メッセージ単位）はプロトコルが保持しない
  • 例: 送信側が ["hello", "world"] と 2 回送っても、受信側で "helloworld" として届く可能性
  • これに対し UDP は 1 通＝1 メッセージ（メッセージ指向 / データグラム指向）
⚠️ 「データが途中で切れる」「2 つのメッセージがくっついて届く」はバグではなく TCP の仕様どおりです。アプリ側で区切り（デリミタ）の設計が必須になります。

✂️ デリミタ（delimiter / 区切り文字）とメッセージ境界

ストリームの中で「ここまでが 1 メッセージ」と判別する仕組み。TCP では自分で決める必要があります（HTTP や MQTT が上位で勝手にやってくれる）。

方式仕組み例
区切り文字方式特定のバイト（多くは \n=改行）で 1 メッセージの終わりを示すテキストプロトコル全般（SMTP・POP3 など）
長さ指定方式先頭に「これから何バイト来るか」を書く（length prefix）多くのバイナリプロトコル
無音時間で区切る「N ミリ秒データが来なかったらそこで区切る」一定時間データが来なければ確定する独自処理

📥 出力モード（ストリーム / 単一）

TCP In ノードの 「出力」設定。届いたバイト列を どの単位で msg にするか を、前のドロップダウン（単位）と後のドロップダウン（形式）の組み合わせで決めます。

単位意味使いどころ
ストリーム届いたデータを区切らずに随時流す。形式が「文字列」のときは「区切り文字」で 1 msg ずつ切り出せるテキストの行指向プロトコル / 後段の Function で自前パースする場合
単一接続が閉じるまで全部を 1 つの msg にまとめる1 回の接続で完結する短い問い合わせ型

形式（後のドロップダウン）は「バイナリバッファ」「文字列」「Base64文字列」から選びます。「ストリーム」かつ「文字列」のときだけ「区切り文字」欄が現れ、改行（\n）などで msg を区切れます。

📦 セグメント（segment）

TCP がネットワーク上で運ぶ 1 回分のデータ単位。アプリが send() したバイト列は、TCP 層で セグメント（多くは MTU - ヘッダ分のサイズ）に切り分けられ、受信側で組み立て直されます。

  • セグメントの大きさは MSS（Maximum Segment Size）で決まる（Ethernet では通常 1460 バイト）
  • 1 つでも欠ければ、TCP が自動再送して順番どおりに組み立てるため、アプリ側は意識しなくてよい

⏱️ Nagle のアルゴリズム

「小さな送信を集めて、ある程度たまってからまとめて送る」最適化。帯域効率を高める代わりに、送信側でわずかな遅延（通常 200ms 程度）が生じます。

  • 1 文字ずつのキー入力など 細かい送信を多数する場合に有効
  • ただし対話型・リアルタイム性が必要な通信では邪魔（チャット、ゲーム、産業制御等）
  • OS の TCP ソケットで TCP_NODELAY オプションを有効にすると無効化できる（Node-RED のノード設定からは直接触れないが、症状を理解しておくと役立つ）

💓 TCP Keep-Alive

長時間データが流れない接続が 「まだ繋がっているか」を OS レベルで確認 する機構。プロキシ・NAT・ルーターが「もう使われていない」と判断して接続を切るのを防ぎます。

  • OS が一定時間（既定 2 時間など）通信がないと、Keep-Alive プローブを送る
  • 応答がなければ「切れた」と判定して接続を閉じる
  • 多くのアプリでは、より上位プロトコル側のハートビート（WebSocket の Ping/Pong、MQTT の Keep Alive など）で対応する方が確実

🔐 TLS over TCP（SSL/TLS でラップ）

TCP 自身は暗号化を持たないため、TCP の上に TLS を重ねて暗号化通信にする。これが HTTPS（HTTP + TLS）や MQTTS の正体です。

  • Node-RED の TCP ノード設定の TLS オプションで、信頼できる証明書を持つ TLS Configuration ノードを指定する
  • 独自の TCP プロトコルでも、相手が TLS 対応していれば暗号化通信が可能
  • 詳しくは 通信の基礎ガイド用語集 の「TLS / SSL / HTTPS」を参照
