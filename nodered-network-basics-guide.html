Network（ネットワーク）セクションの各ガイドは、いきなり「HTTP」「リクエスト」「ポート」といった言葉から始まります。でも、それらは通信という大きな地図の中の一地点にすぎません。地図を持たずに歩き出すと迷子になります。

このページでは、身近なたとえ話だけを頼りに通信の全体像をつかみます。専門用語は最後にまとめて回収するので、いま分からない単語があっても読み飛ばして大丈夫です。

1. そもそも「通信」とは？

通信とは、ひとことで言えば「離れた相手とデータをやり取りすること」です。人間同士のやり取りに置き換えると、実は方法がいくつもあることに気づきます。

📮 手紙を送る

宛先を書いて投函すると、後で相手に届く。送りっぱなしで、その場の返事は期待しない。

📞 電話で話す

つながっている間、お互いがいつでも話せる。双方向で、リアルタイム。

🏪 お店で注文する

こちらが「これください」と頼むと、店員が品物を返してくれる。要求して、応答が返る往復型。

📰 雑誌を定期購読する

出版社が発行すると、購読を申し込んだ全員に届く。発行側と受け取り側が直接知り合う必要はない。

🔦 見回り点検する

担当者が定期的に各機械を回り「いま何度？」と聞いて値を控える。こちらから定期的に問い合わせる方式。

コンピュータの通信も、まさにこの5つのやり方をそのまま電子化したものです。この後で出てくる HTTP・WebSocket・MQTT・TCP/UDP・MODBUS は、それぞれ上のどれかに対応しています。新しい概念はありません——日常にある「やり取りの形」を、機械同士でやっているだけです。

💡 ここだけ覚えれば OK
通信とは「誰が・どこへ・どんな形でデータを送るか」を決めること。次の章から、この「誰が（登場人物）」「どこへ（住所）」「どんな形で（プロトコル）」を順番に見ていきます。

2. 登場人物：クライアントとサーバー

通信には必ず2つの役割が登場します。

🙋 クライアント 頼む側

サービスをお願いする側。お店でいう「お客さん」。「このデータちょうだい」「これ処理して」と要求を出します。

🏪 サーバー 応える側

要求に応える側。お店でいう「店員」。待ち構えていて、頼まれたら品物（データ）を返します。

サーバーは「サーブ（serve＝給仕する）する人」、クライアントは「お得意さん（client）」が語源です。意味そのままです。お店（サーバー）はいつでも開いていて、お客さん（クライアント）が来たら応対する、という関係です。

🔑 Node-RED は「両方」になれる

ここが Node-RED の面白いところです。Node-RED はクライアントにもサーバーにもなれます。

  • クライアントとして: 外部の天気APIに「東京の天気ちょうだい」と聞きに行く（http request ノード）
  • サーバーとして: 自分が窓口を開いて、外部からのアクセスを待ち受ける（http in ノード）

どちらの役をやらせるかで、使うノードが変わります。「いま自分はお客さん側？お店側？」を意識すると、ノード選びで迷わなくなります。

3. 住所のしくみ：どうやって相手を見つける？

手紙を出すには宛先が要ります。通信も同じで、相手を特定する「住所」が必要です。コンピュータの世界の住所は、建物にたとえると分かりやすいです。

通信の用語 建物でたとえると 具体例
IPアドレス 建物の住所（番地） 192.168.1.10
ポート番号 建物内の部屋番号・窓口番号 80（Web窓口）、1883（MQTT窓口）
ホスト名 / ドメイン 建物の看板の名前 example.com
DNS 電話帳（名前 → 住所の変換） example.com → 93.184.216.34

📍 IPアドレス＝建物の住所

ネットワーク上のすべての機器には、世界で（あるいは社内で）重複しない番号が割り当てられます。これが IPアドレスです。192.168.1.10 のような4つの数字でできています。手紙の宛先住所にあたります。

🚪 ポート番号＝部屋番号

1つの建物（＝1台のコンピュータ）の中には、いくつもの窓口があります。「Web の問い合わせはこの窓口」「メールはあの窓口」と分かれていて、その窓口番号がポート番号です。よく使う番号は世界共通で決まっています。

ポート番号 用途（窓口） 関連プロトコル
80 Web（暗号化なし） HTTP
443 Web（暗号化あり） HTTPS
1883 IoTメッセージ MQTT
502 産業機器の通信 Modbus TCP
1880 Node-RED エディタ自身 （Node-RED の管理画面）
💡 1880 はもうおなじみ
ブラウザで Node-RED を開くとき http://localhost:1880 と打ちますね。この 1880 がまさにポート番号です。「自分のパソコン（localhost）の 1880 番窓口にいる Node-RED に、Web（http）でアクセスする」という意味だったのです。

📖 DNS＝電話帳

人間は 93.184.216.34 のような数字を覚えられないので、example.com という名前を使います。名前を実際のIPアドレスに変換してくれる仕組みが DNS（Domain Name System）です。電話帳で「名前」から「電話番号」を引くのと同じです。

🔗 URL＝宛名を1行にまとめたもの

ブラウザのアドレス欄でおなじみの URL は、ここまでの「住所・窓口・要求」を1行に詰め込んだ宛名です。分解すると意味が見えてきます。

https://api.example.com:443/weather/tokyo
部分呼び名意味
https://スキームどの方式（プロトコル）で話すか＝「Webの暗号化通信で」
api.example.comホスト名どの建物（サーバー）か＝DNSでIPに変換される
:443ポートどの窓口か（省略時は方式ごとの既定値。httpsなら443）
/weather/tokyoパスその建物の中の、どの書類・どの機能か
📌 「ポートが省略できる」理由
URL で :443 をふだん書かないのは、「https なら 443 番」と世界で決まっているからです。プロトコル（方式）を指定した時点で、既定の窓口番号が自動的に決まる——だから省略できるのです。

4. プロトコル＝通信の「共通ルール」

プロトコル（protocol）とは、通信における「話し方の約束事」です。日本語では「通信規約」と訳されます。

たとえば電話では「もしもし」→「はい、○○です」という決まった流れがあります。先に名乗らず無言だと会話が成立しません。これと同じで、機械同士も「最初にこう送る、次にこう返す」という手順を事前に決めておく必要があります。その取り決めがプロトコルです。

💡 プロトコルは「方言」のようなもの
HTTP、MQTT、Modbus……これらはすべてプロトコル＝話し方の種類です。日本語と英語で話し方が違うように、相手に通じる方言（プロトコル）をそろえないと会話できません。送る側と受ける側が同じプロトコルを使う——これが通信成立の大前提です。

そして、用途に応じて「どの話し方が向いているか」が変わります。次の章で、代表的な5つの話し方を、第1章のたとえ話に対応させて見ていきます。

5. 通信の「型」5パターン

Network セクションに並ぶプロトコルは、第1章の身近なたとえと1対1で対応しています。「これは手紙型」「これは電話型」と分類できれば、もう半分理解したようなものです。

🏪 HTTP — リクエスト/レスポンス型

たとえ：お店で注文する

クライアントが「これください（リクエスト）」と送ると、サーバーが「はいどうぞ（レスポンス）」と返す往復型。1回の要求に1回の応答がセットです。Web の閲覧、外部API呼び出し、フォーム送信はすべてこれ。最も基本的で、最もよく使う通信です。

👉 Node-RED: http request（聞きに行く側）、http in + http response（窓口になって応える側）

📞 WebSocket — 双方向リアルタイム型

たとえ：電話で話す

一度つなぐと回線を保ったまま、どちらからでも・いつでもデータを送り合えます。HTTP が「1回ごとに電話をかけ直す」のに対し、WebSocket は「つなぎっぱなし」。チャット、リアルタイムな画面更新、ゲームなど、即時性が要る場面で使います。

👉 Node-RED: websocket in / websocket out

📰 MQTT — Pub/Sub（出版・購読）型

たとえ：雑誌を定期購読する

送信者はブローカー（仲介サーバー）の「トピック」にデータを発行（Publish）し、そのトピックを購読（Subscribe）している人全員に配られます。送信者と受信者は互いを知らなくてよいのが特長。1つのセンサー値を複数の受け手に同時配信でき、IoTで多用されます。

👉 Node-RED: mqtt in / mqtt out（第0章⑤でも登場した、IoTの主役プロトコル）

📦 TCP — 低レベルソケット型（書留）

たとえ：書留で手紙を送る

HTTP や MQTT の土台になっている、より低い層の通信です。「確実に・順番どおりに・欠けずに届く」ことを保証します（届かなければ再送）。決まったプロトコルがない独自機器や、古い装置との通信で、生のTCPを直接使うことがあります。

👉 Node-RED: tcp in / tcp out / tcp request

📨 UDP — 低レベルデータグラム型（はがき）

たとえ：はがきを送る／街頭放送

TCP と対照的に、確認も再送もしない代わりに、軽くて速い通信です。1枚届かなくても気にしない用途——映像・音声のストリーミング、ログ送信、ネットワーク上の機器発見やブロードキャスト（一斉送信）に向きます。

👉 Node-RED: udp in / udp out

🔦 MODBUS — ポーリング型（産業機器向け・サードパーティ）

たとえ：見回り点検

工場のPLC・インバータ・電力計・温調計などをつなぐ、産業界の定番プロトコルです。親機（マスター）が子機（スレーブ）に「○番のレジスタの値ちょうだい」と定期的に問い合わせて値を集めます。Node-RED標準には含まれず、サードパーティノード（node-red-contrib-modbus）を追加して使います。

👉 Node-RED: 別途インストールが必要 → MODBUS ノードガイド

6. プロトコル早見表 — どれをいつ使う？

このページで一番持ち帰ってほしいのが、この比較表です。「やりたいこと」から逆引きできるようにまとめました。迷ったらここに戻ってきてください。

プロトコル 通信の型 つなぎ方 信頼性 向いている用途 難易度
HTTP / HTTPS リクエスト/レスポンス（往復） 都度つなぐ 高い Web API、フォーム、Webhook、外部サービス連携 中級
WebSocket 双方向リアルタイム つなぎっぱなし 高い チャット、リアルタイム更新、即時通知 中級
MQTT Pub/Sub（多対多） つなぎっぱなし（ブローカー経由） 高い（QoS選択可） IoTセンサー、多拠点集約、1対多配信 中級
TCP 低レベルソケット つなぎっぱなし／都度 高い（順序保証・再送） 独自プロトコル機器、レガシー装置 上級
UDP 低レベルデータグラム つながない（送りっぱなし） 低い（再送なし） 映像/音声、ログ、機器発見、ブロードキャスト 上級
MODBUS ポーリング（問い合わせ） つなぎっぱなし 高い PLC・インバータ・電力計など産業機器 応用（要追加ノード）
💡 最初の1本に選ぶなら HTTP
5つのうち、まず手をつけるなら HTTP がおすすめです。Web の仕組みそのもので、応用範囲が最も広く、他のプロトコルを学ぶ土台にもなります。IoTでセンサーをつなぎたい人は、その次に MQTT へ進むとスムーズです。

7. Node-RED での位置づけ

これまで学んだ msg（メッセージ）の考え方は、通信でもそのまま活きます。Network ノードは、フロー内の msg を外の世界に送り出したり、外から受け取って msg に変えたりする「出入口」です。

外の世界 Node-RED のフロー ───────── ───────────────── 外部API・他の機器 ⇄ [ Network ノード ] → [ Function ] → [ Debug ] （HTTP/MQTT/…） （通信の出入口） （いつもの msg 処理）

つまり、通信ノードでデータを受け取ってしまえば、あとは今まで通り。Change で整え、Switch で振り分け、Function で加工し、Dashboard で見せる——学んできたことが全部そのまま使えます。「通信」は新しい世界ではなく、これまでのフローに「外との出入口」を1つ足すだけなのです。

🧭 役割で選ぶノード早見

  • 外に聞きに行きたい（クライアント） → http request、tcp request
  • 窓口を開いて待ち受けたい（サーバー） → http in + http response、websocket in
  • IoTで値を配信・購読したい → mqtt out / mqtt in
  • 産業機器から値を集めたい → node-red-contrib-modbus（追加インストール）

8. 用語集

ここまでに出てきた言葉を1行ずつまとめました。各プロトコルのガイドを読んでいて詰まったら、ここに戻ってきてください。

用語ひとことで言うと
通信 / ネットワーク離れた機器どうしでデータをやり取りすること、その仕組み。
クライアントサービスを要求する側（お客さん）。
サーバー要求に応えてサービスを提供する側（お店）。
プロトコル通信の話し方の約束事。HTTP・MQTT などの「方式」のこと。
IPアドレス機器を識別する番号。建物の住所にあたる（例: 192.168.1.10）。
ポート番号1台の中の窓口番号。用途ごとに決まっている（例: HTTP=80）。
ホスト名 / ドメインIPアドレスにつけた人間用の名前（例: example.com）。
DNS名前 → IPアドレスを変換する電話帳の仕組み。
URL方式・住所・窓口・場所を1行にまとめた宛名（例: https://example.com/path）。
API機械が機械に向けて用意した「窓口」。多くはHTTPで提供される。
リクエスト / レスポンス要求 / それへの応答。HTTP通信の基本ペア。
ペイロード実際に運ばれる中身のデータ。Node-REDの msg.payload と同じ発想。
パケットデータを小さく区切った1つの荷物。ネットワークはこの単位で運ぶ。
ソケット通信の出入口の総称。「IPアドレス＋ポート」で1つの接続口を表す。
ブローカーMQTTで送信者と受信者を仲介するサーバー。
トピックMQTTでデータを仕分ける宛先ラベル（例: sensor/temperature）。
Pub/Sub発行（Publish）と購読（Subscribe）で成り立つ配信方式。MQTTの方式。
TLS / HTTPS通信を暗号化して盗み見を防ぐ仕組み。httpsの「s」はこれ。
ポーリングこちらから定期的に「いまどう？」と問い合わせる方式。MODBUSが代表。
ブロードキャスト同じネットワーク内の全員へ一斉送信すること。UDPで使う。

10. まとめ：通信の地図が手に入った

  • 通信は「誰が・どこへ・どんな形で」データを送るか。難しい概念ではなく、日常のやり取りの電子版。
  • 登場人物はクライアント（頼む側）とサーバー（応える側）。Node-RED は両方になれる。
  • 住所はIPアドレス（建物）＋ポート（窓口）。名前で呼ぶための DNS、宛名をまとめた URL。
  • プロトコルは話し方の約束事。送る側と受ける側で同じものを使う。
  • 通信の型はHTTP（注文）・WebSocket（電話）・MQTT（定期購読）・TCP/UDP（書留/はがき）・MODBUS（見回り）の5パターン。
  • 通信ノードはフローに「外との出入口」を足すだけ。受け取った後はいつもの msg 処理。

これで、各プロトコルガイドに出てくる「HTTP」「ポート」「リクエスト」といった言葉が、地図のどこにある話なのか分かるようになりました。安心して次のページへ進んでください。
