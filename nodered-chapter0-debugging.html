🌱 第0章④ 動かないときの進め方

📖 このページの前提③最初の10分ハンズオンを一度やってみた方を想定しています。まだの方は先にハンズオンを走らせてから読むと、例えの意味が腹落ちします。

🎯 得られるものフローが動かない、期待と違う値になる、エラーが出る――そんなときに「どこから手をつければいいか」が分かります。プログラミング全般に使える思考パターンです。

📘 用語を忘れたら①の用語辞典に戻れます。

➡️ 次に読むページ⑤ IoT全体像

1. 「動かない」は誰でも毎日経験する

ベテランのプログラマーでも、書いたコードが一発で動くことは稀です。むしろ「動かない時間の方が、動く時間より長い」のが普通です。初心者が「自分には才能がない」と感じてしまう最大の原因は、この事実を知らないまま画面と向き合っていることにあります。

プログラミングで身につけるべき本当のスキルは「最初から完璧に動くものを書く能力」ではなく、「動かないときに原因を切り分けて、一歩ずつ正解に近づける能力」です。このページで、その進め方を一通り体験してもらいます。

💡 デバッグという言葉
「バグ（bug）」は不具合、「デバッグ（debug）」は不具合を取り除く作業のことです。コンピュータの中から虫（bug）を取り出す、という語源からきています。Node-RED の右サイドバーにある虫アイコン（🐛）もこれが由来です。

2. まず冷静にやる 3 つのこと

画面を前にパニックになる前に、まず次の 3 つを必ずやってください。この 3 つで解決する問題が、体感で6〜7 割あります。

🧘 最初の 3 秒でやる「3 点確認」

  1. デプロイボタンを押したか？ 編集しただけでは反映されません。右上の赤いボタンがグレーになっていれば最新の内容が反映済みです。赤いままなら未デプロイ。
  2. ワイヤーは繋がっているか？ 線が引けているつもりで、実は途中で離していることがあります。ノードの端子から端子まで本当に繋がっているか目視確認。
  3. デバッグサイドバーは開いているか？ 右上の虫アイコンをクリック。開いていないと Debug ノードの出力は見えません。

この 3 点で解決しなかった場合は、いよいよ本格的な切り分けに入ります。

3. トラブルを 4 種類に分類する

「動かない」と一言で言っても、実は中身は次の 4 種類に分かれます。どの種類かを最初に見極めると、打つ手が絞られます。

(a) 全く動かない

Inject ボタンを押しても、デバッグサイドバーに何も出ない。ノードのステータスランプも反応しない。

例: 配線ミス、未デプロイ、ノード無効化

(b) 期待と違う値

動いてはいる。デバッグサイドバーに何か出ている。ただし値が思っていたのと違う。

例: Change の設定ミス、型の取り違え

(c) エラーが出る

赤いエラーメッセージがデバッグサイドバーや画面右下のトーストに出る。

例: 文字列を数値として計算、型エラー

(d) 不定期に動かない

動くこともあれば動かないこともある。朝は動いたのに夕方は動かない等。

例: タイミング、外部サービスの障害

以降、(a) 〜 (d) それぞれの進め方を解説します。自分の状況に当てはまるところから読んでください。

🔴 (a) 全く動かない時

まずは「そもそもトリガーが発火しているか」を疑います。ゴール側から原因を探すのではなく、入口側から「電気が流れてきているか」を見るイメージです。

チェックリスト：発火側から順番に確認

  1. デプロイボタンを押したか？（前述。もっとも多い原因）
  2. Inject ノードが「無効」になっていないか？ ノードをダブルクリックして「有効」になっているかを確認。フロー全体が無効化されていることもある（タブ右上のトグル）。
  3. ノードに赤い三角や赤い枠線が付いていないか？ 設定不備のサイン。ダブルクリックして必須項目が埋まっているか見る。
  4. ワイヤーが途中で切れていないか？ 拡大して端子にしっかり接続されているか確認。
  5. Debug ノードが「メッセージを送る先」は「デバッグサイドバー」になっているか？ 「コンソール」や「ノードステータス」だけだとサイドバーには出ない。
  6. サイドバーの「フィルタ」が特定ノードに絞られていないか？ デバッグサイドバー上部のドロップダウンが「選択したノード」になっていると他のノードの出力が見えない。
💡 デバッグの金科玉条「ワイヤーの途中に Debug を挿す」
動いているかどうか分からないとき、ワイヤーの途中に Debug ノードを分岐で繋いで「この地点まで msg は来ているか」を確認します。これを何箇所かに挿すだけで、どこから先が動いていないかが一気に見えてきます。捨てる前提で気軽に挿しましょう。

🟠 (b) 期待と違う値が出る時（分割統治）

動いてはいるが値がおかしいとき、ベテランが必ず使うのが「分割統治（ぶんかつとうち）」という考え方です。原因の可能性がある範囲を半分に絞り、また半分に絞る、を繰り返します。

やり方：Debug を 2〜3 箇所に挿して「どこで値がおかしくなったか」を特定する

例えば Inject → Change → Change → Debug というフローで最終的な値が想定と違うとき、以下のようにします。

Inject → 🐛 Debug A → Change 1 → 🐛 Debug B → Change 2 → 🐛 Debug C
  • Debug A の値が既におかしい → 原因は Inject 側の設定
  • A は正しいが B でおかしい → 原因は Change 1 の設定
  • A と B は正しいが C でおかしい → 原因は Change 2 の設定

このように挟み撃ちしていくと、「どのノードが犯人か」を機械的に特定できます。頭で考え込むより、まず Debug を挿して事実を見る方が早いです。

💡 型の取り違えは初心者あるある
Inject の payload 種別が「文字列」のまま 25 と入れると、これは文字の "25" であって数値の 25 ではありません。「数値として扱っているつもりが、実は文字列だった」という勘違いは非常に多い落とし穴です。Debug サイドバーでは、文字列は "25" のように色とクオートで区別して表示されます。

🟣 (c) エラーが出る時

赤いエラーメッセージが出ているときは、そのエラー文面こそが最大のヒントです。英語でも怖がらず、1 行目と最後の行だけでも読んでみてください。

エラー文面の読み方

例えばこんなメッセージが出たとします。

TypeError: Cannot read properties of undefined (reading 'temperature')

パッと見難しそうですが、分解すると意味は取れます。

英単語 意味 読み解き
TypeError 型エラー データの種類（型）が想定と違う
Cannot read properties of undefined undefined のプロパティは読めない 空っぽ（存在しないもの）から中身を取り出そうとした
(reading 'temperature') 'temperature' を読もうとした 具体的に temperature というキーを取ろうとして失敗した

つまりこのエラーは「空っぽのものから temperature というプロパティを取ろうとして失敗した」と言っています。原因はたいてい、上流ノードからオブジェクトが来ていないのにプロパティを取ろうとしているケースです。Debug で上流の値を見ると、本当に空かどうかが確認できます。

💡 エラー文面はそのままコピーして検索
メッセージを丸ごとコピーして Google 検索すると、同じエラーで困っている人の記事がほぼ必ず出てきます。エラー文面は、あなただけがつまずいている問題ではありません。他の人の解決策が使えることがほとんどです。

よく出るエラーの早見表

エラー文（キーワード） 原因の方向性
undefined 存在しないプロパティを触った。上流の msg の形を確認。
TypeError 文字列を数値として計算、配列でないものに .length 等、型の不一致。
ECONNREFUSED 外部サーバーに接続拒否された。URL、ポート番号、サーバー稼働状態を確認。
ENOTFOUND ホスト名が見つからない。URL のスペル、ネットワーク（WiFi）を確認。
Timeout 応答がなかった。外部サービスの遅延、ネットワーク不安定。
EACCES / Permission denied ファイル／ポートへのアクセス権がない。権限設定を確認。

🟢 (d) 不定期に動かない時

最も厄介なのが「動くときと動かないときがある」パターンです。原因が実行タイミングや外部サービスに依存しているため、机上で考えても答えは出ません。

不定期トラブルの切り分け方

  1. まず事実を集める: 「いつ動かなかったか」「その時の msg の中身」を記録する。Debug サイドバーは過去のログが残るので、そのままメモ代わりにできる。
  2. 時刻との関連を疑う: 夜動かない＝ネットワーク変動、朝動く＝起動直後だけ OK など、時間帯パターンを見る。
  3. 外部サービスを疑う: HTTP 通信、MQTT ブローカー、API 連携が絡んでいれば、サービス側の障害情報を確認。
  4. 順序依存を疑う: 複数のノードが同時に動く場合、実行順が毎回同じとは限らない。順序が重要なら Delay や明示的な連結で整える。
  5. 再起動で直るかを確認: Node-RED 自体、あるいは Raspberry Pi を再起動して再発するかを見る。再起動で直るなら、起動時の初期化問題が疑われる。
⚠️ 「たぶん直った」で終わらせない
不定期トラブルは、触っているうちにたまたま再現しなくなって「直ったっぽい」となることがよくあります。が、根本原因が分からないまま放置すると、後で本番環境で再発します。なぜ直ったかが説明できるまで追うのが理想です（ただし時間との相談で、再発時に改めて追うのもあり）。

4. よくある勘違い 10 選

初心者が最初の数週間で必ずと言っていいほど踏む落とし穴をまとめました。心当たりがあれば、その場で修正できます。

  1. 「編集したら自動で反映される」と思っている。 ✅ 正しい理解: デプロイボタンを押さないと反映されません。画面上の変更と、実際に動いているフローは別物だと覚える。
  2. Inject に「文字列 25」を入れて計算しようとしている。 ✅ 正しい理解: "25" は文字、25 は数値。計算に使うなら payload 種別を「数値」に切り替える。
  3. msg.payload と msg.topic を同じものだと思っている。 ✅ 正しい理解: payload は「本文（データ）」、topic は「タイトル・識別子」。住所でいうと payload が中身、topic が宛名ラベル。
  4. 「Debug が何も出ない＝動いてない」と決めつけている。 ✅ 正しい理解: Debug の設定が「完全なメッセージ」ではなく、ピンポイントで空のプロパティを指しているだけかもしれない。設定は msg.payload か「完全なメッセージ」に。
  5. Change ノードで「置き換え」と「追加」を混同している。 ✅ 正しい理解: 「値の代入」は既存の値を上書き、「削除」はプロパティそのものを消す。やりたいことに合っているかを動作から確認する。
  6. 「このノード赤いから壊れてる」と慌てる。 ✅ 正しい理解: 赤は「設定不備」か「エラー状態」のサイン。壊れているのではなく、あと一歩の設定が足りていないだけ。
  7. ダッシュボードが真っ白で「動いてない」と判断する。 ✅ 正しい理解: 一度 /dashboard を開いてから、元の値が送られていればウィジェットに出る。ページを開く前に Inject を押していると、ウィジェットには何も残らないことがある。
  8. エラーメッセージを読まずに「なんかエラー出た」で終わる。 ✅ 正しい理解: エラー文面は最大のヒント。最低でも 1 行目と最後の行を読み、分からない単語はそのままコピーして検索する。
  9. ノードを消したらワイヤーが勝手に残って「バグった」と思う。 ✅ 正しい理解: Node-RED は宙ぶらりんのワイヤーも含めて保存するが、警告表示はされる。不要なワイヤーはクリックで選択して Delete キー。
  10. 「自分のやり方が悪いに違いない」と自分を責める。 ✅ 正しい理解: 動かないのはバグの性質上、普通のこと。淡々と (a)〜(d) の切り分けに戻り、事実を一つずつ確認する。感情ではなく事実で判断する。

5. 最終手段：ミニマム再現と人に聞く

(a) 〜 (d) を試しても解決しないときは、次の 2 つのどちらか、または両方をやります。

ミニマム再現フロー

問題が起きているフローから、関係ないノードを全部削って最小にする。そうすると、原因が見えるか、あるいは「消したらなぜか直った」という形で真犯人が見えることがあります。最小化の過程で答えにたどり着くケースは非常に多いです。

💡 ラバーダック法（ひとり言作戦）
誰もいない部屋で、アヒルのおもちゃ（ラバーダック）を相手に「今こういうフローを作って、ここがこうなって、それでうまくいかないんだ」と声に出して説明します。英語圏の開発者の定番手法で、説明している途中に自分で原因に気づくことが非常に多いとされています。アヒルがなければ、スマホのメモアプリやノートでもOKです。

人に聞くときのコツ

Node-RED 公式フォーラム、社内のチャット、Twitter の #NodeRED タグなどで聞くときは、以下をセットで書くと答えが返りやすくなります。

  1. やりたいこと（目的）: 「Inject で 1 分ごとに温度を取得してダッシュボードに出したい」
  2. 作ったもの（手段）: フローのスクリーンショット、またはJSON エクスポートを貼る
  3. 起きたこと（事実）: 「デプロイ後に何も表示されない」「エラー X が出る」
  4. 試したこと（切り分け）: 「Debug を途中に挿したら Change の前までは値が来ていた」

この 4 点セットは、自分で問題を整理する過程でもあります。書いているうちに原因に気づくことも珍しくありません。

6. デバッグ力は経験で育つ

最後に大事なことを 1 つ。デバッグは才能ではなく経験です。「あ、これは前にも見た型エラーだ」「この症状は〇〇ノードの設定ミスっぽい」という勘は、失敗の数だけ育ちます。詰まるたびに、次の 2 つをメモする習慣をつけてください。

  • どんな症状だったか
  • 何が原因で、どう直したか

このメモが 20 件もたまる頃には、ほとんどの「動かない」は自力で 5 分以内に解決できるようになります。

7. まとめ：動かないときのフローチャート

  1. 3 点確認: デプロイ／ワイヤー／デバッグサイドバーを見る
  2. 4 分類: 全く動かない／値が違う／エラー／不定期、のどれかを見極める
  3. Debug を挿す: 分割統治で、どの区間で値が崩れたかを特定する
  4. エラー文を読む: 分からない単語はコピーして検索
  5. 10 勘違いを疑う: 初心者あるあるを総当たりでチェック
  6. ミニマム再現: 関係ないものを削って最小のフローにする
  7. 人に聞く: 目的・手段・事実・試したこと、の 4 点セットで

ここまで読んでくれた方は、もう「動かない」に対する基本的な向き合い方を身につけています。実際に手を動かす中で詰まったら、このページに戻ってきて該当箇所を読み直してみてください。
