Node-REDのSwitchノードについて、初心者向けに説明します。

1. Switchノードとは

「Switch」の役割

Switchノードは、条件に応じてメッセージの送り先を振り分けるノードです。 郵便局の仕分けセンターに例えると分かりやすいでしょう。

郵便仕分けセンターに例えると:

  • 📬 宛先を確認 = メッセージのプロパティを評価
  • 🏷️ 地域ごとに分類 = 条件ルールに基づいて振り分け
  • 📦 該当する配送ルートへ = 対応する出力ポートへ送信
  • ❓ 該当なしは別処理 = どの条件にも合わない場合の処理

基本的な動作

Inject → Switch → 出力1

↘ 出力2

↘ 出力3

Switchノードは入力メッセージを評価し、条件に一致する出力ポートへメッセージを送信します。 複数の条件を設定でき、プログラミングのif-else文やswitch-case文に相当します。

Switchノードの動作イメージ:

2. Switchノードの条件タイプ

比較 値の比較

==, !=, <, >, <=, >= などで比較

例: payload == 100 例: payload > 0 例: payload != "error"

範囲 値の範囲

値が指定した範囲内かどうか

例: 10 <= payload <= 50 例: payload が 0〜100の間

正規表現 パターンマッチ

正規表現で文字列をマッチング

例: ^error.* 例: .*@gmail\.com$

型チェック データ型

値のデータ型をチェック

例: is null 例: is not null 例: is of type number

シーケンス 順序制御

メッセージシーケンスの順序で判定

例: head (最初のN件) 例: tail (最後のN件) 例: index (特定位置)

JSONata 式評価

JSONata式で複雑な条件を記述

例: payload.temp > 30 例: $count(items) > 5

条件タイプ一覧

条件タイプ 記号/名称 説明 例
等しい == 値が等しい payload == 100
等しくない != 値が等しくない payload != 0
より小さい < 値より小さい payload < 0
以下 <= 値以下 payload <= 100
より大きい > 値より大きい payload > 0
以上 >= 値以上 payload >= 0
範囲内 between 範囲内にある 0 <= payload <= 100
含む contains 文字列を含む payload に "error" を含む
正規表現 matches regex 正規表現にマッチ ^[0-9]+$
型チェック is of type データ型を確認 is of type string
空でない is not empty 配列・文字列・オブジェクトが空でない payload is not empty
それ以外 otherwise どの条件にも一致しない場合 デフォルト処理

設定項目

設定項目説明既定値
プロパティ各ルールで評価する対象の値。msg のプロパティや式を指定する。msg.payload
条件振り分けの判定ルール。1 行が 1 つの出力端子に対応し、上から順に評価される。複数追加できる。1 件（==）
評価の範囲条件リストの下にあるドロップダウン。「全ての条件を適用」は一致したすべてのルールの端子へ送る。「最初に合致した条件で終了」は最初に一致した端子だけに送り、残りは評価しない。全ての条件を適用
メッセージ列の補正分割済みメッセージ（msg.parts 付き）を振り分けたとき、各端子で parts を振り直して列として整合させる。Split と組み合わせるときに使う。オフ
名前エディター上の表示名。動作には影響しない。（空）

パターン1: 数値の正負判定

数値が正・ゼロ・負かを判定して振り分けるパターンです。

-10 / 0 / 10 → Switch
(正負判定) → Negative → Debug

ルール設定:

1 payload < 0 → 出力1（負の数）
2 payload == 0 → 出力2（ゼロ）
3 payload > 0 → 出力3（正の数）
-10 → 出力1: negative
0 → 出力2: zero
10 → 出力3: positive

設定例:

プロパティ: msg.payload ルール1: < 0 → 出力1 ルール2: == 0 → 出力2 ルール3: > 0 → 出力3 全ルールを確認: 有効（複数条件に一致する可能性がある場合）

パターン2: トピックによるルーティング

msg.topicの値でメッセージを振り分けるパターンです。

temp/humid/press → Switch
(topic) → Temperature

センサー振り分けの用途:

  • 複数センサーからのデータを種類ごとに処理
  • MQTTトピックに基づくメッセージルーティング
  • デバイスIDごとの処理分岐
temperature → 出力1: 温度処理
humidity → 出力2: 湿度処理
pressure → 出力3: 気圧処理

設定例:

プロパティ: msg.topic ルール1: == "temperature" → 出力1 ルール2: == "humidity" → 出力2 ルール3: == "pressure" → 出力3

応用パターン（サンプルフローには含まれていません）:

  • 「全ルールを確認」を有効にすると、複数条件に一致した場合すべての出力ポートに送信される
  • 「otherwise」でどの条件にも一致しない場合のデフォルト処理を設定できる
  • 「is between」で値が特定範囲内かを判定できる（温度監視など）
  • 「matches regex」で文字列パターンを柔軟に判定できる

4. 実践演習

演習1: 成績判定初級

テストの点数に応じて評価（A/B/C/D）を出力するフローを作成してください。

要求仕様:

  • Inject、Switch、Debugノードを使用
  • 90点以上: A、70点以上: B、50点以上: C、50点未満: D
  • 各評価ごとに別のDebugノードに出力

期待される動作:

  • 85点を入力 → 出力2（B評価）に送信

成功の条件:

  • 85点を入力したとき、B評価のDebugノードにのみメッセージが届く（A・C・D評価のDebugには届かない）
  • 95点を入力するとA評価、45点を入力するとD評価のDebugノードに正しくルーティングされる
  • 境界値（90点、70点、50点）が正しい評価に分類される
  • 「最初に一致で終了」が有効なため、複数のDebugノードに同時にメッセージが届かない
ヒント

Switchノードの設定:

  • プロパティ: msg.payload
  • ルール1: >= 90 → A
  • ルール2: >= 70 → B
  • ルール3: >= 50 → C
  • ルール4: otherwise → D

「最初に一致で終了」を有効にする（または条件を排他的に設定）。

演習2: コマンド処理中級

文字列コマンド（start/stop/restart）を判定し、それぞれ異なる処理を行うフローを作成してください。

要求仕様:

  • msg.payloadにコマンド文字列が入力される
  • "start" → 「システムを起動します」
  • "stop" → 「システムを停止します」
  • "restart" → 「システムを再起動します」
  • その他 → 「不明なコマンドです」

期待される動作:

  • Changeノードでメッセージを設定し、Debugで表示

成功の条件:

  • "start" を送ると、デバッグパネルに "システムを起動します" が表示される
  • "stop" を送ると "システムを停止します"、"restart" を送ると "システムを再起動します" が表示される
  • "hello" など未定義のコマンドを送ると "不明なコマンドです" が表示される
  • 各コマンドに対して1つのメッセージのみが出力され、他の出力ポートには届かない
ヒント

Switchノードの設定:

  • プロパティ: msg.payload
  • ルール: 文字列の完全一致（==）を使用
  • 最後に「otherwise」で不明コマンドをキャッチ

Changeノードの設定:

  • 各出力ポートにChangeノードを接続
  • msg.payloadを対応するメッセージに変更

演習3: 温度アラート中級

温度センサーの値に応じて、正常/警告/危険のアラートレベルを判定するフローを作成してください。

要求仕様:

  • 範囲チェックを使用
  • 18〜26度: 正常（緑）
  • 26〜30度または10〜18度: 警告（黄）
  • 30度以上または10度以下: 危険（赤）

期待される動作:

  • 各レベルに応じたメッセージを出力

成功の条件:

  • 22度を入力すると "正常" 系のメッセージが表示される
  • 28度を入力すると "警告" 系のメッセージが表示される
  • 35度や5度を入力すると "危険" 系のメッセージが表示される
  • 境界値（18度、26度、30度、10度）が正しいカテゴリに分類される
ヒント

Switchノードの設定:

  • プロパティ: msg.payload
  • 「is between」条件を使用
  • 複数の条件を組み合わせる

条件の順序:

  • より具体的な条件を先に配置
  • または「最初に一致で終了」を使用

演習4: 配列の正負分離上級

数値の配列を入力し、正の数と負の数を別々の配列に分離してください。

要求仕様:

  • Split、Switch、Joinノードを組み合わせる
  • 入力: [3, -2, -1, 1, -3, 2]
  • 出力1: [3, 1, 2]（正の数）
  • 出力2: [-2, -1, -3]（負の数）

期待される動作:

  • 配列を分割 → 正負で振り分け → 再結合

成功の条件:

  • デバッグパネルに2つのメッセージが表示される（正の数の配列と負の数の配列）
  • 正の数の配列が [3, 1, 2]（順序は入力順）になっている
  • 負の数の配列が [-2, -1, -3]（順序は入力順）になっている
  • 0が入力に含まれた場合、どちらの配列にも含まれない（0は正でも負でもない）
ヒント

フローの構成:

  1. Inject: JSON配列を入力
  2. Split: 配列を個別メッセージに分割
  3. Switch: > 0 と < 0 で振り分け
  4. Join: 各出力を配列に再結合（手動モード・カウント=3）

Joinノードの設定:

  • モード: 手動（custom）
  • 結合の数: 3（各出力に振り分けられる件数）
  • 各メッセージ部: 配列（array）

なぜ自動モードではないのか: Joinの自動モードは「分割前の総数（ここでは6件）」が揃うまで待って再結合します。間にSwitchを挟むと各Joinには3件しか届かないため、自動モードでは6件に到達せず出力されません。振り分けてから再結合する場合は、手動モードで件数（ここでは3）を指定します。

5. まとめ

Switchノードのポイント

  • 条件分岐: if-else文やswitch-case文に相当
  • 複数の条件タイプ: 比較、範囲、正規表現、型チェックなど
  • 複数出力: 条件ごとに異なる出力ポートに送信
  • 全ルール確認: 複数の条件に一致する場合、すべてに送信可能
  • otherwise: どの条件にも一致しない場合のデフォルト処理
  • プロパティ選択: payload以外のプロパティ（topic等）も評価可能

よくある間違い

  • 条件の順序: 「最初に一致で終了」の場合、条件の順序が重要
  • 型の不一致: 文字列"10"と数値10は異なる（型を確認）
  • 全ルール確認の誤解: 有効だと複数の出力に送信される
  • 範囲の境界: betweenは両端を含む（>=と<=）
  • otherwiseの位置: 必ず最後のルールとして配置
  • 正規表現のエスケープ: 特殊文字は適切にエスケープ

次のステップ

次は以下のノードも学習するとよいでしょう:

  • Changeノード: メッセージの変更・削除・移動
  • Functionノード: より複雑な条件処理
  • Split/Joinノード: 配列の分割・結合との組み合わせ
  • Filterノード: 値の変化検知

6. トラブルシューティング

よくある問題と解決方法

問題 原因 解決方法
どの出力にも送信されない 条件に一致しない otherwiseルールを追加、または条件を見直し
複数の出力に送信される 全ルール確認が有効 「最初に一致で終了」を有効化、または条件を排他的に
数値比較が動作しない 文字列として比較されている 比較値の型を「数値」に設定
正規表現がマッチしない パターンエラー 正規表現をテストツールで確認
期待と異なる出力に送信 条件の順序問題 条件の順序を見直し（厳しい条件を先に）
範囲チェックが想定外 境界値の扱い betweenは両端を含むことを確認

7. 実務での活用例

ケース1: IoTセンサーデータの振り分け

複数センサーからのMQTTメッセージ ↓ Switchノード（msg.topicで判定） ├─ sensors/temperature → 温度処理フロー ├─ sensors/humidity → 湿度処理フロー ├─ sensors/pressure → 気圧処理フロー └─ otherwise → ログ記録 ※ センサー種別ごとに異なる処理を実行

ケース2: HTTPリクエストのルーティング

HTTP Inノード ↓ Switchノード（msg.req.methodで判定） ├─ GET → データ取得処理 ├─ POST → データ作成処理 ├─ PUT → データ更新処理 ├─ DELETE → データ削除処理 └─ otherwise → 405 Method Not Allowed ※ RESTful APIのルーティング

ケース3: エラーレベル別処理

エラーメッセージ受信 ↓ Switchノード（msg.payload.levelで判定） ├─ "critical" → 緊急通知 + 自動復旧 ├─ "error" → アラート送信 ├─ "warning" → ログ記録 └─ "info" → 統計更新のみ ※ ログレベルに応じた処理の分岐

ケース4: ユーザー権限チェック

ユーザーリクエスト ↓ Switchノード（msg.user.roleで判定） ├─ "admin" → 全機能アクセス ├─ "editor" → 編集機能のみ ├─ "viewer" → 閲覧のみ └─ otherwise → アクセス拒否 ※ 権限ベースのアクセス制御
