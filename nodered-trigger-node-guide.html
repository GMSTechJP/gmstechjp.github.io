📚 1. Triggerノードとは?

🤔 「Trigger」って何?

Triggerノードは、メッセージを受け取ったときに、時間差で2つの値を出力するノードです。 日常生活で例えると、自動ドアのセンサーのようなものです。

🚪 自動ドアに例えると:

  • 🚶 人が近づく = メッセージを受信（トリガー発動）
  • 🔓 ドアが開く = 1つ目の値を出力（即座に）
  • ⏳ 一定時間待機 = 設定した時間が経過
  • 🔒 ドアが閉まる = 2つ目の値を出力（時間後に）

📦 基本的な動作

Inject → Trigger → Debug

Triggerノードは、メッセージを受け取ると最初の値を即座に出力し、 指定した時間後に2つ目の値を出力します。 LED点滅、タイムアウト検知、ウォッチドッグタイマーなど様々な用途に使えます。

⏱️ Triggerノードの動作タイムライン:

入力
IN
0秒
出力
1
0
0秒 待機中... 2秒後

⚙️ 2. Triggerノードの動作モード

基本 ワンショット

入力で1つ目の値、時間後に2つ目の値を出力

用途: LED点滅、一時的な状態変更 例: "1" → 2秒後 → "0"

監視 ウォッチドッグ

一定時間メッセージがないときに警告を出力

用途: 接続監視、生存確認 例: 5秒間無入力 → "timeout"

延長 タイマー延長

入力ごとにタイマーをリセット・延長

用途: 活動検知、自動ログアウト 例: 入力で延長、無入力で発火

再送 定期再送

指定間隔で繰り返しメッセージを送信

用途: キープアライブ、定期通知 例: 無入力時に1秒ごとに"0"を送信

設定項目

Trigger ノードの設定欄は 1 つのパネルに上から並ぶ。チェックボックスはエディターの表示文がそのままラベルになっている。

設定項目説明既定値
送信データトリガーを受けて最初に送信する値（型と値を指定）。「なし」を選ぶと最初は何も送らない。1
送信後の処理最初の送信後の動作を選ぶ。「初期化されるまで待機」／「指定した時間待機」（待機後に再送信データを送る）／「指定した時間間隔毎に送信を繰り返す」。指定した時間待機
時間間隔「指定した時間待機」「繰り返す」を選んだときの長さ。ミリ秒／秒／分／時間で指定する。250 ミリ秒
新たなメッセージを受け取った時に遅延を延長オンにすると、待機中に新しいメッセージが届くたびにタイマーを延長（リセット）して待ち直す。オフ
msg.delayを用いて遅延時間を上書きオンにすると、受信メッセージの msg.delay（ミリ秒）で待機時間を上書きできる。オフ
再送信データ待機後に送信する値。「なし」なら待機後に何も送らない。0
2つ目の出力端子に2番目のメッセージを送信オンにすると出力端子が 2 つになり、再送信データを 2 つ目の端子から出す。オフ
初期化条件:待機中のタイマーを途中でリセットする条件。「msg.resetを設定」または「msg.payloadが次の値」のとき初期化する。（なし）
処理対象タイマーを全体で 1 つにするか、トピック毎に分けて持つか。「全メッセージ」か「（トピック）毎」を選ぶ。全メッセージ
名前エディター上の表示名。動作には影響しない。（空）

パターン1: ワンショット出力（LED点滅）

用途: 入力で"1"を出力し、一定時間後に"0"を出力

Inject → Trigger
("1"→2秒後→"0") → Debug

🔄 動作の流れ:

入力
メッセージ
→
Trigger
2秒待機
→
1 即座に "1" を出力
2 2秒後に "0" を出力

📌 Raspberry Pi LEDの点滅例:

  • "1" → GPIOをHIGHに → LED点灯
  • 2秒後 "0" → GPIOをLOWに → LED消灯
  • ボタン押下でLEDが2秒間点灯する仕組み

設定例:

送信データ: 文字列 "1" 再送信データ: 文字列 "0" 待機時間: 2秒 延長: 無効 出力: 1ポート（同じ出力）

パターン2: ウォッチドッグタイマー

用途: 一定時間メッセージがない場合にタイムアウトを検知

ハートビート → Watchdog
(5秒延長) → Timeout!

⏱️ 延長モードの動作:

入力
1
2
3
0秒 2秒 4秒
タイマー
!
9秒
タイムアウト

⚠️ 「延長」オプションの重要性:

  • 延長 有効: 新しいメッセージでタイマーをリセット（ウォッチドッグ向き）
  • 延長 無効: 新しいメッセージでも既存のタイマーは継続

設定例:

送信データ: 何も送らない 再送信データ: 文字列 "timeout" 待機時間: 5秒 延長: ✓ 有効（入力ごとにタイマーリセット） 動作: 5秒間入力がない → "timeout"を出力

パターン3: ストリーム停止検知と定期送信

用途: データストリームが停止したらプレースホルダーを定期送信

データ → 検知
(2秒) → 定期送信
(2秒毎) → 出力

📌 2つのTriggerノードの連携:

  1. 1つ目: ストリーム停止を検知（延長モード）
    • 入力あり → "reset"を出力（2つ目のTriggerをリセット）
    • 2秒間入力なし → trueを出力（定期送信開始）
  2. 2つ目: 定期的にプレースホルダーを送信
    • "reset"受信 → 定期送信を停止
    • true受信 → 2秒ごとに"0"を送信

設定例（1つ目 - 検知）:

送信データ: 文字列 "reset" 再送信データ: 真偽値 true 待機時間: 2秒 延長: ✓ 有効

設定例（2つ目 - 定期送信）:

送信データ: 数値 0 再送信データ: 文字列 "0" 待機時間: -2秒（負の値 = 繰り返し送信） リセット: "reset"

パターン4: タイムアウト処理（2出力）

用途: 処理の完了またはタイムアウトを検知

処理開始 → Trigger
(2出力) ↗ 処理 ↘ 結果

🔀 2出力モードの動作:

入力
→
Trigger
3秒タイムアウト
→
1 出力1: 即座にpayloadを送信 → 処理へ

設定例:

送信データ: ペイロード（入力値をそのまま） 再送信データ: 文字列 "0"（または"timeout"） 待機時間: 3秒 出力: 別の出力に送信（2ポート） 出力1 → 処理フローへ（即座） 出力2 → タイムアウト処理へ（3秒後）

演習1: 簡単なワンショット初級

📝 課題:

ボタンを押すと"ON"を出力し、3秒後に"OFF"を出力するフローを作成してください。

🎯 要求仕様:

  • Inject、Trigger、Debugノードを使用
  • Injectボタンクリック → 即座に"ON"出力
  • 3秒後 → "OFF"出力

📊 期待される動作:

  • デバッグウィンドウに"ON"が表示され、3秒後に"OFF"が表示

✅ 成功の条件:

  • Injectボタンを押した直後、デバッグパネルに "ON" が表示される
  • ボタンを押してから約3秒後に "OFF" が自動的に表示される
  • ボタンを連続して押しても、それぞれのボタン押下から3秒後に "OFF" が出力される
💡 ヒント

Triggerノードの設定:

  • 送信データ: 文字列 "ON"
  • 再送信データ: 文字列 "OFF"
  • 待機時間: 3秒

演習2: センサー監視中級

📝 課題:

センサーからの信号が10秒以上途切れたら「センサー異常」を出力するウォッチドッグを作成してください。

🎯 要求仕様:

  • Inject（センサー信号代わり）、Trigger、Debugノードを使用
  • 10秒間センサー信号がない → "センサー異常"を出力
  • 信号を受信したらタイマーをリセット

📊 期待される動作:

  • Injectを10秒以内に押し続ける → 何も出力されない
  • 10秒間押さない → "センサー異常"が出力

✅ 成功の条件:

  • Injectボタンを8秒ごとに押し続けている間、アラート用Debugノードには何も表示されない
  • ボタンを10秒以上押さずに放置すると、デバッグパネルに "センサー異常" が表示される
  • 異常検知後に再度ボタンを押すと、タイマーがリセットされて次の10秒間は異常が検知されない
💡 ヒント

Triggerノードの設定:

  • 送信データ: 何も送らない（センサー正常時は何も出力しない）
  • 再送信データ: 文字列 "センサー異常"
  • 待機時間: 10秒
  • 延長: ✓ 有効（重要！）

演習3: 自動ログアウト中級

📝 課題:

ユーザーアクティビティがない状態が5秒続いたら「ログアウト」メッセージを出力するフローを作成してください。

🎯 要求仕様:

  • アクティビティ（Inject）があるたびに「アクティブ」を出力
  • 5秒間アクティビティがない → 「ログアウト」を出力
  • "logout"というpayloadでリセット（強制ログアウト機能）

✅ 成功の条件:

  • アクティビティInjectを押すたびに、デバッグパネルに "アクティブ" が表示される
  • 5秒間アクティビティInjectを押さないでいると、デバッグパネルに "ログアウト" が表示される
  • 強制ログアウトInjectを押すと、5秒待たずに即座に "ログアウト" が出力される
  • 4秒おきにアクティビティを送り続けている限り、「ログアウト」が出力されない（タイマー延長が機能している）
💡 ヒント

Triggerノードの設定:

  • 送信データ: 文字列 "アクティブ"
  • 再送信データ: 文字列 "ログアウト"
  • 待機時間: 5秒
  • 延長: ✓ 有効
  • リセット: "logout"

強制ログアウト（即時出力）の作り方: リセットはタイマーを取り消すだけで何も出力しません。そこで「強制ログアウト」Injectは2方向に配線します。

  • ① Triggerへ（payload "logout" がリセット条件に一致し、保留中の自動ログアウトを取り消す）
  • ② Changeノードへ → payload を "ログアウト" にして Debug へ（5秒待たず即座に表示）

演習4: 処理タイムアウト上級

📝 課題:

処理を開始し、3秒以内に完了しなければタイムアウトとするフローを作成してください。2出力モードを使用します。

🎯 要求仕様:

  • Triggerノードの2出力モードを使用
  • 出力1 → 処理を実行（Functionノードで1秒/5秒の遅延処理）
  • 出力2 → 3秒後にタイムアウト
  • 処理完了またはタイムアウトの早い方を結果として出力

📊 期待される動作:

  • 1秒処理 → "done: 1s"が出力（タイムアウト前に完了）
  • 5秒処理 → "timeout"が出力（処理完了前にタイムアウト）

✅ 成功の条件:

  • 「1秒処理」Injectを押すと、約1秒後に "done: 1s" がデバッグパネルに表示される（タイムアウトメッセージは表示されない）
  • 「5秒処理」Injectを押すと、約3秒後に "timeout" がデバッグパネルに表示される（処理完了メッセージは表示されない）
  • どちらの場合も最終的なデバッグパネルへの出力は1つだけ（重複して表示されない）
  • 2出力モードのTriggerノードで出力1（処理実行）と出力2（タイムアウト）が別々のパスに接続されている
💡 ヒント

フローの構成:

  1. Triggerノード: 2出力モード、3秒タイムアウト
  2. 出力1 → Functionノード（遅延処理）→ 結果へ
  3. 出力2 → Changeノード（"timeout"設定）→ 結果へ
  4. 結果用Triggerノード: msg._msgidで重複排除

重複排除のTrigger設定:

  • 送信データ: ペイロード
  • 再送信データ: 何も送らない
  • トピック別: msg._msgid
  • 待機時間: 10秒（処理時間より長く）

🎓 5. まとめ

Triggerノードの重要ポイント

  • ✅ 2つの出力値: 即座の出力と時間後の出力を設定可能
  • ✅ 延長モード: 入力ごとにタイマーをリセット（ウォッチドッグ用）
  • ✅ リセット機能: 特定の値でタイマーをキャンセル
  • ✅ 2出力モード: 別々のフローへ出力可能
  • ✅ 負の時間: 繰り返し送信（-2秒 = 2秒ごとに送信）
  • ✅ トピック別: トピックごとに独立したタイマーを管理

⚠️ よくある間違い

  • 延長を忘れる: ウォッチドッグで延長が無効だとリセットされない
  • リセット値の不一致: リセット条件の値と送信値が一致しない
  • 単位の誤り: ミリ秒と秒を間違える
  • 2出力の使い分け: 同じ出力 vs 別出力の選択ミス
  • トピック別の誤解: 全メッセージ処理とトピック別の違い
  • 負の時間の意味: 繰り返し送信になることを忘れる

📚 次のステップ

Triggerノードをマスターしたら、以下のノードも学んでみましょう:

  • Delayノード: シンプルな遅延処理
  • Filter (RBE)ノード: 変化検知
  • Switchノード: 条件分岐との組み合わせ
  • Functionノード: より複雑なタイマー処理

🔧 6. トラブルシューティング

よくある問題と解決方法

問題 原因 解決方法
ウォッチドッグが毎回発火する 延長が無効 「延長」オプションを有効にする
リセットが効かない リセット値の不一致 リセット条件の値を正確に設定
2つ目の出力がない 「何も送らない」設定 op2typeを確認、必要な値を設定
タイマーが重複 トピック別の誤設定 処理対象の設定を確認
メッセージが繰り返される 負の時間設定 時間を正の値に変更
即座の出力がない op1が「何も送らない」 op1typeを確認、必要な値を設定

💡 7. 実務での活用例

ケース1: IoTデバイスのハートビート監視

MQTTでデバイスからハートビート受信 ↓ Triggerノード（延長モード） - 1つ目: 何も送らない - 2つ目: "offline" - 待機時間: 30秒 - 延長: 有効 ↓ 30秒間ハートビートなし ↓ "offline"出力 → アラート送信 ※ デバイスの生存監視

ケース2: LEDインジケータ制御

ボタン押下 or センサーイベント ↓ Triggerノード - 1つ目: 1 - 2つ目: 0 - 待機時間: 2秒 ↓ GPIO出力ノード ↓ LEDが2秒間点灯して消灯 ※ Raspberry PiのLED制御

ケース3: 自動セッションタイムアウト

ユーザーアクティビティ（API呼び出し等） ↓ Triggerノード（延長モード） - 1つ目: 何も送らない - 2つ目: "session_timeout" - 待機時間: 15分 - 延長: 有効 ↓ 15分間アクティビティなし ↓ セッション無効化処理 ※ Webアプリケーションのセッション管理

ケース4: 処理タイムアウトと自動リトライ

API呼び出し要求 ↓ Triggerノード（2出力） - 出力1: 即座にAPIへ - 出力2: 5秒後にタイムアウト ↓ ├─ 出力1 → HTTP Requestノード → 成功処理 └─ 出力2 → リトライカウンター → 再試行 ※ API呼び出しのタイムアウト処理
